# 1・2・2 入出力管理

> **章**: 第1章 情報システム
> **節**: 1・2 オペレーティングシステム

プログラムの実行に伴って行われるデータの入力，処理結果の出
力は，機械や人間が関連するため計算処理よりもはるかに低速にな
ることがある．また，入出力装置はさまざまで個々に対応をしなけ
ればならないが，入出力という操作を統一的に扱えるようにするこ
ともまた重要である．これらの機能はOSの入出力管理プログラム
（デバイスドライバ）によって提供され，プロセスから見た入出力
は，OSを介したデータ構造とアクセス機構のモデルで表される．
ユーザプロセスは，通常，システムコールによりOSを通じて入出
力装置を利用する．
この入出力管理の関係を図1･12に示す．図中のOSカーネル
（kernel ）の機能は，すべてが一つの実行プログラムとして実現さ
れている場合もあるが，一部を別のプロセスとして実現している場
図1･12入出力管理
図1･13 デバイスドライバ
合もある．必要最小限の機能だけをカーネルとし，管理機能やデバ
イスドライバの多くを独立したサーバプロセスとして実装する手法
をマイクロカーネル構法（micro kernel architecture）という［マ
イクロカーネルOSに対し，システムサーバをカーネル内にもった
OSをモノリシック（monolithic）OSと呼ぶ］．
動作速度の異なる入出力装置とCPUを効率よく稼働させるた
め，プログラムがデータをすぐに利用できるようにし，CPUにプ
ログラムを続行させながらデータを出力できるようにするため，入
出力データを特定の記憶場所に一時的に蓄えておく．これをバッフ
ァリング（buﬀering），この記憶場所をバッファ（buﬀer ）と呼ぶ．
バッファリングを有効活用するため，また，いつ到着するかわか
らないデータの到着を受けてただちに処理をするため，そのような
事象（event ）が生じた場合は実行中のプログラムを中断して入出
力処理プログラムを実行し，その後，元に戻るようにする．このよ
うに実行を中断して別のプログラムを実行させることを割込み
（interrupt ）という．
入出力装置には，磁気ディスクなどのようにデータをブロック単
位で入出力するものや，キーボードのように文字単位で入出力する
ものがある．ブロック単位の入出力では，データチャネルが主記憶
装置への直接アクセス（DMA:direct memory access ）によって
データ転送を行い，動作の終了をCPUに割込みで知らせる．
デバイスドライバの仕組みを図1･13に示す．デバイスドライバ
は，OSの立ち上げ時や入出力装置が接続されたときに対応するデ
バイスに対して動作やインタフェースの設定などの初期化を行う．
OSによっては，OSの立ち上げ時や装置が接続されたときにデバ
イスドライバのインストールを行う．
プロセス（必ずしもユーザプロセスとは限らない）から入出力要
求があったときには，装置が可動状態であれば入出力を開始する
が，そうでなければ要求を到着順に待ち行列に入れ，可動状態にな
ってから順次処理する．
外部から非同期的な入力があったときや，装置がひとまとまりの
動作を終えたときなどに，デバイスから割込み要求信号が入力され
る．割込みが発生すると対応したデバイスドライバの割込み処理ル
ーチンが起動される．磁気ディスクへの入出力などの場合にはデー
タ転送はデータチャネルが行い，割込みにより転送の終了を知らせ
るので，要求元のプロセスに終了を知らせ，もし待ち行列に別の要
求があれば順に次の処理を起動すればよい．データチャネルによら
ない文字型の入出力デバイスの場合は，バッファとデバイスの間の
データ転送もデバイスドライバの制御ルーチンでCPUが行う．複
数のプロセスがディスクの入出力を要求する場合や，プリンタへの
出力を要求する場合など，デバイスドライバでは複数の処理が非同
図1･14ストリーム入出力システム
期的に生起するので，プリンタやバッファなどの共有リソースを使
う際の排他制御が必要である．
装置の速度や重要度に従って割込み優先度がハードウェアで決め
られており，割込み処理ルーチンの実行中に，より優先度の高い割
込みが発生すると，低い優先度の処理は中断されて高い順位の割込
み処理が開始される．処理中の割込みと同じかそれ以下の順位の割
込みは，それを要求する事象が生じても信号がハードウェアで止め
られるので発生しない．そのため，割込み処理中は同じ優先順位の
事象が生起していても割込み信号が受け付けられていない可能性が
あるので，割込み処理ルーチンでは緊急性のある処理だけをできる
だけ短時間に行うようにする必要がある．
入出力の要求が待ち行列にたまっている場合に，デバイスの特性
に合わせて処理を高速化することもデバイスドライバで行うことが
ある．例えば，磁気ディスクへのアクセスにおいてヘッドを大きく
動かす必要があるときに，途中に別のアクセス要求のあるシリンダ
があれば順に寄って移動するようなスケジューリングを行うことに
より，全体としてのアクセス時間を減らす．
デバイスを直接制御するデバイスドライバが提供するデータ構造
やアクセス機構を，ユーザプロセスからじかに使うこともできる
が，デバイスドライバとユーザプロセスの間に，データ構造やアク
セス機構を変換するモジュール（フィルタ）を介して，より高度な
ユーザインタフェースをもつ入出力システムを構成する手法もあ
る．UNIXではこのデータの流れをストリームと呼んでいる．
ストリーム入出力システムの概念を図1･14に示す．ストリーム
入出力システムでは，ユーザプロセスからデバイスに向かう流れを
ダウンストリーム，逆をアップストリームと呼ぶ．ストリーム入出
力の考え方は，ネットワークの物理的なデバイス制御のためのドラ
イバから，階層的なプロトコルのレベルに応じた仮想的なデバイス
のドライバを順に実現する場合にも有効である．
