# 1・3・3 データベース

> **章**: 第1章 情報システム

データベースとは，各種のアプリケーションが利用するデータを
統一的に収集管理し，容易に参照や共有ができるようにすることを
目的とした，データの倉庫のようなものである．この機能を実現す
る基本ソフトウェアはデータベース管理システム（DBMS）と呼
ばれ，アプリケーションに対してデータの一定の見え方を提供し，
データの参照や保護などを行う．
データを統一的に管理するうえでは，同種のデータを一つの型と
してまとめる抽象化が有効である．このようにしてデータベース内
のデータの構造や関係，制約などを記述したものをスキーマと呼
ぶ．また，このスキーマに従って格納された個々のデータ群をイン
スタンスと呼ぶ．
a.データモデル 蓄積しようとするデータの構造やそのデー
タに適用可能な操作の集まりをデータモデルと呼ぶ．データモデル
には，データのもつ構造の記述の規約，データに対して可能な操作
のセット，データが矛盾なく現実の世界を表現すために必要な制約
の三つが要素として含まれる．
.ネットワークデータモデル このデータモデルでは，
個々の要素をデータの並びであるレコードとして表現する．また，
レコード間には親子関係を定義することができて，ループ状のポイ
ンタを用いて一対多の親子関係を実現することができる．
このデータモデルでは，このようにしてあらかじめ用意されたデ
ータの間を，カーソルと呼ばれるポインタで順次走査していくこと
によって，必要なレコードを発見することができる．
.階層データモデル このデータモデルでは，個々の要素
を表すレコードを節とする木構造を基本単位とする．しかし，これ
だけでは一対多の関係を表現するのは容易であるが，多対多の関係
を表現するには効率が悪いため，異なった木に属するレコード間を
つなぐ論理的な関係を導入して，効率的な運用を図っている．
.関係データモデル このデータモデルはリレーショナル
データモデルとも呼ばれ，ここでは関係（リレーション）を中心と
してデータベースを構築する．関係とは表のようなもので，表中の
各列に並ぶ事柄を属性と呼ぶ．一方，各行はタプルと呼ばれ，この
タプルの集まったものが関係データベースのインスタンスとなる．
このモデルにおけるデータ操作は関係代数と関係論理と呼ばれる
形式的な体系によって規定されている．関係代数は関係に対する基
本的な操作を行う代数演算を定義する．一方，関係論理は1階述語
論理に基づく．
b.関係データベース（relational database) 関係データベ
ースは関係データモデルに基づくデータベースであり，現在広く使
われているものである．
.関係代数 関係代数は関係データベースのデータ操作の
理論的な基礎となるもので，リレーショナル代数とも呼ばれる．基
本的な関係代数演算子は5種類で，タプルの集合である関係の和と
差，直積，さらにある関係のもつ属性のうち必要ないものを削除し
た関係を生成する射影と，ある関係のタプルの中からある属性の値
と定数またはもう一つの属性の値を大小比較した結果，もしくはそ
のような複数の比較の結果を論理演算したものの真偽によってタプ
ルを取捨選択した選択，の五つの演算である．
これらは基本演算であり，これらの組合せで任意のデータ操作を
合成することができる．
.関係データベース言語SQL SQLはISOの規格にもな
っている標準的な関係データベースの操作用言語である．プログラ
ミングのための利便性を図るために幾分か純粋な関係データモデル
から逸脱したところもあるため，SQLでは関係を「表」，属性を
「列」，タプルを「行」と呼び替える．基本的な問い合わせは
という形式で行い，条件式のところに選択演算のための条件式を指
定できる．また，条件式のところに問い合わせを含む式を書くこと
もできて，さらに複雑な問い合わせを行うこともできる．

d.アクティブデータベース アクティブデータベースとは，
データの参照や変更などの何らかの契機があったときに，データベ
ース側からアプリケーション側に対して何らかの働きかけをする能
力をもつデータベースを意味し，データモデルによる分類とは異な
った見地からの分類である．
アクティブデータベースの行う働きかけには，データベース内の
データの更新やアプリケーションに対するメッセージの送信などが
考えられ，データやイベントによって駆動されるアプリケーション
の中核を構成する．
