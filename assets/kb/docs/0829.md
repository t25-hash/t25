.データ記述・操作体系と言語 DBMSはビジネスロジ
ック層の複数のアプリケーションからの利用を可能とするため，対
象データとそれに対する操作体系を抽象化した形態で提供する必要
がある．この抽象化した形態をデータモデルと呼ぶ．データモデル
により，物理的なデータ配置等を気にせず，論理的なデータ記述と
操作が可能となる．
データモデルの詳細についてはc項に後述するが，代表的な例と
してリレーショナルデータモデルがある．図5･20は，Jリーグの
チーム，選手に関するデータを表現したものである．
リレーショナルデータモデル（relational data model ）では，デ
ータベースを表の集まりとして表現する．表のことをリレーション
（関係）と呼び，各リレーションを関連付けながら，各種の操作を
提供する．
DBMSは，データモデルに基づいたデータ記述および操作のた
めの言語を提供する．データ記述のための言語をデータ定義言語
（data definition language）と呼び，操作のための言語をデータ操
作言語（data manipulation language）と呼ぶ．また，機密保護
やトランザクションの制御のための言語をデータ制御言語（data
control language）と呼ぶ場合もある．これらは，個別の言語とし
て提供される場合と，一つの言語体系に統一される場合とがあり，
現在のデファクトスタンダードであるSQL は後者である．SQL
は，System R（IBMサンホセ研究所で開発されたリレーショナル
DBMS）の研究開発において提案された言語であり，SEQUEL
（structured English query language）と呼ばれ，その後，SQLと
改名された．それをもとに，ISO（International Organization for
Standardization）にて規格制定が行われた．
大量のデータを格納・管理するために，種類の同じデータを型と
してグループ化して，共通の構造や関連を導き出すことにより抽象
化し，さらに各種の整合性制約等を記述したものをスキーマ
（schema）と呼ぶ．前述の例では，二つのリレーションの枠組みが
スキーマの構成要素である．スキーマに基づいて格納されたデータ
の集まりをインスタンス（instance）やオカレンス（occurrence ）
と呼ぶ．同様に前述の例では，図5･20中に表されたリレーション
中のデータがインスタンスである．スキーマは，インスタンスのデ
ータの構造や諸元，関連，制約を記述したデータということもでき
る．このようなデータのことをメタデータ（meta data）と呼び，
このメタデータの管理もDBMSの機能の一つである．
スキーマは，その抽象度合いから，人が概念的に全情報の関係を
理解できる概念スキーマ，個々のアプリケーションごとの視点から
抽出して定義した外部スキーマ，物理的データ格納方式等を記述し
た内部スキーマとに層別することが一般的であり，抽象化度を，そ
れぞれ概念レベル，外部レベル，内部レベルという．この層別は，
ANSI / SPARC（American National Standard Institute/System
Planning and Requirements Committee，アメリカ規格協会の標
準化計画委員会）内の研究グループにより提案されたため，
ANSI /SPARCモデル と呼ぶ（図5･21）．これらスキーマ間の
データの写像（mapping）を行うことも，DBMSの機能の一つで
ある．
また，概念スキーマと外部スキーマ間の独立性を論理的データ独
立性（logical data independence），概念スキーマと内部スキーマ
間の独立性を物理的データ独立性（physical data independence）
と呼ぶ．
.処理効率 DBMSは，集約・蓄積された大量のデータ
からアプリケーションに必要な部分を高速に取出す，あるいは，ア
プリケーションが追加・更新することが可能となるために，種々の
物理データ格納方式や検索の効率化のための索引情報を格納・管理
する．これらの工夫により，システム全体の効率を保証することも
DBMSの装備すべき機能といえる．
.整合性維持 記述されたデータモデルに基づき，おのお
ののデータの構造や関係をデータが満たすべき制約下に保持するこ
ともDBMSの機能の一つである．例えば，図5･20の例では，
個々の選手はユニークな選手番号をもつ，あるいは，一人の選手は
同時に一つのチームにしか所属できないという制約があるといえ
る．これらをデータの更新時に検査することもDBMSの役割であ
る．
.機密保護 ユーザ認証等の情報をもとに，各ユーザがど
のデータにアクセス可能か否か，すなわち，検索，挿入，削除，更
新が可能かを管理することもDBMSの機能の一つである．
.トランザクション制御 アプリケーションからのデータ
ベース処理のまとまりをトランザクションと呼ぶ．トランザクショ
ンは，通常複数並行処理される．この場合，同一のデータに対する
複数トランザクションからの検索，更新等の要求が制御なしに行わ
れると，データベースの整合性がとられていない状態のデータが読
み出せたり，他のトランザクションの更新により，有効なデータの
更新が打ち消されてしまったりする．このような問題を解消するた
めに，DBMSはデータのロックやコミット，ロールバックといっ
た機構を用いて，並行処理制御（concurrency control ），すなわ
ち，トランザクション制御機能を提供する．
.障害回復 コンピュータシステムの信頼性が，ハードウ
ェア，基本ソフトウェア両面での対応により高まることは，情報シ
ステムにとって非常に有効であるが，各種の障害が全くなくなるこ
とはない．障害によってデータベースに不整合が発生した場合，こ
れを回復するための機能，リカバリ機能が必要となる．
DBMSは，各トランザクションと実施されたデータ操作をログ
に記録し，さらに，バックアップされたデータと組合わせることで
リカバリ機能を提供する．
.データ移行 情報システムは，新たなビジネスロジック
の導入により進化していく．しかし，ビジネスロジックのみの変
化，すなわち，アプリケーションの変更・追加のみでは対応しきれ
ない場合，データベースの変更が必要となる．このような場合にお
いて，データモデルの変更に基づき，簡便な移行手段，移行機能を
提供することもDBMSの機能の一つである．
c.データモデル データモデル（data model ）とは，実世界
の情報の構造や関係を論理的，体系的に矛盾なく定義したデータベ
ースと，それを操作する枠組みである．言い換えれば，DBMSの
インタフェースとして，物理的データ格納形態や，内部の検索手順
などと独立した論理的なデータ記述と操作を可能とすることと，対
象となる情報システムが取扱う，実世界の複雑な情報構造，機能構
造を抽象化，モデル化するための枠組みである．
データモデルは，データベースに関する研究の中心的なテーマと
して，時代背景やニーズの差により，さまざまなものが考案されて
いる．ここではその代表的な例として，階層データモデル，ネット
ワークデータモデル，リレーショナルデータモデル，ER（entity
relationship）モデルについて概説する．

.リレーショナルデータモデル リレーショナルデータモ
デルは，1970年にコッド（Codd) により提案された，数学的

タモデル，ネットワークデータモデルが，リンク，ポインタといっ

ンスタンスは，タプルの集まりとして表現される．

間の特定フィールド値の一致に基づくナチュラルジョインによるリ
ンク機構のみが用いられる．すなわち，どのような条件を満足する
リレーションの，どのような属性をアクセスするかという指定のた
めの操作系が必要となる．これを実装するのが，基本操作であるリ
レーショナル代数と，1階述語論理に基づくリレーショナル論理に
より開発されたSQLである．
リレーショナルデータモデルでデータの重複をなくすために論じ
られる最も重要な正規形は，第三正規形（third normal form）で
ある．非正規形から第三正規形への分解プロセスを図5･24に示す．
第三正規形の条件が満たされていれば，他のリレーションを考慮せ
ずにデータ更新を行っても，関数従属性にかかわる矛盾は発生しな
い．関数従属性とは，任意のインスタンス中の任意の2タプルに関
して，一方の値が等しいならば，もう一方の値も必ず等しいという
関係が成り立つことをいう．
正規形の考え方は非常に論理的であり美しいが，実際のデータベ
ースの整合性を規定する従属関係は，関数従属のみではない．ま
た，意味論的な整合性の保持が問題とされることも多い．このよう
な問題は，関数従属性の観点のみから保証することは困難である．
さらに，正規形にこだわりすぎるとリレーションの数がむやみに
多くなり，効率面での問題を引き起こすことも考えられる．
以上のようなことから，実際のデータベースにおいては，関係表
現をリレーション（表）により表現するが，リレーションの各フィ
ールドに別の関係を含むことを許すことで，実用上の効率を確保し
ている．

造ノウハウ等の非形式化データを取扱うことも，CAD/ CAM/ CAE
システムのDBMSの大きな役割の一つとなりつつあり，さまざま
な研究，実験が行われている．
このようなデータも含めて，CAD/CAM/ CAEシステムの特徴
として，データモデルの変動が大きい，あるいは変動そのものがア
プリケーションの機能であるといったモデル変更性も重要な要件の
一つといえる．実際の設計行為は，一つの設計手法に基づいた手順
の羅列のみでは成し得ず，さまざまな形状の変形や構造的変更を加
え，さらに評価するといった，設計サイクルを何重にも何種類も実
施し熟成していく作業である．この観点からは，前述したような固
定的なデータモデルでは表現が困難な場合も考えられる．このため
には，高次なセマンティックスの表現方法とその実用化が検討課題
である ．
(３)データサイズが膨大
一般に，CAD/ CAM/ CAEシステムの取扱うデータサイズは大
きいといえる．ワイヤフレーム→サーフェス→ソリッド→フィーチ
ャパラメトリックに代表されるCADシステムの表現能力の進歩に
伴い，個々のデータのサイズも肥大化している．さらにCAD/
CAM/CAEシステムの対象が単なる図面，部品から，アセンブリ
ー情報や機構シミュレーションまでに拡張されていることも一因で
ある．昨今の自動車製造に関連する生産準備部門，特にプレス型設
計過程においては，型全体を表現するためのデータベースが数千部
品，データサイズにして数GBとなる例もまれではなくなってきて
いる．
(４)トランザクション処理単位がさまざま
CADシステムの機能単位は，例えば，点を創成，線を変更のよ
うな処理単位から，部品単位，アセンブリー単位の複写や保存ま
で，その対象データの単位はさまざまである．多くのCAD/ CAM/
CAEシステムのDBMSは，図面，部品，アセンブリーといった
一括・操作・格納単位の処理系と，形状や設計手順といった個々の
データの処理系を分離しているものが多く，後者はおもにクライア
ント機器上に実装されている．インタネットの進化に伴い，一部
に，すべてのトランザクション処理系をクライアント/サーバのア
ーキテクチャ上に実装しているDBMSもあるが，実用化にはブロ
もできる．
ードバンド，グラフィックアクセラレータの進化，廉価なメモリ等
のインフラストラクチャの高度化と，圧縮展開処理や表示用データ
作成処理の効率化が必須である．
(５) 協同設計・協調設計環境支援
一方で，設計作業全体の効率向上のニーズとCAD/CAM/ CAE
システムの機能自体の向上により，協同設計環境，協調設計環境を
提供することもCAD/CAM/CAEシステムのDBMSの必須要件
である．ここでは，複数人の設計者による同期管理，変更の伝ぱと
整合性管理がDBMSの主要な機能となる．
(６) データ交換
IGES（Initial Graphics Exchange Specification），STEP
（Standard for the Exchange of Product model data）等のデータ
交換標準形式や方式の適用，研究も進む一方で，CAD/CAM/ CAE
システム自体の内部表現形式は，システムの表現能力の高度化に伴
い統一方向にあるとはいえない．事実，すべてを同一のCAD/
CAM/CAEシステムにより統一しているユーザはまれである．こ
こでは，事前データチェックやヒーリング等のデータ交換アプリケ
ーションの高機能化が必要となることのほかに，データの諸元とし
て，どのようなシステム間を流れたデータであるのかを管理するこ
とも新たな要件となってきている．
.CAD/ CAM/ CAEシステムのデータベースの今後
CAD/ CAM/CAEシステムの扱うデータの構造化，階層化が進み，
さらに，これらを処理するアプリケーションの性能も高度化した．
また，性能を支えるハードウェア等の進化もCPU性能の向上，グ
ラフィックアクセラレータの性能向上と合わせて，進化度合いは速
いといえる．CAD/ CAM/CAEシステムもオブジェクト指向技術
の適用が盛んに行われている．
向DBMSの基本要件を備えたオブジェクトリレーショナル
DBMSへと進化しつつある．オブジェクト指向DBMSの基本要
件とは，(１)複合オブジェクト，(２)オブジェクト識別性，(３)カ
プセル化，(４)クラス，(５)クラス階層，(６)再定義・多重定義・
遅延束縛，(７)計算完備，(８)拡張可能性，(９)永続性，(10)二次
記憶管理，(11)同時実行制御，(12)障害回復，(13)アドホックな問
い合わせ である．
図5･27生産情報システムの変遷
クライアントシステム上での処理系において，ⅰ項(１)に述べた処
理効率やデータサイズの問題から概念的な導入にとどまっていると
いえ，内部構造はリレーショナルデータモデルに基づいたDBMS
により，アプリケーションに対応し最適化された独自の実装がなさ
れている．しかし，このようなシステムはオブジェクト指向技術に
より，概念的にカプセル化された単位でのアプリケーションインタ
フェースを備えた，オブジェクトデータモデルDBMSと見ること
すなわち，内部的な実装方式は別として，CAD/CAM/ CAEシ
ステムのDBMSは，構造化，階層化されたオブジェクトデータモ
デルを操作するアプリケーションインタフェースの集合であるのが
現状の一般的な見方である．一部に，すべてを純粋なオブジェクト
指向技術により実装したCADシステムも見受けられるが，処理効
率や量的なデータ表現能力でいまだ実用的でない．
以上に加えて，CAD/ CAM/ CAEシステムの機能向上により，
マルチメディアデータベース，全文検索データベースの技術的な取
込みも必要となり，CAD/ CAM/ CAE分野以外において実用化が
進みつつあるXML（extensible markup language）や，XVL
（extensible virtual world description language）などの活用も盛
んとなるであろう．
一方で，ⅰ項で示したPDMシステムは，CAD/ CAM/ CAEシ
ステムのDBMSの一部と重複する機能を有しているが，その視点
は大きく異なり，業務，ビジネスの流れが基本である．例えば，部
品構成管理はPDMシステムの重要な機能の一つであるとともに，
CADシステムにおけるアセンブリー情報の一部である．しかし，
PDMシステムでは，単に親子の関係だけがあればよいのに比較し
て，CADシステムにおいては，どのように親子関係を構築するか
といった部品間の配置拘束情報や，部品間のクリアランス等を定義
するための，設計のための情報を表現しなければならない．PDM
システムはビジネスの範囲変化に伴い，さまざまなシステム間の連
携を取ることが必要となり，いずれはビジネス全体のデータの流
れ，業務の流れを管理するためのシステム，製造業情報基盤
DBMSへと変化していく必要があると考える．CAD/ CAM/ CAE
システムのDBMSは，この基盤DBMSとのインタフェースを備
えることで，CAD/ CAM/ CAEシステムをビジネス全体の一つの
一般的なDBMSは，リレーショナルDBMSにオブジェクト指
コンポーネントとして，他のビジネスコンポーネントとの情報共有
を可能とする必要がある．
