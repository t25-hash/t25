> **章**: 第12章 インテリジェントシミュレーション
> **節**: 12・10 オブジェクト指向

a.UML アルゴリズムの記述にフローチャートが有効であ
ったのと同じように，オブジェクト指向開発においても図形表示は
有効である．これまでにさまざまな表示法が提唱されてきたが，近
年，そのいくつかを統一した統一モデリング言語（UML:unified
modeling language) が提唱され，標準となっている．UMLで
は9種類の図が定義されているが，ここでは，重要なものについて
のみ説明する．
.クラス図（class diagram) クラス図はクラス間の静的
な関係を表したものであり，簡単なものは分析段階で，設計段階で
はより詳細なものが用いられる．クラスを表す箱とそれらを結ぶ線
分，あるいは矢印から構成される．基本的な表記法を図12･39に示
す．
図12･42 状態図の表記法
.ユースケース図（use case diagram) システム全体の
ふるまいをモデル化するために用いられる．個々のふるまいはユー
スケースとして述語（～を行うという形）で表現される．システム
外の関連するシステムあるいはオブジェクトは，アクターとして定
義される．ふるまいを図だけで示すのは困難であるから，その説明
となる概要，および使用例を表すシナリオを伴う．ユースケースの
表記法を図12･40に示す．
.シーケンス図（sequence diagram) オブジェクトの動
的関係，すなわちメッセージパッシングによる相互作用を，時間軸
に沿って表したもので，分析および設計の段階で用いられる（図
12･41)．
.状態図（statechart diagram) オブジェクトに注目し
て，その状態の遷移とその条件を示したもので，イベントに対する
反応を記述するのに適している（図12･42)．
b.デザインパターン（design pattern) オブジェクト指向と
いう方法論では，プログラムの再利用性を高めることができるが，
その上のレベルに対しては効果がない．しかし，類似の問題に対し
て，似たような解決法が与えられるのは容易に予想できることであ
り，これを問題解決のパターンとしてカタログ化したのがデザイン
パターン である．デザインパターンの概念自体は非常に広いも
のであるが，実際にはオブジェクト指向設計の枠組の中で，カタロ
グ化されている．ガンマら はデザインパターン利用の目的の一
つとして，システムの変更に対して，再利用性の高いプログラミン
グをすることを挙げている．したがって，本節の最初に挙げた計算
力学分野のソフトウェアに対する要請と合致しており，デザインパ
ターンの利用を検討すべきである．
c.コンポーネント（component ) 再利用性を高めるもう一
つの工夫が，コンポーネントの利用である．コンポーネントとはプ
ログラムの部品という意味であるが，インタフェースを統一したオ
ブジェクトを利用することにより，コンポーネント間の相互依存性
が少なくなり，オブジェクト指向アプローチの重要な概念となって
いる．JavaBeans では，入出力インタフェースを統一したオブ
ジェクト（bean）を用いることにより，GUI （graphical user
interface）を用いたアプリケーション開発が可能となっている．
