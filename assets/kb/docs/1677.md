# 1・2・2 入出力管理

> **章**: 第1章 情報システム
> **節**: 1・2 オペレーティングシステム

プログラムの実行に伴って行われるデータの入力、処理結果の出力は、機械や人間が関連するため計算処理よりもはるかに低速になることがある。また、入出力装置はさまざまで個々に対応をしなければならないが、入出力という操作を統一的に扱えるようにすることもまた重要である。これらの機能はOS の入出力管理プログラム(デバイスドライバ)によって提供され、プロセスから見た入出力は、OS を介したデータ構造とアクセス機構のモデルで表される。ユーザプロセスは、通常、システムコールによりOS を通じて入出力装置を利用する。この入出力管理の関係を図1・12に示す。図中のOS カーネル(kernel)の機能は、すべてが一つの実行プログラムとして実現されている場合もあるが、一部を別のプロセスとして実現している場図1・11 木構造ファイルシステム7―8情報・ソフトウェア図1・12 入出力管理合もある。必要最小限の機能だけをカーネルとし、管理機能やデバイスドライバの多くを独立したサーバプロセスとして実装する手法をマイクロカーネル構法(micro kernel architecture)という[マイクロカーネルOS に対し、システムサーバをカーネル内にもったOS をモノリシック(monolithic)OS と呼ぶ].動作速度の異なる入出力装置とCPU を効率よく稼働させるため、プログラムがデータをすぐに利用できるようにし、CPU にプログラムを続行させながらデータを出力できるようにするため、入出力データを特定の記憶場所に一時的に蓄えておく。これをバッファリング(buffering)、この記憶場所をバッファ(buffer)と呼ぶ。バッファリングを有効活用するため、また、いつ到着するかわからないデータの到着を受けてただちに処理をするため、そのような事象(event)が生じた場合は実行中のプログラムを中断して入出力処理プログラムを実行し、その後、元に戻るようにする。このように実行を中断して別のプログラムを実行させることを割込み(interrupt)という。入出力装置には、磁気ディスクなどのようにデータをブロック単位で入出力するものや、キーボードのように文字単位で入出力するものがある。ブロック単位の入出力では、データチャネルが主記憶装置への直接アクセス(DMA : direct memory access)によってデータ転送を行い、動作の終了をCPU に割込みで知らせる。デバイスドライバの仕組みを図1・13に示す。デバイスドライバは、OS の立ち上げ時や入出力装置が接続されたときに対応するデバイスに対して動作やインタフェースの設定などの初期化を行う。OS によっては、OS の立ち上げ時や装置が接続されたときにデバイスドライバのインストールを行う。プロセス(必ずしもユーザプロセスとは限らない)から入出力要求があったときには、装置が可動状態であれば入出力を開始するが、そうでなければ要求を到着順に待ち行列に入れ、可動状態になってから順次処理する。外部から非同期的な入力があったときや、装置がひとまとまりの動作を終えたときなどに、デバイスから割込み要求信号が入力される。割込みが発生すると対応したデバイスドライバの割込み処理ルーチンが起動される。磁気ディスクへの入出力などの場合にはデータ転送はデータチャネルが行い、割込みにより転送の終了を知らせるので、要求元のプロセスに終了を知らせ、もし待ち行列に別の要求があれば順に次の処理を起動すればよい。データチャネルによらない文字型の入出力デバイスの場合は、バッファとデバイスの間のデータ転送もデバイスドライバの制御ルーチンでCPU が行う。複数のプロセスがディスクの入出力を要求する場合や、プリンタへの出力を要求する場合など、デバイスドライバでは複数の処理が非同図1・13 デバイスドライバ図1・14 ストリーム入出力システム期的に生起するので、プリンタやバッファなどの共有リソースを使う際の排他制御が必要である。装置の速度や重要度に従って割込み優先度がハードウェアで決められており、割込み処理ルーチンの実行中に、より優先度の高い割込みが発生すると、低い優先度の処理は中断されて高い順位の割込み処理が開始される。処理中の割込みと同じかそれ以下の順位の割込みは、それを要求する事象が生じても信号がハードウェアで止められるので発生しない。そのため、割込み処理中は同じ優先順位の事象が生起していても割込み信号が受け付けられていない可能性があるので、割込み処理ルーチンでは緊急性のある処理だけをできるだけ短時間に行うようにする必要がある。入出力の要求が待ち行列にたまっている場合に、デバイスの特性に合わせて処理を高速化することもデバイスドライバで行うことがある。例えば、磁気ディスクへのアクセスにおいてヘッドを大きく動かす必要があるときに、途中に別のアクセス要求のあるシリンダがあれば順に寄って移動するようなスケジューリングを行うことにより、全体としてのアクセス時間を減らす。デバイスを直接制御するデバイスドライバが提供するデータ構造やアクセス機構を、ユーザプロセスからじかに使うこともできるが、デバイスドライバとユーザプロセスの間に、データ構造やアクセス機構を変換するモジュール(フィルタ)を介して、より高度なユーザインタフェースをもつ入出力システムを構成する手法もある。UNIX ではこのデータの流れをストリームと呼んでいる。ストリーム入出力システムの概念を図1・14に示す。ストリーム入出力システムでは、ユーザプロセスからデバイスに向かう流れをダウンストリーム、逆をアップストリームと呼ぶ。ストリーム入出力の考え方は、ネットワークの物理的なデバイス制御のためのドライバから、階層的なプロトコルのレベルに応じた仮想的なデバイスのドライバを順に実現する場合にも有効である。
