# 1・2・3 並列処理の技術的問題

> **章**: 第1章 情報システム
> **節**: 1・2 オペレーティングシステム

a. 並列処理におけるプロセス間同期複数のプロセスが互いにデータの受渡しなどを行いながら,見掛け上同時に処理を遂行することを並行処理(concurrent processing)と呼ぶ.複数のプロセスを複数の実際のCPU を用いて実行することを並列処理(par-allel processing)と呼ぶ.さらに,ネットワークで結合された複数の計算機上で処理が行われるのを分散処理(distributed process-ing)と呼ぶ.並列処理では,複数のプロセスが相互に協調し合って目的の処理を進めるので,相互に情報を交換する必要がある.このためのモデルには大きく分けて,(1) 共有記憶(shared memory)モデル(2) プロセス間通信(inter-process communication)モデルがある.これらのモデルは抽象的な計算モデルであるとともに,並列計算機のハードウェアアーキテクチャのモデルにもなっている.抽象モデルの代表的なものに,共有記憶モデルとしてはPRAM(parallel random access machine)モデル,プロセス間通信モデルとしてはCSP(communicating sequential processes)モデルがある.ハードウェアのモデルとは独立に,適当なソフトウェアを介することによって抽象計算モデルを使用することができる.例えば分散処理システムはハードウェア的には共有記憶をもたないが,分散共有記憶(distributed shared memory)として,見掛け上共有記憶と同様に見えるようにできる.共有記憶のハードウェアアーキテクチャをもつ並列計算機にも,どのCPU も全主記憶に同じようにアクセスできるUMA(uni-form memory access)と,メモリアクセスの時間などが一様でないNUMA(non-uniform memory access)がある.また,どのCPU も同様に,いろいろな処理を対等に分担するようになっているシステムをSMP(symmetric multi-processor),対称型マルチプロセッサと呼ぶ.一般に,複数のプロセスが一つの資源を共有する場合,例えば2人のユーザのプリンタ出力が行われた場合などには,お互いの出力が交じり合わないように制御する必要がある.並列処理では,さらに複数のCPU で実行されるプロセスが協調して目的の仕事(タスク,ジョブ)を処理するため,次のような問題が起こる.(1) 相互排除問題(mutual exclusion problem)(2) 信号送受同期問題(conditional synchronization)これらは,次のような具体的問題を用いて論じられてきた.(3) 生産者-消費者問題(producer-consumer problem)(4) 読み書き問題(readers-writers problem)(5) 哲学者の食事問題(dining philosopher problem)(6) 哲学者の飲酒問題(drinking philosopher problem)(7) ビザンチンの将軍問題(Byzantine general problem)相互排除とは,共有記憶モデルにおいて複数のプロセスが同一の資源を利用する際に,同時にはたかだか一つのプロセスしかアクセスを許さないようにすることである.例えば,複数のプロセスによって累算を行う場合に,一つのプロセスがそのときまでの累算値を読み取ってその値にさらに加算値を加えて書き戻す間に,他のプロセスが累算の処理に入らないようにすることである.共有資源にアクセスする部分を危険部分(critical section)と呼ぶ.図1・15に相互排除問題の例を示す.共有資源である変数にアクセスする部分が危険部分である.危険部分を実行する前に,各プロセスは共有資源を占有し,危険部分を出る前にそれを解放する.OS には相互排除を行うための命令が用意されており,ロック/アンロック(lock/unlock)命令は代表的なものである.あるプロセスが変数を占有している間は他のプロセスはを占有することができないので,の値が逐次的に更新される.図1・15 相互排除問題の例図1・16 デッドロック7―9占有のためのロック/アンロック命令自身も共有変数を読み書きすることによって実現されるので,相互排除の解決には相互排除動作を実現したハードウェア命令が必須である.そのような命令には,(1) テストアンドセット(test and set)命令(2) リードモディファイライト(read modify write)命令(3) コンペアアンドスワップ(compare and swap)命令(4) フェッチアンドアッド(fetch and add)命令などがある.これらの命令はすべて情報を読み出し,かつ変更する動作を,非可分操作[indivisible operation,アトミック操作(atomic operation)ともいう]として実現する命令である.テストアンドセット命令は,メモリ中の1ビットが0か1かを判定すると同時にそのビットを1にセットする命令で,この命令を一つのCPU が実行している間は他のCPU はそのビットにアクセスすることができない.これらのハードウェアの命令は,そのまま使うこともあるが,実際の並列処理ではこれらの命令を利用してOS に用意されているロック/アンロック命令やバリア(barrier)命令,あるいはセマフォ(semaphore)やホーア(Hoare)のモニタ(monitor)などの仕掛けを使ってプログラムを作成する.セマフォシステムは,プロセス間の相互協調の状態を整数値で保持するセマフォ変数,ならびに非可分操作でその値を増やす命令と減らす命令とで構成される.共有記憶をもたないシステムで並列処理を行う場合は,同期と通信にプロセス間通信を用いる.実用的には,CSP,UNIX のソケット通信,Adaのランデブ機構(rendezvous mechanism),PVM(parallel virtual machine),MPI(message passing interface)などのライブラリが用いられる.プロセス間通信による並列処理では,基本的にはデータやメッセージの送出(send)命令と受取(receive)命令を用いて通信を行う.通信を行うことが同時に関係するプロセス間の同期をとることになる.b. デッドロック(deadlock)二つのプロセスが,互いに相手の占有する資源が解放されるのを待ち合ったり,相手からの通信を待ち合ったりして,永久に停止してしまう状態をデッドロックという.これは三つ以上のプロセスの間でも起こりうる.その例を図デッドロックが生じる条件は,プロセスが資源を占有しながら他の資源を要求し,他のプロセスを待ち合うという関係が循環的になっていることである.したがって,この条件を満たさないように,資源間に優先順位を決めるなどあらかじめ資源割当て法を決めてデッドロックを予防(deadlock prevention)するか,あるいは資源割当て時にデッドロックが生じないように割当てることにより,デッドロックを回避(deadlock avoidance)することが必要である.予防や回避のために資源を余分に使いたくなければ,デッドロックが生じたことを検出して除去する方策をとる場合もある.しかし,デッドロックが起きてしまうと,除去するためにプロセスから強制的に資源を解放するため,当該プロセスの処理の破棄につながる.また,トランザクション処理などにおいては,単にデッドロックが生じないだけでなく,並列処理が正当であることの性質として,その処理結果が個々のプロセスを順次実行した場合と同じ結果になるという,順次化可能性(serializability)が求められる.7―10情報・ソフトウェア쑯A:アプリケーションプロセス쑯M :ミドルウェアプロセス쑯S :システムサービスプロセス図1・17 マイクロカーネル構法による分散処理システムc. 並列/分散処理OSマイクロカーネル構法を用いた分散処理システムの概念を図1・17に示す.쑯A はアプリケーションプロセス,쑯M は並列/分散処理環境を提供するミドルウェアプロセス,쑯Sはシステムサービスプロセス(システムサーバをミドルウェアに含めることもある)を表す.個々のハードウェアはCPU,メモリ,ネットワークコントローラなどを含む処理システムであるが,並列/分散処理システムにおいては,ノード(node),処理単位(processing unit),サイト(site)などと呼ばれる.マイクロカーネル構法は,システムサーバやミドルウェアがモジュール化されるため開発などがやりやすく,サーバも他のプロセスと同じように扱われるので並列/分散処理しやすいが,実行時にはプロセス間通信が多くなるため処理が遅くなる.そのためもあって,マイクロカーネル構法を用いずに,サーバ,ミドルウェアの機能をカーネル内にもつOS や,サーバをカーネル内にもつOS とユーザプロセスとしてのミドルウェアを組合わせたシステムもある.並列/分散OS の目的は,ユーザの使い勝手を損なうことなく,多量の情報,資源の可用性を高め,信頼性,性能の向上を達成することである.そのため,位置透過性(location transparency)を高めるための大域的名前付けや,負荷分散のためにプロセス移行(process migration)を行うなど,大域的プロセス管理の機能の性能向上,透過性向上が図られている.また,並列計算機におけるキャッシュコヒーレンシと同様,分散共有記憶における情報のコンシステンシを保つことが重要である.透過性や可用性,処理速度の向上は必ずしも互いに両立しない.そのため,ある程度他の性能の低下があってもユーザの望む性能を優先することができるようにすることもある.例えば,処理速度を重視する場合には,ユーザ空間のデータをノード間でコピーする際にシステム空間を経ることなく直接コピーして通信時間を減らす仕掛けを提供することなどがある.このために,ユーザがシステム領域を読み書きできるようにする方法や,逆にユーザデータにネットワークコントローラがアクセスするために,ユーザデータ領域を物理メモリに固定(pin down)する方法などがある.分散OS の例には,LOCUS,Mach,Chorus,VSystem,Amoeba,Plan9 やSolarisに影響を与えたSpring などがある.
