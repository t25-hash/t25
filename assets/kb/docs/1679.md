# 1・2・4 リアルタイムOS(real-time operating system)

> **章**: 第1章 情報システム
> **節**: 1・16 に示す.

a. リアルタイム性とそのカテゴリ計算機によってさまざまな計算処理を行う場合,その計算処理時間に強い時間的な制約がある場合とない場合がある.例えば,集積されたデータに統計処理を施したり,全地球的な大気循環のシミュレーションを行う,といった計算処理は,その処理ができるだけ短時間で行われることが期待されるにせよ,その処理結果が得られるべき時刻に強い時間的な制約があるわけではない.その計算処理を行わせたユーザは,その計算結果が出るまで単に待てばよい.一方,例えば機械システムのサーボ系をソフトウェアで構成する場合には,その計算には強い時間的な制約がある.ソフトウェアで実現するサーボ系では,数ミリ秒,あるいはそれより短い時間の周期ごとに同様の計算処理を繰返し行うソフトウェア構造を実装する.この場合,一回のサーボ周期がめぐってきたというイベントにおいて行われるべき計算処理は,遅くともその周期の時間内に終了しなくてはならない.あるいは,そのイベントの発生時刻から,その周期時間の10%以内の時間でその計算処理を終了させなければならない,というような時間制約がつく場合もあろう.このように,その計算処理を開始すべきイベントが発生してから,定められた時間内にそのイベントに対応する計算処理を正しく行うことが要求されるのが,リアルタイムシステム(real-timesystem)の特徴である.航空機の自動制御やNC(numerical con-trol)工作機械の制御,医療機器の制御,ロボットの制御など,一般に機械システムの制御を,計算機を利用してソフトウェアで実現する場合には,その系はリアルタイムシステムとして実現されることになる.その計算処理を行うイベントが発生してから定められた時間内に計算処理を正しく終了するという観点で見るとき,その計算終了に要求された時刻を守る制約がどれほど厳格であるかにより,次のようなカテゴリ分けがなされている.. ハードリアルタイムシステム(hard real-time system)そのイベントの発生に対応する計算処理が完了されるべき時刻が厳格であり,その計算処理がその時刻を越えてしまうとシステム全体に致命的な影響を与えてしまうシステム.. ソフトリアルタイムシステム(soft real-time system)そのイベントの発生に対応する計算処理が完了されるべき時刻が定められており,その時刻を守るようにその計算処理を行うが,その処理がその時刻を越えてしまっても,ただちにそのシステム全体の性能を決定的には悪くしないシステム.このシステムでは,その計算処理が定められた時刻に終了しなくても,ただちにその計算処理を中止するわけではない.. ファームリアルタイムシステム(firm real-time system)例えばマルチメディアにおける時系列圧縮画像データからフレームを再生する際に,何らかの理由によって時間内にそのフレームをつくり出すことができない場合,そのフレームを間引いてしまうことができる.このようなシステムをファームリアルタイムシステムと呼ぶ.すなわち,その計算処理が定められた時刻に終了しない場合,その計算処理の価値はないと判断し,その処理を強制終了できるのである.b. リアルタイムOS の必要条件リアルタイムOS では,そのOS の上での計算処理(タスク)が,ハードリアルタイムの意味で実行されることを要求されたとき,そのタスクの実行がその意味において保証されることが期待される.もちろんこのリアルタイム性を保証しながらのタスク処理速度と同時実行可能なタスク数はその計算機の性能に依存するので,限りない数のタスクを限りなく短い時間で達成することは要求できない.したがって,(1)実行が要求されたタスクがすべてハードリアルタイムの意味において実行可能かどうかを判断し,かつ,(2)そのタスクに対するCPU 割当てスケジュールを計画し管理する,(リアルタイム)タスクスケジューリング(task scheduling)の理論が必要である.この理論に基づくタスクスケジューリングアルゴリズムが実装されていることが,リアルタイムOS であることの必要条件であるといってよいであろう.実際に,リアルタイム性を実現するタスクスケジューリングの理論やアルゴリズムは種々提案されており,文献(1)には,原典の引用とともにそれらの詳細が解説されている.一般にリアルタイムなタスクスケジューリングの理論は,(1) そのタスクを要求するイベントが周期的なもののみか,散発的(そのイベントが集中的に発生すると,しばらく発生がなく,また集中的な発生することをいう.通信におけるパケットの到着と情報の復元の場合などがこれに当たる)なものも含むのか否か(2) そのタスクを行うCPU は一つか複数か(3) そのタスクの優先度は固定(固定優先度)か,実行中に優先度を変化させることがある(動的優先度)か(4) 現在実行中のタスクA の優先度より高い優先度をもつタスクB の実行開始イベントが生起したとき,現在実行中のタスクA を中断してタスクB を実行開始(プリエンプト)できるか否か(5) そのタスクが他のタスクとの間に相互依存関係(他のタスクの計算結果がでるのをそのタスクが待つ,複数のタスクが共通のハードウェア資源を利用するために相互排除が必要,など)があるか否かなどの前提を種々考慮しながら構成される.代表的なスケジューリングアルゴリズムとしては,固定優先度,単一プロセッサ,周期的タスク実行,その周期時間が計算終了時間に等しい,相互に依存関係のないタスクのみなどの仮定のもとに設計されたレートモノトニックスケジューリング,その周期時間が計算終了時間に等しいとの仮定を外したデッドラインモノトニックスケジューリング,動的優先度スケジューリングの方法であるEDF(earliest deadline fast)スケジューリングやLL(least laxity)スケジューリング,タスク間に依存関係がある場合の動的優先度スケジューリングである優先度継承プロトコル,優先度シーリングプロトコルなどがある.タスク間に相互依存があるケースは,実用的なリアルタイムOSでは当然に仮定されるべきである.この場合,タスク間で優先度継承が行えることは,リアルタイムOS では必要な条件であるといってよい.これは優先度の逆転と呼ばれる現象を防ぐために必要である.優先度の逆転とは,(1)そのOS の上で実行されているタスクA,B,Cがあり,(2)最低優先度タスクCの実行結果待ちで最高優先度タスクA の処理を中断している状態で,(3)結果的に中優先度タスクB のみが実行され,(4)Cも中断されたままで,結果としてA にタスクの割当てがまわってこず,(5)最高優先度のAの処理が進まずにそれより低い優先度のB の処理のほうが進む,というような場合である.優先度継承は,この優先度の逆転を検出し,自動的にCの優先度を上げてA の処理を結果的に速く進ませるようにすることである.このためにも,そのOS がプリエンプト可能であるべきことは必要な条件となる.さらにリアルタイムOS の設計と実装に当たり,そのOS 自体にCPU 割当てがある期間においても,プリエンプト禁止や割込み禁止で処理を行う時間を極力減らすこと,また,ハードウェア割込みがかかってから,割込み処理が行われるまでの時間も極力短いこと(割込み反応速度が速いこと)などの工夫も必要である.
