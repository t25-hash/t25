# 1・3・3 データベース

> **章**: 第1章 情報システム
> **節**: 1・3 計算機ソフトウェア

データベースとは、各種のアプリケーションが利用するデータを統一的に収集管理し、容易に参照や共有ができるようにすることを目的とした、データの倉庫のようなものである。この機能を実現する基本ソフトウェアはデータベース管理システム(DBMS)と呼ばれ、アプリケーションに対してデータの一定の見え方を提供し、データの参照や保護などを行う。データを統一的に管理するうえでは、同種のデータを一つの型としてまとめる抽象化が有効である。このようにしてデータベース内のデータの構造や関係、制約などを記述したものをスキーマと呼ぶ。また、このスキーマに従って格納された個々のデータ群をインスタンスと呼ぶ。a. データモデル蓄積しようとするデータの構造やそのデータに適用可能な操作の集まりをデータモデルと呼ぶ。データモデルには、データのもつ構造の記述の規約、データに対して可能な操作のセット、データが矛盾なく現実の世界を表現すために必要な制約の三つが要素として含まれる。. ネットワークデータモデルこのデータモデルでは、個々の要素をデータの並びであるレコードとして表現する。また、レコード間には親子関係を定義することができて、ループ状のポインタを用いて一対多の親子関係を実現することができる。このデータモデルでは、このようにしてあらかじめ用意されたデータの間を、カーソルと呼ばれるポインタで順次走査していくことによって、必要なレコードを発見することができる。. 階層データモデルこのデータモデルでは、個々の要素を表すレコードを節とする木構造を基本単位とする。しかし、これだけでは一対多の関係を表現するのは容易であるが、多対多の関係7―15を表現するには効率が悪いため、異なった木に属するレコード間をつなぐ論理的な関係を導入して、効率的な運用を図っている。. 関係データモデルこのデータモデルはリレーショナルデータモデルとも呼ばれ、ここでは関係(リレーション)を中心としてデータベースを構築する。関係とは表のようなもので、表中の各列に並ぶ事柄を属性と呼ぶ。一方、各行はタプルと呼ばれ、このタプルの集まったものが関係データベースのインスタンスとなる。このモデルにおけるデータ操作は関係代数と関係論理と呼ばれる形式的な体系によって規定されている。関係代数は関係に対する基本的な操作を行う代数演算を定義する。一方、関係論理は1階述語論理に基づく。b. 関係データベース(relational database)関係データベースは関係データモデルに基づくデータベースであり、現在広く使われているものである。. 関係代数関係代数は関係データベースのデータ操作の理論的な基礎となるもので、リレーショナル代数とも呼ばれる。基本的な関係代数演算子は5種類で、タプルの集合である関係の和と差、直積、さらにある関係のもつ属性のうち必要ないものを削除した関係を生成する射影と、ある関係のタプルの中からある属性の値と定数またはもう一つの属性の値を大小比較した結果、もしくはそのような複数の比較の結果を論理演算したものの真偽によってタプルを取捨選択した選択、の五つの演算である。これらは基本演算であり、これらの組合せで任意のデータ操作を合成することができる。. 関係データベース言語SQLSQLはISOの規格にもなっている標準的な関係データベースの操作用言語である。プログラミングのための利便性を図るために幾分か純粋な関係データモデルから逸脱したところもあるため、SQLでは関係を「表」、属性を「列」、タプルを「行」と呼び替える。基本的な問い合わせはSELECT 表。列、表。列、表。列FROM 表、表WHERE条件式という形式で行い、条件式のところに選択演算のための条件式を指定できる。また、条件式のところに問い合わせを含む式を書くこともできて、さらに複雑な問い合わせを行うこともできる。c. 知識ベース(knowledge base)1階述語論理に基づき、事実に関する知識と事実から新しい事実を導き出すルールとを組合わせることによって、与えられた質問を証明する能力をもったシステムをいい、その代表的なものは1・3・2・f・i項のPROLOG 言語処理系である。このシステムでは知識やルールは述語により記述され、これらを順次用いて最終的には与えられた質問の否定から相矛盾する二つの結論を導出することができることを示して、最初に与えられた質問が真であることを証明する。この過程を分解証明法、もしくは導出原理と呼ぶ。d. アクティブデータベースアクティブデータベースとは、データの参照や変更などの何らかの契機があったときに、データベース側からアプリケーション側に対して何らかの働きかけをする能力をもつデータベースを意味し、データモデルによる分類とは異なった見地からの分類である。アクティブデータベースの行う働きかけには、データベース内のデータの更新やアプリケーションに対するメッセージの送信などが考えられ、データやイベントによって駆動されるアプリケーションの中核を構成する。e. 同時実行制御データベースは複数のアプリケーションに対して情報を提供することを要求されるが、複数のプロセスがデータベースを操作する場合、互いに関連する複数の記憶をすべて一瞬で更新することができないため、タイミングによっては整合性のない情報を他のアプリケーションに利用させてしまう可能性がある。同様に、複数のアプリケーションが同じ記憶を変更しようとした場合や、データベースの更新途中にシステムの障害が起こった場合には、記憶内に永続的な矛盾を残してしまう可能性がある。このような事態を避けるのが同時実行制御であり、割込まれては7―16情報・ソフトウェア困る一連の動作をトランザクション(transaction)として他の操作から分離する。他のアプリケーションに対しては、操作の前または後の状態だけが存在してその中間の過渡的な状態にはロックを掛け、あたかも存在しない過程と見せかけることによって他のアプリケーションとの干渉を排除し、かつ耐障害性の強い、つねに整合性を保った動作を行わせている。. デッドロックデータベース内のデータを更新する場合には、そのデータを他のトランザクションがアクセスしないようにロックを掛ける必要があるが、複数のトランザクション処理が同時に進行する場合にはデッドロック(deadlock)が発生する可能性がある。デッドロックとは、同時に複数のトランザクションがすでに獲得したデータを解放しないまま新しいデータを獲得しようとするときに起こる、すくみ状態である。一度この状態に陥ると、これらのトランザクションは処理を進めることができなくなり、また他のトランザクションも競合している資源のロックが外れるのをいつまでも待ち合わせることになる。データベース管理システムでは、例えば、トランザクションの発生時刻を記録してその順に順序付けを行い、ロックの競合が起こりそうなときには後から開始されたトランザクションを中止させてすくみ状態が起こらないようにしている。このため、一度中止されたトランザクションも同じ発生時刻で再処理を繰返せば、いずれは必ず中止されることなく完了することができる。. コミット何らかの理由でトランザクションを無事に完了できなかったプロセスがあると、ロールバック(roll back)と呼ばれる不完全な処理を取り消して開始前の状態を回復する処理を行わなければならない。この実現の方式の一つがコミット(com-mit)である。データベースの処理に用いられるコミット動作は、あらかじめトランザクションによって更新する前と更新した後との双方の状態をログもしくはジャーナルと呼ばれるファイルに記録し、このファイル書込み動作が完了後に本当に変更を反映させたい場合には、コミットする旨をログに書き込んでからデータベースを更新する。どの段階で中断されていたとしても、ログにコミットが記録されていれば操作後の状態を、コミットされていなければ操作前の状態を、ログに書き込まれた更新前後の状態のいずれかから回復することができる。なお、障害が度重なって発生してもよいよう、ログの中には差分情報ではなく実際にデータベース内に残るべき情報を含める必要がある。
