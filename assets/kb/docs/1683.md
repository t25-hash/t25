# 1・3・4 ソフトウェア実行環境

> **章**: 第1章 情報システム
> **節**: 1・3 計算機ソフトウェア

a. 分散システム分散システム(distributed system)とは,ネットワーク上に分散して配置される計算機群をあたかも一つの計算機であるかのようにユーザに見せかけるシステムであり,さらに複数の計算機がもつ計算能力を生かすとともに単一の計算機の故障がシステム全体に致命的な影響を与えないようにする能力をもつ.この目的のために,分散システムでは情報を交換したり共有したりするので,プロセスどうしの間で通信を行い,互いにタイミングを図り,他の計算機上でプロセスを起動したり,すでに起動されたプロセスを他の計算機へ移送したりできる必要がある.このためには,システム内のいろいろなオブジェクトに統一的な名前付けの体系を設定することが肝要である.. デッドロック分散システム上では複数の計算機上のプロセスが並行に動作しながら,システム内の各種の資源を共有していく必要があるため,1・3・3・e項のデータベース同様,つねにデッドロックの危険と隣合わせることになる.このような状態にならないようにするため分散システムでは,一度起こってしまったデッドロックを検出してそれを解決する能力,もしくは資源を要求してもすぐに獲得させないことによってデッドロックを回避する能力,もしくはデッドロックが起こり得ないようにシステムを構成することが必要となる.例えば,すべての資源に一元的な順序を付け,プロセスが資源を獲得する際にはこの順序に従って要求することとすると,デッドロックが起こり得ないことが知られている.. 分散システムにおける並行性制御複数の計算機上のプロセスがデータを操作する場合,互いに関連する複数の記憶が複数の計算機にまたがる可能性があるため,単一計算機内で運用されるデータベースなどとは異なった制御を行う必要がある.このような事態を避けるのが並行性制御(並列性制御,concur-rency control)であり,単一計算機内での場合にはコミット操作を行うが,分散システムにおいては複数の計算機で足並みをそろえてコミット操作を行うための2フェーズコミットプロトコルが用いられる.. 2フェーズコミット複数の計算機間で行うコミットのためのプロトコルであり,システムに一つの調停者が関連する計算機上に一つずつ存在する代理人と通信しながらコミットを進める.動作は二段階からなり名前の由来となっている.まず最初のフェーズでは,調停者が各代理人に準備メッセージを送る.各代理人は各自の行うべき処理の更新する前後の状態をログファイルに記録し,この書込みが完了すればログファイルに準備完了を記録したのち準備完了メッセージを調停者に返信する.うまく準備できなければ中止メッセージを送る.調停者は全代理人から準備完了を伝えられれば次のフェーズに進むが,時間切れになったり中止メッセージを受けたりした場合は,全代理人に中止を伝え,ログファイルに中止を書き込み,代理人からの中止完了メッセージを待つ.次のフェーズでは,調停者はログファイルにコミットを記録し,代理人にコミットメッセージを送る.代理人はメッセージを待ち,それがコミットであればログファイルに従って更新後の状態を実現した後,調停者に処理完了通知を送る.一方,代理人の受けたメッセージが中止メッセージであれば,更新前の状態を実現した後,調停者に中止完了を伝える.調停者は全代理人からの処理完了通知を待ち,その後ログファイルに完了を書いて一連の動作を終える.この操作が調停者側で中断された場合,調停者がログファイルにコミットを記録する前の中断であれば代理人に中止を伝えてアトミックオペレーション自体をなかったことにする.ここで,アトミックオペレーションとは,意味的に一連でかつ分割不可能な動作のことである.この操作を行っている主体以外から見たときには,全く着手していないかあるいは完全に終了しているかのどちらかとして観測され,その動作を行っている途中の状態としては観測されないことが保証されているような動作をいう.並行性制御の基本となる概念である.調停者がログファイルに完了を記録する前であれば代理人にコミットを送るところから再開できる.一方,代理人側で中断された場合,代理人のログファイルに準備完了が記録されていれば,更新の前後の状態の準備ができているので,他の代理人の不都合がなければアトミックオペレーションを完了できる.代理人のログファイルに準備完了を記録する前の中断であればこのオペレーション自体をなかったことにする.b. セキュリティ分散システムでは,システムが物理的に広い範囲に分布し,通信回線などの第三者から守りにくい構成要素をも含むため,セキュリティの維持が問題となる.セキュリティには,アクセス制御・通信情報保護(暗号,ディジタル署名)・ユーザ認証・ホスト認証などがある.. アクセス制御システム内の資源をアクセスすることを許可された利用者の名前と,許可された利用の仕方とを並べたものをアクセス制御リストと呼ぶ.資源をアクセスしようとするものはそのつどこのリストで確認された後アクセスを許される.許可されたアクセスの仕方を変更したり,ある資源に誰がアクセスすることができるかを知るのは容易だが,頻繁にアクセス制御リストを参照する必要があるため,アクセスが遅くなる場合がある.一方,資源ごとに利用許可証をユーザに発行し,この許可証を示したユーザに対してアクセスを許す方式があり,この許可証をケイパビリティと呼ぶ.利用者は一度ケイパビリティを提示すればその後はアクセス権限の確認をする必要はなく実行効率は良いが,一度許可したアクセスを後から取り消したい場合に備えた実現が必要になる.. 暗号暗号化(encryption)とは,鍵なしでは解読できないように情報を変換するものであり,変換された暗号文から元の平文を復元できないようにするとともに,第三者が暗号文を偽物とすりかえたことを検出できることが必要である.このとき,暗号化と解読とに使う鍵が同じ(もしくは片方から他方を容易に知ることができる)ものを対称暗号化方式もしくは共通鍵暗号(commonkey cryptosystem)方式,異なる鍵を使うものを非対称暗号化方式もしくは公開鍵暗号(public key cryptosystem)方式と呼ぶ.共通鍵暗号(例えばDES 暗号など)では鍵を公開する必要がないので,鍵を容易に作ることができ暗号の強度も強いが,通信相手に鍵を送るためには特別の配慮が必要となる.一方,公開鍵暗号(例えばRSA 暗号など)では暗号化鍵のみを公開して自分に通信したい相手にはこの鍵で暗号化してもらうが,解読鍵は秘密のままにしておくためこれを公開鍵から容易に計算できてはならない.現在の公開鍵暗号では素因数の積の因数分解を基本にしており,大きな素数二つの積は容易に求められるが,逆に積から素因数を求めるのは非常に困難であり時間が掛かることを利用している.. ディジタル署名暗号化された通信を受け取ったとき,受信者は通信に添付された発信者の署名(signature)から,その通信が第三者によって発信されたものではなく確かに署名者の発信したものであることを証明できなければならない.この目的で通信に付加されるものをディジタル署名と呼ぶ.公開鍵暗号を用いたディジタル署名では,署名者が自分の秘密にしている解読鍵を使って送ろうとする通信を変換し,署名者だけが作成できるディジタル署名として相手に送信する.これを受信者側が署名者の公開されている暗号鍵を使って変換し,無事に平文に変換することができれば,その暗号化は送信者の公開鍵に対応する秘密鍵で行われたことが証明され,確かに署名者が署名したものであることが判明する.. ユーザ認証計算機を利用しようとする個人を特定するのがユーザ認証である.通常はパスワードなどの利用者個人が秘密にしている情報を計算機に提示することによって行うが,第三者がパスワードを試行錯誤によって探すという攻撃に耐えられなければならない.現在の計算機の処理速度では5文字程度のパスワードを攻撃するのには数時間程度といわれ,またパスワードとしては人間が覚えやすいものを選ぶ傾向が強いので,単語辞書を用いるとパスワードのうちの数割を破ることができるともいわれる.このため,誤ったパスワードを数回入れてしまうとログイン手続きをしばらく受け入れないことも行われる.. 通信相手の認証自分が通信している相手を特定しようとするのが通信相手の認証が目指すものであり,第三者が通信を傍受して自分の意図する通信相手になりすますことを防ぐものである.このことは公開鍵暗号のディジタル署名を用いると次のように行える.認証を行おうとする計算機は,まず適切なメッセージを平文で作成し,これを自分の秘密鍵で変換して自分だけが生成できるディジタル署名を作る.さらにこれを認証したい通信相手の公開鍵を使って変換し,望みの通信相手だけが解読できる暗号文にしてその相手に送信する.認証される計算機は受信した通信文を自分の秘密鍵で解読して発信者のディジタル署名を得ることができ,これを発信者の公開鍵で変換して平文にすることができれば,その通信が確かにその発信者から送られたことを確認できる.これを相互に行って,通信しようとする双方が互いを確認できる.7―17
