# 1・4・4 TCP/IP におけるアプリケーションプロトコルの例

> **章**: 第1章 情報システム
> **節**: 1・4 ネットワーク

ここでは,TCP/IPにおけるアプリケーションプロトコルの例として,電子メール(SMTPとPOP)およびWWW(HTTP)を取上げる.a. SMTPまず,TCP/IPで用いられる代表的な電子メールプロトコルSMTP(simple mail transfer protocol)について簡単に述べる.詳細については,文献(3)を参照されたい.このプロトコルでは,図1・28に示すクライアント・サーバモデルを仮定している.クライアントは,ユーザから送信された電子メールを受取り,指定された宛先に示された受取り先のサーバにこの電子メールを送付する役割を果たす.サーバはこれを受取り,自分が管理するユーザ宛であることを確認してから,そのユーザ用に設けられた所定のファイルシステムに電子メールを書き込む.このように,SMTPにおけるクライアントとサーバは電子メールを転送する動作をすることから,MTA(mail transfer agent)と呼ばれ,ユーザとMTA との間のインタフェースとなるMUA(mailuser agent)とは区別される.SMTPの通信は,クライアントからサーバへコマンドを送り,それに対して,サーバがクライアントに対して応答を返すことで行われる.クライアントから送られるコマンドは,コマンドの種類を表すアルファベットで始まる文字列で構成される.これに対して,サーバからの応答は,応答の種別を表すコード番号から始まる文字列で構成される.コマンドと応答はともに,改行文字CR(ASCIIコード13)とLF(ASCIIコード10)で終端される.図1・29に,taro@gama.ac.jp からhanako@frog.ac.jp とichiro@frog.ac.jpに同一内容の電子メールをSMTPにより送信したときの例を示す.ここで,「S :」で始まる行はサーバからクライアントへ,「C:」で始まる行はクライアントからサーバへ送られたメッセージを示す.以下では,この例を用いてSMTPの動作を説明する.図1・28 SMTPのモデル図1・29 SMTPでの通信例(1) クライアントがサーバにアクセスすると,サーバよりメッセージが送られる.ここで,200番台の応答が返されているが,これはpositive completion replyと呼ばれ,実行が正常終了し,次のコマンドが受入れ可能であることを意味する.これに対し,異常時には400番台(transient negative completion reply)や500番台(permanent negative completion reply)の応答が返される.(2) 次に,クライアントが電子メールを送付することをサーバに伝える.EHLOはExtended HELLOの意味である.SMTPの最初の仕様書RFC821ではEHLOの代わりにHELO(HELLO)のみが定義されていた.その後,SMTPに対してさまざまな拡張機能が加えられており,これらの機能を用いたいときは,クライアントからHELOの代わりにEHLOを送信する.これにより,HELOしか受付けることができない古いSMTPに対する互換性を保つことができる.なお,EHLOの後にはクライアントのFQDN(fully-qualified,domain name)が続く.(3) EHLOに対して,サーバはどの拡張機能が利用可能であるかを応答する.ここでは,PIPELININGと8BITMIMEの利用が可能であることがわかる.前者は,以下の(4)と(5)でクライアントからサーバに送られる(この例では三つの)メッセージを一度に送付し,それらの応答をサーバからクライアントに一度に返す機能である.これにより,クライアントとサーバ双方の待ち時間が短縮される.後者は電子メール内に最上位ビットが1の文字を含むことを認める拡張である.SMTP本来の仕様では,ASCII以外の文字の使用は認められていない.また,この拡張を用いるためには,(4)にBODY キーワードを追加する必要がある.(4) 次に,クライアントはMAIL FROM :で送信者が誰であるかをサーバに通知する.(5) そして,クライアントはRCPT TO:で電子メールの受信者をサーバに通知する.ここでは,2名に対して通知を行っている.(6) 次に,電子メールの内容を送付することをサーバに宣言する.これに対して,サーバは300番台の応答(positive intermedi-ate reply)をしているが,これは,正常終了したが,続くデータを要求していることを意味する.(7) クライアントは,まず,電子メールのヘッダを送付する.(8) そして,空行を入れた後に,電子メールの本文を送付する.(9) 本文の最後に,ピリオド「.」だけの行を送付する.ここで,ピリオドで始まる行を本文として送るときは,クライアントはその前にピリオドを一つ挿入し,サーバはこれを除去するという作業が行われる.これにより,ピリオドだけの行を送付することが可能である.(10) 最後に,クライアントはQUIT を送付し,SMTPの接続を終了する.b. POP次に,もう一つの代表的な電子メール用プロトコルPOP(post office protocol)バージョン3について簡単に述べる.詳細については,文献(13)(14)を参照されたい.POPは,SMTPなどでメールサーバに到着した電子メールを読み出すためのプロトコルである.このプロトコルは,ユーザの端末が常時ネットワークに接続されていない場合や,SMTPサーバを準備できない場合などに用いられる.POPは電子メールの転送とサーバからの削除を主目的としており,より複雑な機能が必要な場合はIMAP(internet message access protocol)などを利用する.POPの通信は,SMTPと同様に,クライアントからサーバへコマンドを送り,それに対してサーバがクライアントに対して応答を返すことで行われる.クライアントから送られるコマンドは,コマンドの種類と(必要ならば)引数を表す印字可能なASCII文字列で構成される.これに対してサーバからの応答は,肯定応答ならば「+OK」で,否定応答ならば「-ERR」で始まる文字列で構成さ7―23れる.応答に続けてデータが送付される場合は,最後にピリオドのみを含む行で終了する.ただし,ピリオド「.」で始まる行をデータとして送付する場合は,その前にピリオドを一つ挿入して送る.なお,コマンドと応答の行は,いずれも改行文字CR(ASCIIコード13)とLF(ASCIIコード10)で終端される.図1・30に,電子メールをPOPにより取出した例を示す.ここで,「S :」で始まる行はサーバからクライアントへ,「C:」で始まる行はクライアントからサーバへ送られたメッセージを示す.以下では,この例を用いてPOPの動作を説明する.(1) クライアントがサーバにアクセスすると,サーバよりメッセージが送られる.ここで,サーバはユーザのAUTHORIZA-TION 状態に入る.(2) ユーザを認証する手段として,USER コマンドでユーザ名を,PASS コマンドでパスワードを送付する方法を実装することが必須となっている.しかし,この方法ではパスワードが平文で流れるため,これを第3者に傍受される危険性がある.このため,認証手段のオプションとしてAPOPコマンドが規定されている.このコマンドでは,図1・30に示すように,最初の引数にユーザ名(hanako)を,第2引数にダイジェスト値を示す.ダイジェスト値は,(1)でサーバから与えられたに囲まれた文字列の後ろにユーザのパスワードを連結した文字列を,MD5アルゴリズムによって変換することで求められる.MD5アルゴリズムの出力は128ビットの値であるので,これを16進数32けた表記して(図1・30では一部省略)サーバに提示する.サーバから与えられる文字列は,認証をするたびに変化する.これにより,パスワードを平文で送ることなくユーザを認証することが可能となる.さて,ユーザ認証に成功すると,サーバはTRANSACTION 状態に入り,クライアントは電子メールの閲覧が可能となる.(3) STAT コマンドにより,サーバに電子メールが何件届いており,合計何バイトあるかが示される.(4) LIST コマンドにより,各電子メールがそれぞれ何バイトであるかが通知される.また,LIST で番めの電子メールのサイズを取得することもできる.(5) 番めの電子メールを取出す場合は,RETR コマンドで行われる.(6) 番めの電子メールを削除するときは,DELE コマン図1・30 POP での通信例7―24情報・ソフトウェアドが実行される.ただし,ここでは削除マークが付くだけであり,実際の削除は後述の(8)まで行われない.(7) また,削除マークの解除は,RSET コマンドで行われる.(8) POPを終了する際には,QUIT コマンドが送信される.このとき,サーバはUPDATE状態に入り,削除マークが付いた電子メールをすべて削除する.ただし,AUTHORIZATION 状態でQUIT コマンドが実行されたり,QUIT コマンドを実行する前にコネクションが切断された場合はUPDATE状態には入らず,削除マークが付いた電子メールは削除されない.c. HTTP次に,HTTP(hypertext transfer protocol)について簡単に述べる.以下の記述は,おもにHTTPバージョン1.1に基づいている.HTTPは世界規模で分散しているさまざまな形態の情報を転送するために用いられるプロトコルである.このプロトコルの発達が,インタネットの利用を一般家庭にまで浸透させる大きな原動力となったと言っても過言ではない.HTTPにおける情報(リソース)は,URL(universal re-source locator)と呼ばれる以下の形式で指定される.http://ホスト:ポートパス?引数ここで,は省略可能であることを示す.先頭のhttpはHTTPで接続することを示す.ポートは,接続する際のポート番号で,省略した場合は80番が仮定される.パスは,接続先ホスト内にあるリソースの位置を表すパス名を示し,そのリソースに渡される引数が最後に付く.また,パスが省略された場合は,スラッシュ「/」が指定されたものとみなされる.ただし,パスなどで区切り文字などの特別な文字を使う場合は,そのASCIIコードを2けたの16進数で表し,「%」に続けて記述する.例えば,「」は%7Eとなる.また,「%」自身を表記する場合は,%25としなくてはならない.上記に示すURLを用いて,HTTPのクライアントはサーバに対して要求を送り,サーバはこれに対する応答をクライアントに返す.HTTPバージョン1.0では,サーバからの応答終了直後にコネクションを切断する実装が多かった.このため,要求と応答のたびにコネクションの開設と切断を行うこととなり,これに要するオーバヘッドが大きかった.バージョン1.1では,一つのコネクションで要求と応答を複数回繰返すことが容易にできるよう配慮されている.ここで,HTTPでクライアントおよびサーバから送られるメッセージは次のような形式を取る.まず,最初の行に要求または応答の種別を示す.必要であれば,その次の行からヘッダと呼ばれる付加情報を続けることができる.そして,空行を挟み,メッセージ本体を続ける.なお,メッセージ内の各行はいずれも,改行文字CR(ASCIIコード13)とLF(ASCIIコード10)で終端される.ここで,クライアントがサーバwww.gama.ac.jpに対し,ファイルindex.htmlを要求したと仮定する.このときのクライアントからサーバへの要求を図1・31に,これに対するサーバからの応答を図1・32に示す.以下では,この例を用いてHTTPの動作を説明する.まず,図1・31の1行めにおいて,クライアントはGET メソッド(method)を送っている.これは,その次に続くindex.htmlの内容を送信するよう要求するものである.最後に,HTTPプロトコルで用いられるバージョン番号が記される.GET メソッド以外には,リソースの情報のみを通知させるHEAD メソッドや,メッセージ本体にあるデータを指定したリソースに対する入力として送るPOST メソッド,そのリソースに対してどのような要求ができるかを問い合わせるOPTIONS などがある.GET メソッド以下の行は,すべてヘッダになる.ヘッダの最後は空行で終端しており,メッセージ本体はない.User-Agentはクライアントで使用しているプログラム名とそのバージョン番号が示される.Acceptはそのプログラムで受付け可図1・31 HTTP クライアントからの要求図1・32 HTTP サーバからの応答能なデータの種類を示す.図1・31ではテキスト(text)と画像(image)全般を受付け可能である.Accept-Encoding はレスポンスで使用できる符号化方式を示す.この例では,gzipによる圧縮を受付ける.Accept-Languageは,クライアントで利用できる言語であり,この例では日本語(ja)と英語(en)を指定している.ところで,英語に関してはさらに=0.5が指定されているが,この値はquality valueと呼ばれ,ユーザの好みを表している.最小は0,最大は1であり,大きいほど好ましいことを意味する.また,=0のときは特に,その指定が受付けられないことを示す.なお,この指定が省略されたときは=1が仮定される.最後のHostはクライアントがアクセスしたホスト名を示すもので,HTTPバージョン1.1では必須のヘッダとなっている.このため,一つのサーバで複数のサーバに見せかけることが容易に実現できる.例えば,www.gama.ac.jpというホストにnews.gama.ac.jpという別名を与えておき,クライアントから来たメッセージのHostの値がどちらの名前になっているかによって,サーバの挙動を変えることができる.このようなサーバはmulti-homed serverと呼ばれる.次に,図1・32に示したサーバからの応答について説明する.まず,最初の1行めにはサーバの応答の内容が示されており,サーバのバージョン番号と状態コード,およびそれに対する人間向け説明で構成される.状態コードは,100番台はサーバからの情報通知(処理継続中),200番台は要求に対する処理の成功,300番台は要求の実現にはさらに処理が必要であること,400番台はクライアントの要求エラー,そして500番台はサーバの処理エラーを意味する.2行めから空行の直前(7行め)まではヘッダであり,Dateは処理が行われた日時,Serverはサーバのプログラム名とそのバージョン番号,Last-Modifiedはそのリソースの最終修正日時を表す.また,Content-Lengthでメッセージ本体のサイズを表し,Con-tent-Typeでメッセージ本体の内容を示す.この例では,HTMLで記述されたテキストであることがわかる.空行以降(9行め以下)はメッセージ本体であり,クライアントから要求されたindex.htmlの内容が転送される.文献(1)
