# 2・2・1 リアルタイム処理システム

> **章**: 第2章 メカトロニクス
> **節**: 2・2 ソフトウェア

a. リアルタイム処理システムの特徴リアルタイム処理の特徴の一つは、結果の有効性が論理的な正しさのほかに、その結果を生成した時刻に依存するという点である。すなわち、与えられた時間制約を満たすということが、論理的に正しい結果を生成するということと同様に重要である処理をリアルタイム処理という。また、このようなリアルタイム処理を含むシステムをリアルタイムシステム(real-time system)と呼ぶ。実際のリアルタイム処理システムにおいても、時間制約を満たしていない高品質の結果より、時間制約を満たしている品質の低い結果のほうが有益であることが多い。例えば、衝突回避システムにおいては、接近しつつある障害物に対して、その正確な位置を遅れて知らせることには意味がない。それよりも、障害物のおおよその位置を時間以内に知らせることのほうが重要である。また、テレビ会議システムのようなマルチメディアデータを扱うアプリケーションにおいても、高品質な画像が遅れて届くより、時間以内に粗い画像が届くことのほうが望ましい。リアルタイム処理におけるもう一つの特徴は、速いということがあまり重要ではないということである。速いということは相対的な概念であり、リアルタイムシステムにおいて要求される時間予測性とは異なる概念である。現状では依然として、リアルタイム処理と高速な処理とが等価に扱われる場合が多いようであるが、これは明らかに間違いである。高速に処理を行うことによりさまざまな時間制約を満たそうという考え方は、平均パフォーマンスを向上させることによりシステムのパフォーマンスを向上させようとする従来の考え方の延長にすぎない。そして、このような概念に基づく方法では、個々の処理の最悪の場合のパフォーマンスを保証することができないので、リアルタイム性を満たすことができない。高速であるということが意味をもたないシステムの例として、VoD(video-on-demand)システムを挙げることができる。VoDシステムにおいては、要求される時刻より早くフレームが到着しても、そのフレームの再生される時刻には影響がない。むしろ、早すぎる時刻にフレームが到着する可能性がある場合、受信側でそれらを保持するためにより大きなバッファが必要となってしまう。一方、受信側での再生速度に間に合うのであれば、フレームが一定時間間隔で到着することが保証されていることのほうが望ましい。この場合、再生される画像には影響がないうえに、受信側のバッファ容量は非常に小さくて済むという利点がある。b. 時間制約多くのシステムにおいて、リアルタイム処理の時間制約(time constraint)は、処理を開始可能な最も早い時刻と、処理を終了しなければいけない最も遅い時刻により定義される。これらの時間制約のうち、前者の時刻をリリースタイム(release time)と呼び、後者をデッドライン(deadline)と呼ぶ。上記の2種類の時間制約に加えて、周期的にある処理を実行しなければならないような場合、毎回の処理におけるリリースタイムか7―30情報・ソフトウェアらのずれが重要であることも多い。このずれに関する時間制約をジッタ(jitter)と呼ぶ。それぞれの処理に対する時間制約は、その厳しさによって二つに分類することができる。まず、ある処理に与えられた時間制約を少しでも守れなかった場合にシステムに対する貢献度が即座に零になってしまうような時間制約をハードリアルタイム(hard real-time)という。一方、時間制約を満たせなかった場合でもシステムに対する貢献度が多少低下するだけで済むような時間制約をソフトリアルタイム(soft real-time)という。また、与えられた時間制約がいかにきついかということによっても時間制約を分類することが可能である。例えば、時刻0から30の実行時間が必要な処理に対してデッドラインが31に設定されていた場合、その時間制約はきつい(tight)という。それに対して、デッドラインが60に設定されていた場合、その時間制約は緩い(loose)という。c. 時間予測性複雑な時間制約を要求するリアルタイムシステムを実現するためには、ハードウェアおよびソフトウェアのすべての処理に時間予測性(predictability)が必要である。時間予測性とは、ある特定の処理が予測された時刻とどれだけ近い時刻に終了するかということである。ある処理の終了する時刻があらかじめ予測されたものとほぼ同じであれば、その処理の時間予測性は高いということになる。逆に、同じ処理の実行時間に大きなばらつきがある場合、その処理の時間予測性は低いということになる。. ハードウェアにおける時間予測性近年、ハードウェアの処理能力は飛躍的に向上したが、これらの技術は高い時間予測性を提供するためには障害となることが多い。例えば、現在のほとんどのプロセッサはキャッシュやパイプラインを備えている。この場合、平均的な処理能力は向上するが、キャッシュミスやパイプラインストールを生じた場合のペナルティも大きい。この実行時間の差が時間予測性を低下させる大きな原因の一つとなっているのである。同様にDMA,TLB、さらにはハードウェアによる投機実行などのプロセッサの処理能力を向上させるための技術も時間予測性の低下を招く。そのため、時間制約が非常に厳しいリアルタイム処理には、比較的単純なアーキテクチャのハードウェアが用いられることが多い。. ソフトウェアにおける時間予測性ソフトウェア開発に際しての問題点としては、まず、従来から用いられてきた開発言語には、時間制約を明示的に指定することができないものが多いということである。その場合、プログラマは個々の処理に対して十分に細かい粒度で時間制約を指定することができない。それに対して、新たな言語仕様を設計し、専用のコンパイラを開発するなどの方法をとっているものもある。しかしながら、これらの特殊な言語は一般的に普及しているとはいい難いのが現状である。開発言語と関連して問題となるのは、コンパイラの最適化である。今日のコンパイラには、実際のオブジェクトコードを生成する前に、プログラムの論理構造を解析し、実際の実行時にとりうる可能性の高い実行パスを予測するというアプローチをとっているものが多い。この方法では平均実行時間を短くすることが可能であるが、予測したパスを実行した場合と異なるパスを実行した場合とで実行時間に大きな差が生じてしまう。結果として、そのような処理の時間予測性は低下することになる。近年では単純な組込みシステムを除いて、ほとんどのシステムでマルチプログラミングを採用している。マルチプログラミングは複数のプログラムでCPU を共有するための方法であり、資源を効率的に使用することができる。しかし、個々の処理の実行時間は他の処理の影響を受けることになり、リアルタイム性の保証はより困難になる。d. リアルタイムOS予測性の低下を避ける一つの方法は、リアルタイムOS(RT-OS : real-time operating system)を使用することである。従来のOS が平均パフォーマンスを最大にすることを目的としていたのに対して、RT-OS では個々の処理に対して時間予測性を提供することを目的としている。そのためにRT-OSでは最悪の場合のパフォーマンスを保証するためのさまざまな機構を提供している。. 優先度RT-OS では個々の処理に対して時間制約をもとにした優先度を割当てる。優先度は固定的であること(固定優先度、fixed priority)もあれば、システムの動作中に変化すること(動的優先度、dynamic priority)もある。. スケジューリング固定優先度を用いて個々の処理をスケジューリングするアルゴリズムの代表的な方法にRM(ratemonotonic)アルゴリズムがある。RM 法は周期的な処理のスケジューリングに適し、優先度は周期が短い処理ほど高い値となる。一方、動的優先度を用いるアルゴリズムの代表的なものにEDF(earliest deadline first)アルゴリズムがある。EDF法ではデッドラインの早いものに、より高い優先度を割当てる。. 優先度逆転問題と共有資源アクセスプロトコルRT-OS は、通常時には全処理の中から最も優先度の高い処理を実行することにより時間制約を満たそうとする。その一方で、優先度の低い処理により優先度の高い処理が制限なくブロックされる場合もあり、この状態を優先度逆転(priority inversion)と呼ぶ。優先度逆転はシステムの予測可能性を低下させ、デッドラインミスにつながる大きな要因であるため、できる限りこの状態を避けることが必要である。優先度逆転は、おもに資源を共有することから生じる問題である。例えば、ある処理がI/Oなどに関連した際どい領域を実行している場合、より優先度の高い処理であっても、同じI/Oを使用するためには、現在の処理がそのI/Oを使用し終えるまで待たなければいけないことになる。従来の時間制約を考慮しない同期機構を用いた場合、このような優先度逆転が生じている時間の長さをバウンドすることができない。一方、RT-OS ではPriority Inheritance Protocol,PriorityCeiling Protocol、さらにはStack Resource Policyなどの洗練された技術を用いて、ブロック時間が最小の長さになることを保証している。. 割込み処理予測性を低下させるもう一つの大きな要因として、I/Oからの割込みに対する処理が挙げられる。一般的に、割込みは任意の時刻に生じる可能性があり、その影響をバウンドするのは困難である。これらの影響を緩和するために割込みを無効にし、アプリケーションレベルでポーリングを行う形態をとっているシステムもある。また、割込みを使用する場合でも、その影響を極力小さくするために割込み処理を二分化しているものが多い。e. 分散リアルタイム処理複数のノードがネットワークを介して接続されリアルタイム処理を行うようなシステムを、分散リアルタイムシステム(distributed real-time system)という。このような場合、通信経路にリアルタイム性を保証することが必要となる。また、このようなネットワークにおける制約はQoS(quality-of-service)と呼ばれることがある。QoS はパフォーマンス、信頼性、さらにはシステムの再構成能力を含む広い概念である。例えば、パフォーマンスに関連するものでは、通信遅延やバンド幅などが挙げられる。QoS を保証するための方法には、ハードウェアにより優先度をサポートする場合と独自のプロトコルを用いる場合がある。また、これらのネットワークをリアルタイムネットワーク(real-timenetwork)と呼ぶ。f. リアルタイムシステムの構築実際にリアルタイムシステムを構築するためには、ハードウェアからアプリケーションまでのすべての時間制約を保証する必要がある。現状では、制御分野や人工知能などを統合的に扱った研究はまだ少ないが、今後はシステムのさまざまなレベルに存在する時間制約を考慮し、統合的に保証する方法が重要となるであろう。
