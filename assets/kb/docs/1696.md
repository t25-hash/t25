# 2・4・2 組込みOS

> **章**: 第2章 メカトロニクス
> **節**: 2・4 組込みシステム

a. 概要組込みシステムが機器の制御を行うためのソフトウェアの構成も、機器の多様性を反映して一様ではない。その中で、近年、組込みシステムの資源管理を行う基本ソフトウェア、すなわち組込みOS(embedded operating system)の重要性が高まっている。OS の機能は、ハードウェア資源を抽象化してソフトウェアの開発を容易化し、保守性・移植性を高め、またマルチタスク環境を提供するものである。近年の組込み製品は、マルチメディア対応やネットワーク接続など機能の高度化が進む一方で、そのライフサイクルは著しく短くなっており、市場投入までの開発期間の短縮への強い要求がある。組込みOS は、CPU を含めたハードウェアの操作を容易化する点でソフトウェアの生産性を高め、またミドルウェア(各種通信プロトコルやGUIモジュールなど)の利用を可能とするため、導入の機会は増加しつつある。しかしながら、前述のように組込みシステムの対象はさまざまな規模があり、OS を内蔵する必要がないほど小規模な応用や著しく高速な応答時間が求められる場合、あるいはメモリ容量への厳しい制限がある場合などでは、OS を利用することができないことも決して少なくはない。組込みOS は、機器の制御を対象とするシステムであり、ほとんどの場合、リアルタイムの事象に対応できることが必要である。リアルタイム性の要求の程度は制御対象により定められるが、一方で、CPU の性能向上が現今のように進展している中では、CPU 性能との対比において要求の困難さも決まってくる。すなわち、シングルタスクのOS あるいはOS なしでも、CPU が十分に高速ならば逐次処理を上手に書くことで所望の制御を比較的容易に実現できることもある。しかし、ある程度以上複雑な制御を行う際には、リアルタイム性を考慮したOS、すなわちリアルタイムOS を用いたほうがソフトウェアの生産性は向上する。b. リアルタイムOSリアルタイムOS(real-time operatingsystem)は、リアルタイムの事象を扱う前提で、通常のOS の機能のほかに、時間管理機能が強化されたOS である。時間管理とは、外部の事象に対して決められた時間以内で処理結果を出すこと、また決められた時間(時刻)あるいは指定された周期で繰返し出力を行うことなどが挙げられる。これらの機能を、複数の要求が同時並行して発生する中で実現することが求められる。通常、同時に発生する要求の数だけCPU を用意することはできないので、処理すべき要求を切換えながら実行することになる。このとき、要求に対応する処理をタスクと呼ぶと、複数のタスクに優先度を付け、時間的制約が厳しいものからCPU 資源を割当てるようにスケジューリングし、タスクの切換え(task dispatch、ディスパッチ)を行いながら実行する(マルチタスク)。タスクの発生は外部事象または時間の進行と関係し、任意時点であるため、すでに実行しているタスクを停止して優先度の高いタスクを実行できる(プリエンプティブ)必要がある。これらに関して、タスクの状態管理、スケジューリング、ディスパッチ(割込みを含む)などの処理を担当するのが、リアルタイムOS の中心部分と考えられ、カーネル(kernel)と呼ばれる。外部の事象に同期するためには、ハードウェアによる信号入力を受けて処理を起動する必要があるが、これをハードウェア割込み(interrupt)と呼び、その処理プログラムが割込みハンドラ、または割込みサービスルーチンである。一定時間ごとに周期的に実行するタスクを実現するには、ハードウェアタイマの割込み(タイマイベント)で時刻を管理してタスクを起動するように設定する。これらの割込みでは、該当する処理が実行される際の前後処理などがカーネルで行われる。複数のタスクが実行される際、一般にはタスク間の同期(syn-chronization)と情報交換が必要であり、同期・通信の機能が提供される。同期・通信の機構としては、通常、セマフォ、ミューテックスなどの相互排他を行う仕組みや、メッセージキュー、共有メモリなどのデータ交換の仕組みが用意される。これらの機能を使用したプログラムについて、全体の実行時間を予測したり、最適なスケジューリングを行うことは一般に不可能であるため、割込みの最大応答時間(maximum response time)やシステムコールの最大処理時間などがそれぞれの実装について示される場合が多い。したがって、開発者は開発環境のツールを使用して、プログラムの挙動を確認しつつ、要求を満たすプログラムを開発することになる。近年の組込み機器は、高度な通信機能やヒューマンインタフェースなどを内蔵する場合が多くなりつつあり、リアルタイムOS の基本機能だけでなく、さまざまな拡張機能に対応するソフトウェア部品も同時に用いられる。例えば、TCP/IP,Bluetoothなど通信モジュールや、Java VM などのGUI構築用モジュールも同時に利用されるため、多様な機能のソフトウェアコンポーネントを搭載できることも必要条件となってきている。以下に、組込みシステムに用いられるOS の例を示す。c. ITRONITRON(industrial TRON)は、機器組込み制御用のリアルタイムOS に関する仕様であり、1984年に産学協同プロジェクトが開始されて、1999年からITRON 4.0となっている。ITRON 仕様準拠の実装は、多くのメーカ等から販売・配布されている。2000年のアンケート調査では、最近開発された組込みシステムの中でITRON が使用された割合は3分の1以上とのデータもあるように、国内では普及率が最も高い組込みOS といわれている。組込みシステム用のリアルタイムOS への要求として、ハードウェアの性能を最大限に引出すこと、ソフトウェアの生産性向上、各種の対象システムへのスケーラビリティ、オープンな仕様などを挙げて、ITRON では次のような設計方針が設定された。それぞれのハードウェアに対するOS の適応化、組込みアプリケーションに対するOS の適応化、ソフトウェア技術者教育の重視、豊富な機能の提供、仕様のレベル分けなどの方針であり、また、すべてを共通化するのではない弱い標準化を基本としている。その一方で、移植性を考慮して厳格に決められたスタンダードプロファイルも用意されている。結果として、リアルタイム性、コンパクト、適応化、豊富な機能といった特徴が実現されたとしている。ITRON 4.0で用意されるサービスコール(service call、またはシステムコール)は166種類、静的API(初期化用)は20種類(そのうち、スタンダードプロファイルは、それぞれ71種類と12種類)であり、さらにスタンダードプロファイルのサブセットと追加機能を有する自動車用プロファイルがある。ロードモジュールは使用されるサービスコールだけリンクされるためメモリ容量を小さく保つことができ、カーネルだけならば、数kB(キロバイト)から数十kB とすることも可能である。サービスコールでは、タスク管理、タスク付属同期、タスク例外処理、同期・通信、拡張同期・通信、メモリプール管理、時間管理、システム状態管理、割込み管理、サービスコール管理、システム構成管理の諸機能が用意されている。カーネル周辺の仕様として、デバッグ環境を規定するITRONデバッギングインタフェース仕様があり、リアルタイムOS にデバッグ支援のための負担を与えないように、主としてホスト側に情報をもたせる構成が規定されている。また、オーバヘッドを低減し、動的なメモリ管理を最小化するように修正されたITRON TCP/IPAPI仕様、Java実行環境をITRON 仕様カーネル上に実装した場合のJavaプログラム-ITRON タスク間の通信インタフェースを定めるJTRON 仕様などがある。d. MS-DOSMS-DOS(Microsoft disk operating system)は、インテル社の8086系CPU で動作するシングルタスク16ビットOS であり、リアルタイムOS ではない。IBM PCのために1981年に開発され、1990年にDOS/V となりIBM PC AT 互換機でも日本語が利用できるようになった。NEC PC-9800シリーズ用MS-DOS とともに、わが国でも普及した。MS-DOS は、少量のメモリでも動作する小規模なOS であるが、現在のIA-32ファミリーの32ビットCPU でも使用可能である。シングルタスクで仮想記憶を用いないため、アプリケーションプログラムの実行においてOS のオーバヘッドはない。したがって、現在の高クロック周波数のCPU(Celeron,Pentium4など)では、ユーザプログラムを大変高速に動作させることができる。保護機構がないため、アプリケーションプログラムは、メモリやI/Oを高速にアクセスすることが可能である反面、誤った使用をすれば、システムは容易に暴走してしまう。しかし、構造が複雑でないプログラムで、割込みの使用を注意して行えば小規模な組込みに有効であり、全体的にコストを下げることもできる。ただし、タスク管理の機構がないため、ユーザが処理の優先度と実行時間を考慮して、プログラムの構造を決定しなければならない。すなわち、タスクのスケジューリングと各タスクを明確に分離することは難しくなる。現在は、PC DOS J 7.0/V の後継としてPC DOS 2000(IBM)が販売されている(MS-DOS は、1993年のVersion 6.22が最終版)。MS-DOS は制御機器やPOS 端末などで現在も使用されている。互換DOS として、DR-DOS7.03,FreeDOS,ROM -DOS 7.1などがあり、ネットワーク機能などの拡張が行われているものもある。e. VxWorks機器制御用のリアルタイムOS の代表例の一つであり、小さなカーネル(約20kB)に多様なコンポーネントを組合わせることで、固有の構成を容易に作ることができる。メモリ保護機能付きOS,DSP用OS,BSD OS、自動車制御用OS などの関連製品がある。1800を超えるAPIで、ネットワーク機能、I/Oファイルシステム、開発ツール群を利用することができる。クロス開発環境(Tornade)が充実しており、ブートローダだけをROM化しておけば、ネットワーク経由でOS とオブジェクトコードをロードしてデバッグ作業を行うことが可能となっている。例えば、WindViewと呼ばれるグラフィックツールでは、タスクの切換えの状態遷移がタイムチャートとして表示され、把握が難しい並行プロセスの挙動がわかりやすく提示される。f. Windows CE機器組込み用あるいは、PDA などの小規模情報機器用として1996年に公開され、現在はWindows CE5.0となっている。Windows CE3.0からはリアルタイム性が高められてはいるが、当初からリアルタイムOS として設計されたものではないはん用型のOS である。特徴はメモリ容量が小さくハードディスクを内蔵しない機器でも動作することであり、最小のフットプリント(メモリ使用量)となるOS コアだけならば200kB 程度である。はん用機としてのさまざまな機能は、コンポーネント化が進められている。PDA として利用されるPocket PCと呼ぶ標準的構成では、ROM が32MB(メガバイト)以上、RAM が16MB 以上必要となるだけであり、その一方で通常のWindows OS との整合性が高いことが利点である。現在はVoIPなどリアルタイム通信や、マルチメディア対応(ムービー再生)などが可能となっている。g. RTXRTX は、Windows NT,2000,XPにリアルタイム機能を追加するソフトウェアである。Windows環境の多くのツールやAPIをそのまま利用することができるため、Windowsプログラミングに習熟しているユーザにとっては非常に簡単に利用できる利点がある。リアルタイム特性をもたせたHALとリアルタイムサブシステム(RTSS)をWindowsに追加することにより、Win-7―37dowsに優先するリアルタイムスケジューラを動作させることが可能となっている。実行モジュールは、Windowsの環境(Win32環境)とRTSS 環境のそれぞれで動作するように作成することができ、RTSS 環境で動作するモジュールにリアルタイム性をもたせることができる。両者の環境において、割込みルーチンの設定や、外部I/Oの直接操作を行うRTAPIを使用できるため、デバイスドライバ開発の知識がなくともハードウェアの操作が可能である。
