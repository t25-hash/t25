ERP の機能アメリカのMES 協会が1993年9月に発行した関連用語集では,「ERPとは,MRPのロジスティクス的拡張である」と表現されている.ここでいうMRPはMRP-IIであり,ロジスティクスとは生産活動のはじめ(受注)から終わり(納品)までを意味し,この表現は当初の単発的な計画システムから,生産活動全般にわたる企業の資源を計画管理するシステムになってきていることを意味しており,情報の流れとものの流れを統合するという意味では,簡潔かつ直感的でよい.しかし,ERPの出自から見れば,このようなロジスティクスへの拡張を含めて2系統ある.すなわち,(1) MRPがロジスティクス情報システムの主要機能である需要予測,販売計画,在庫管理,購買管理といった生産全体の計画・管理システムと連携し,さらにそれらの情報をもとに企業会計等の基幹管理業務を行うように発展したもの(2) もともと会計,財務管理等の基幹管理業務処理中心のシステムからロジスティクス機能の統合へ進展したものである.どちらにしてもERPのカバーする範囲は広く,その業務処理を行うための豊富なサブモジュールがあり,それらは詳細な業務処理機能群からなっている.また,最近のERPシステムでは,能力計画の充実による現場等での能力を考慮した納期回答機能(ATP: available to promise)などへの拡張が見られる.APICS(American Production and Inventory Control Society,アメリカ生産・在庫管理学会)では,ERPを,MRP-IIを中核とした会計指向の生産情報システムととらえ,1)An accounting-oriented information system for identify-ing and planning the enterprisewide resources needed totake,make,ship,and account for customer orders. An ERPsystem differs from the typical MRPIIsystem in technicalrequirements such as graphical user interface, relationaldatabase,use of fourth-generation language,and computer-ized software engineering tools in development, client/server architecture,and open-system portability.2)More generally, a method for the effective planning andcontrol of all resources needed to take, make, ship, andaccount for customer orders in a manufacturing, distribu-tion,or service company.と定義している.ここでは,(1) GUI(graphical user interface)(2) 関係データベース(3) 4GL(fourth generation language,第4世代言語)(4) CASE(computer aided software engineering)をベースとしたシステム開発環境(5) C/S アーキテクチャ(6) オープンなシステムの可般性を有する企業全体に関わる生産活動資源の計画を挙げている.代表的なERPの機能として,表5・3にSAP社のR/3のモジュール機能例を示す.また,SCM への拡張等の最新機能についてはホームページ参照.c. ERP システムの導入ERPシステム開発の方法論は種々の書籍で論じられているので詳細は省略し,以下ではパッケージによるシステム構築法についてのみ述べる.パッケージによりシステムを構築するには,提供されているビジネスプロセス例から自社のビジネスプロセスに従ってパラメータ指定で自社に適したプロセスを選択し,モジュール機能と組合せシステムを作成する.例えば,生産方式は見込み生産,受注生産,半見込み・受注生産といった方式から選択する.パラメータ設定によるプロセス選択や必要データの指定はカスタマイズと呼ばれ,これで処理内容を設計していくことがERPシステムの特徴である.パラメータとは,例えば,自社の組織体系のコードや勘定科目の設定,各種マスタデータを構成するデータ設定(例えば,品目マスタへの価格,勘定科目,リベート方式等の登録),モジュールの処理機能の詳細選択(例えば,消費税の計算法や簡易課税方式の選択)等である.これらのビジネスプロセスは,ERPベンダから業界のベストプラクティスが提供されており,それに併せてBPR(businessprocess reengineering)を行い,変更した業務プロセスをもとにシステム構築をするのであれば,カスタマイズ工数は最小で済むというメリットがある.このように,ERPではパラメータ指定での業務処理システムの選択,ビジネスプロセスのワークフローに従った業務の流れの制御の定義により業務処理が行われるが,提供される機能群ではどうしても処理できない場合に新機能を追加(アドオン)するための機能がある.またC/S アーキテクチャによるオープンな環境でのシステム構築機能[API(application interface)表5・3 SAP R/3のモジュール構成略号モジュール名称機能サブモジュールの例SD Sales& Distribution販売管理受注管理,出荷管理,請求管理,販売支援,販売計画MM Materials Management在庫・購買管理在庫管理,倉庫管理,返品管理,購買管理,購買分析PP Production Planning生産管理・計画需要予測,生産計画,MRP,CRP,部品表管理,製番管理QM Quality Management品質管理検査計画,検査情報管理,分析情報管理PM Plant Maintenanceプラント保全予防保全,保全計画,保全文書管理,保全コスト管理HR Human Resource人事管理組織計画,人事計画,勤務管理,給与管理,旅費管理FI Financial Investment財務会計一般会計,連結会計,売掛・買掛管理,財務管理,与信管理,債務管理,為替管理,独自レポートのための特別元帳CO Controlling管理会計原価会計,間接費会計,収益性分析,ABC 分析,プロジェクト会計,予算管理AM Asset Management資産管理固定資産会計,技術資産管理,投資管理PS Project Systemプロジェクト管理プロジェクト計画・実績管理,予算管理WF Work Flowワークフローワークフロー定義,ワークフロー実行・管理,テンプレートISIndustry Solution業界別ソリューションハイテク・電子,自動車,消費財,通信,小売り等のモデル(http://www.sap.co.jp/solutions/industry/)BC BasisベーシスシステムOS,DBMS,GUI,アドオンのための第4世代言語BAPI7―111を公開して,その組合せで処理を進行する仕組み]などがある.アドオンは自社の強みを伸ばせる部分に集中して行うならば効果的であるが,自社の慣行で行われている業務プロセスを変更したくないために,アドオンによりパッケージ機能を変更するといった例は,それが自社競争力を伸ばす目的ではないため得策ではない.また,アドオンを行うとシステムのバージョンアップ時の移行変更工数は大きくなることも考慮して最小限にすべきである.稼働に至る以前にシステム運用のための教育は不可欠であるが,一般にユーザは実際に使用を始めるまでは実感がもてず,結局運用開始後に多くの問題が噴出するので,この運用開始直後の混乱への対応は事前に十分考慮しておくべきである.また,意外と時間がかかり,ミスが出やすいのが,移行時における旧データの移植である.初期入力のデータとともに,移行データも相当綿密な計画で何度もシミュレーション実験をするなどして完全を図らないと,支払条件や単位の新旧の違い,計算方式の違いで発注や納入ミスが出ることがある.d. EAI(enterprise application integration)技術企業情報システムの全体系を構築するには,さらに製造の計画,実行系であるSCM,MES,サプライヤ管理であるCSM(component andsupplier management)あるいはSRM(supplier relationshipmanagement)や顧客管理のCRM(customer relationship man-agement)といったシステムを導入する必要がある.これらのシステムとの結合法としては,以下の二つがある.(1) ERPと同一ベンダーの提供するシステムで構成する場合(2) 複数のベンダーの提供するシステムで構成する場合上記(1)はERPスイート(ERP suit)と呼ばれ,一元化されたデータベースを基に処理系としては整合化されたシステムが作成できる.しかし,ERPのパッケージシステムは種々のベンダーから提供されており,システムごとに特徴があり,機能や性能から見たとき別のシステムのほうが自社に向いた機能が豊富であったり,データ量に応じた高速処理が可能であったりして,必ずしもERPベンダー1社で閉じたシステムを作ることが最適であるとはいえない.そのため(2)のケースで構築することも多い.これを,自社に最も適したベンダーのシステムを選択して使用するという意味で,ベストオブブリード(best of breed)と呼ぶ.これでも零からのスクラッチ開発によるアドオンよりは効率がよい.このように複数のベンダーの提供するシステムを使用するときに,その相互結合の容易化のために使用される技術がEAI技術である.また,ERPシステムの構築に当たって,今までの資産(レガシーシステム)をすべて廃棄して新しいERPに置換えるという導入7―112情報・ソフトウェア法(これをビッグバン方式と呼ぶ)をとる場合もあるが,構築期間の短縮のため既存のいくつかのシステムやデータベースは残し,それらとERPとを結合するという導入方式をとることが多い.次いで,それをさらに高度化していくという段階的システム構築をするケースがある.このようなときにもEAI技術が使用される.これらは,既存のシステムやデータベースとのインタフェースを守り,システムの境界を明確にして新しいシステムを構築していく技術である.しかし,システムの作りやすさ(変更量の少なさなど)の観点から見るとき,種々のシステム境界が考えられる.EAI技術はそれらの境界から先を使用する側に意識させないようにすること(ラッピングすること)である.例えば,既存のシステム機能をCORBA(common object request broker architecture)のオブジェクト形態として利用可能とするためのシェルプログラム(ラッパー)をラッピング対象ごとに作成し,それを介してシステム間のインタフェースをとるような仕組みである.そのラッピングの分類を表5・4に示す.入出力のデータ形式そのものを標準化して,ERPデータを標準形式に変換して出力することで,別のアプリケーションが,個々のERPごとのデータ形式を意識しないで利用可能とするような試みも出てきている.これには,XML(extensible markup lan-guage)が使用される.XMLは,データにそのデータの意味を示すタグを付して記述する技術である.そのタグの標準化がいろいろな分野で進んでいる.製造分野の例としては,スケジューリング結果のデータを別のガントチャート表示モジュールで表示するといったケースでの,受渡しデータやスケジューラに対する入力データの標準タグ定義が行われている.これによれば,あるスケジュールデータはどんなガントチャート表示プログラムでも表示可能になるし,また,スケジュールデータはどのようなスケジューラにも入力できるようになる.この標準化は,PSLX(planning and scheduling language onXML)コンソーシアムで行われている.その他の製造分野でのXMLによる標準化例としては,電子部品の購買を目的としたロゼッタネット(RosettaNet)がよく知られている.また,サプライヤとメーカが協調して需要予測をしたり,生産計画を立案し,補充するような仕組みであるCPFRでもデータ記述の標準化を進めている.また,電子部品の形状,性能といった仕様のデータシートの標準化としては,ECIX(the
