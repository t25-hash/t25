# 1・2・1 設計の検証:シミュレーション

> **章**: 第1章 総論
> **節**: 1・2 設計の構成

設計ではまず思い描くことが最初であるが、その次の過程はそれを第三者に伝達できる形で表現することである。この表現法については2章で扱っている。G.モンジュ以来の二次元による表現法と、最近可能になった三次元による表現法である。このような形になると、設計の当事者でなくても設計を理解することができるようになる。この段階で重要なことは設計の客観的な検証である。図1・4からわかるように、設計の結果得られたモデルがはたして期待した性質をもっているかどうかを検証する手段は、大きく分けて解析計算と実験である。この解析と実験はいずれもシミュレーション(simulation)と呼ばれる。シミュレーションとはまねという意味であり、生物学では擬態のことである。何をまねするか、それは設計の結果としてのモデルである。このときに図1・4に示した設計を具現化したモデルのほかにシミュレーションモデルが必要になる。これは計算によるシミュレーションの場合には数値モデルであり、実験によるシミュレーションの場合には実験装置である。そこで図1・4を次の図1・7のように書き改める。図1・7では、設計の際に直接システムを作ってしまうことは現在では少なくなっているので、システムを省略する。シミュレーションには、シミュレーションモデルを作る段階と実行の段階とがあり、本当にシステムを模擬するためには、よいシミュレーションモデルの製作と正確な実行が必要である。機械の設計でも、最近は計算機の中の三次元モデルで設計を進めることも多くなってきているので、これが設計の結果としてのモデルである。しかし実物をかなり単純化したように見える計算機内の三次元モデルでさえそのままシミュレーションのモデルに使うには複雑すぎるのが一般的である。相当に単純化しないと能率的な設計作業の中でシミュレーションに活用できないのである。そこで、設計結果であるモデルをさらに単純なシミュレーションのための数値モデルに変換する必要がある。ただし、このモデルの変換には十分な注意が必要である。設計モデルのもっている性質を失ってしまうことも、設計モデルにない性格をもってしまうことも、誤ったシミュレーションにつながるからである。設計の検証(design verification)の場合に難しいのは、設計の結果であるモデルがまだ実体としては存在しないことである。この設計結果としてのモデルと、数値モデルまたは実験装置とを比較して、よいモデルになっているかどうかを判断しなくてはならない。たとえよいモデル化が行えても、まだいくつかの落とし穴がある。一つは実際のシステムのもつ非線形性と不確実性である。非線形性の原因には、機械的には摩擦、衝突、流体の存在、回転の影響などがあり、そのためにシステム全体がカオスの状態を示すことがありうる。その場合には製作誤差のような非常にわずかな変化のためにシステムの挙動に大きな違いが生じることがある。このようなシステムでは計算だけで性能を予測することは困難であるし、実験を行って、ある条件下で安定であるということがわかったとしても、製造や組立によるばらつきでのパラメータの変化範囲において十分に安定であるかどうかの判定には慎重でなければならない。非線形性によるカオス的な挙動の問題は、設計者の経験によってそのようなことの起こる可能性を極力排除するしかない。そのためには軸受の使い方や流体力の見積りなどに経験と勘を働かさなくてはならない。ただ幸いにもこのようなことの起こる可能性は一般的な機械設計の中では多くない。もう一つの問題は、単純化したモデリングによるシミュレーションに本質的な問題である。つまり、モデルを作る際に想定しなかった問題は実験によっても計算によっても出てこないということである。この二つの問題を解決しなければ、計算によって試作を減らし、開発期間を短縮することはできない。第二に挙げたモデリングの問題は大きな、そして深刻な問題である。かつては計算機の能力が限られていたために、解析のためのモデルには極端な簡単化が行われた。そのために当然起こるはずの現象までモデル化の段階で排除されてしまうことがしばしばであった。モデリングで不必要に簡単化しないことを心がけなくてはならない。設計の検証は必ずしもシミュレーションだけの仕事ではなく、何度か行われるデザインレビュー(design review)も含まれるのであるが、その検証で重要なことは設計者の思い違い、見落としなどによって生じた致命的な欠陥を内蔵していないかのチェックである。機械設計に関する実験にしても計算にしても、シミュレーションに際して設計段階で見落としをしていると、モデル化の段階でもそれを除いたモデルにしてしまうおそれがある。これではいかに厳密な実験を行っても、あるいは緻密な計算をしてみても欠陥は現れてこない。そのため、従来、設計の検証は実機実験によることが多かった。できるだけ実物に近い形で行えば、設計のときに見落としていたことも実験装置の中に入ってきて見落としに気がつくことが多い。このときに注意することは、できるだけ実機に近い複雑な実験装置で実験することである。実験の目的を絞って実験装置を単純化することは、起こるはずのことが起こらなくなってしまう可能性があるので危険である。最近は計算機の進歩によって、数値計算で実験に代えることが多くなってきている。その際にも以前の非力な計算機の時代から続く習慣なのか、なるべく単純なモデルをたてて計算することがスマートであるかのような錯覚をしがちであるが、これも単純な実験装置による実験と同様に危険なことである。このように実験装置または数値モデルを単純化してしまう過ち図1・7 設計検証のためのシミュレーションは、図1・2のような科学的アプローチでのシミュレーションとの混同が原因である場合がある。科学の場合はシステムが現実に存在している。したがってシステムに起こる現象もある程度わかっている。そこに存在する普遍的な原理を見出した場合には、システムを模擬しながら、その原理が明確に現れるような、できるだけ単純な実験装置または数値モデルを構築するのがよい方法である。しかし、設計の検証のためのシミュレーションをこのような原理確認のための実験と混同すると、注目した原理とは違った側面での問題点を初めから切り捨てることになってしまう。そこで将来の設計では実物をなるべく忠実な解析モデルにすることが望まれる。これは実物をそっくり試作してみることに相当する。このような製品そっくりの試作は家電製品や情報機器のような大量生産品ではつねに行われていることであるが、それでも何度も繰返すことは困難である。ましてや巨大なシステムである発電所やプラントでは全く不可能である。そこで計算機の上で作ってみるわけである。これは設計したことがすべて含まれているモデルであり、設計されたものの全体シミュレーション(total simulation)である。計算機上に実物のモデルを作ることは、実際に実物を作るより大変かもしれない。ただ、モデル化の過程はできるだけそっくりに作るほうが簡単だともいえる。以前は強度計算のためにはりに置換えたり、振動計算のために串団子モデルに置換えたりしていたのが、有限要素法や境界要素法で実物に近いモデルが使えるようになったことは、この全体シミュレーションへの方向と考えることもできる。今後ますます大きなモデルによるシミュレーションが行われるようになり、これによって多くの問題を避けることができるが、一方で巨大モデルを作るうえでの人為的な間違いが入ってくる可能性が増加することになる。
