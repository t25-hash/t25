UI フローチャートUIフローチャートを使って情報提示構造を明確にする.その際には,下記の項目に留意する.. タスクのリンク構造タスクのリンク構造には,最も基本的な線形構造,ツリー型,そしてハイパーリンク型など基本構成がある.. タスクの分岐おもなタスクの分岐には,下記の種類がある.(1) 次のタスクへ自動的に移行する.(2) 選択,条件により,タスクを分岐する.(3) システムの起点に戻る.(4) タスクの分岐点に戻る.(5) 一つ前のタスクに戻る.(6) 現タスクを一時中断して,別のタスクに分岐する.. タスクの制御タスクの内容が下記の条件のとき,次のタスクへの移行前に,タスクの完了,中止を判断するタスクまたはサブタスクを追加する.(1) 現タスクの完了を確認する.(2) 現タスクの操作を中止し,現タスクの初期状態へ戻る(例:入力ミスからの復帰).(3) 操作の結果がシステムの状態に重要な影響を及ぼすもの(例:設定値の変更).(4) 操作により後戻りのできない処理が行われる(例:発券,現金払出し).(5) 複数項目の設定操作を要求するもの(例:一括入力).e. UI デザインUIのデザインは,UIデザインフォーマット(図3・146)を使って行う.1画面で扱う情報を分解し,それに相当するUIパーツを組合わせてデザインするのである.まず,UIデザインフォーマットでタスク-サブタスクの関係を明確にした後,サブタスクの機能を実現するためのグラフィカルユーザインタフェース(以下,GUI)パーツやそれに関係するUI設計項目(29項目)を決める.例えば,ON 入力をユーザにさせる機能の場合,わかりやすいボタンのGUIパーツを用い,設計項目として一貫性や適切な用語・メッセージなどが抽出される.さらに,これらのGUIパーツ間の関係性を明確にし,ユーザは自然にそれらのパーツ間の関係が理解できるよう画面内の情報の流れを検討する.以上の要素のほか,画面内で最初の手がかりとなるタイトルと次の画面へと遷移させる確認の手がかり(場合によっては,手がかりをなくして自動化させることも,コンセプトを参照しながら決める)も検討する.最後に,1画面内の情報の流れは,通常上から下へ,左から右へと考えるのが自然である.f. UI 評価評価方法は下記の5条件をクリアする必要がある.(1) 評価者のレベルに対応できること.(2) ユーザの視点で評価できること.(3) ポイントを押さえた評価方式にし,評価者に負担を極力かけないようにする.(4) UIコンセプト,設計に連動した評価方法であること.(5) 画面内情報の流れから評価ができること.以上のコンセプトに対し三とおりの評価方法が考案されている.一つは,UIシステム全般を詳細に評価するタイプ(UI設計項目の29項目を評価項目としたもの),二つめはシステム全般を簡易に評価するタイプ,そして三つめは画面のみを評価する簡易タイプである.三つめの画面のみ評価するタイプは,UIシステム全般を評価するタイプと併用してもよい.これらの方法では,使用者は自分のスキル,対象システムの複雑さなどを考慮してどのタイプを使えばよいのかを判断する.評価はユーザの視点で予測しつつ行う.評価レベルは+1,0,-1点の3段階である.+1点はよく考えられているレベル,0点は一応問題はないレベル,-1点は問題があり解決を要するレベルである.このとき,その情報の重要度も考慮しながら評価する.特に,UIコンセプトに基づくUI設計項目は特に重要視して評価し,これらの項目はすべて+1点になっていることが望ましい.
