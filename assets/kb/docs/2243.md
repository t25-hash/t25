# 4・2・2 製品開発組織のタイプと効率

> **章**: 第4章 設計の管理
> **節**: 4・2 設計開発プロジェクト計画と管理

a. 部門横断チーム(cross functional team)とそのタイプ開発プロジェクトにおいてリードタイムの長期化や設計完成度低下の最大原因に、縦割り機能的組織による情報伝達の遅延がある。この欠点を補完するために、多様な部門のメンバから構成される部門横断チームが有効である。部門横断チームは、チームリーダのもとにマーケティング、製品開発、生産、購買、品質管理などの関係者を一つのチームに集め、これらのメンバの非公式な情報交流を促進する環境を実現し、開発初期段階で製品ライフサイクル全体を考慮するのに有効な組織形態である。例えば、製品に関する情報を全員で共有化し、開発初期段階でチームメンバ間の異なるニーズをトレードオフできるため、全体的な視点から最適解を見つけ出すことが可能となる。また、チームに製品開発の責任をもたせることにより、意識高揚や一体感の醸成が図られるとともに、情報交流が加速化され、製品開発と生産との対立解消にも通じる。製品開発のような非定型的な作業では、多大な工数と期間が、技術・専門知識やノウハウの情報収集に費やされている。すなわち、開発作業そのものより開発環境の整備や事前準備に多くの時間が割かれている。これは、情報の共有化が効率的にできていない環境にあることによる。情報共有化の最善の方法は、組織的・物理的な近接配置環境に関係者全員を集めることであり、相互に目で見る管理ができる環境を作ることである。近接配置環境を創造する手段として最も有効なのが、部門横断チームの採用である。部門横断チームを運用することによって、開発者個人の能力、考え方、ノウハウといった属人的属性や、開発作業の進捗状況や必要な技術レベルといった作業的属性を全員が明確に把握できるようになる。したがって、必要とする情報の収集効率が飛躍的に向上することから、事前準備作業の必要性が少なくなり、付加価値作業に専念できることになる。すなわち、部門横断チームは縦割り機能的組織の壁を打破する組織的・物理的近接によって、開発目標の共有化、情報伝達の迅速化と開発作業の効率化を実現する組織形態といえる。しかしながら、部門横断チームを有効活用できるか否かは、チームメンバ間で情報が円滑に伝達される仕組みができているか否かにかかっており、チーム組織化の方法によってその成否に大きな差が出る。製品開発プロジェクトの組織形態に関しては、大きく分けて二つの形態があるといわれている。一つは、専門知識や技術のインプット側に対応した組織形態で、機能的組織形態と呼ばれるものである。一般に基礎研究分野の開発組織では、同じ研究分野の開発者を一つの組織下に配属することによって、専門知識の伝達や共有を容易にしている。これは、専門知識の生成と伝達が、一般に専門分野別に行われていることからも明らかである。機能的組織形態では、1―157製品開発が各専門領域ごとに独立の組織のもとで行われていく。専門領域ごとの開発遂行の権限は機能的組織の管理者にあり、開発リソースの配置や他の組織との調整およびコンフリクトの解決を担当する。通常、一人の管理者が開発プロジェクト全体を統括することは少ない。この組織形態は、製品開発プロセスが独立の工程に分割でき、個々の工程の最適化が全体の最適化につながることが前提条件である。したがって、開発者間の相互理解が十分でなく、明確な調整ルールや手続きが確立していない場合には、他の組織との調整が困難になる傾向がある。もう一つは、アウトプット側に対応した組織形態で、製品を軸として組織化する部門横断的組織形態である。これは、特定製品を開発するのに必要な機能と技術・専門知識をもったリソースを一つの組織下に配置することで、迅速で効率的な開発を実現することを目指した組織形態である。部門横断的組織形態では、一貫した開発作業が一つの組織下で行われ、多くの専門機能が集約されている。この組織は、独立企業と同等の機能をもった組織であり、最終的な開発責任は当該組織の管理者にある。部門横断的組織形態は、開発目標の共有化や製品としての統合化を容易にするが、長期的には他の組織の同じ分野の開発者との技術情報や専門知識の交換の機会が失われ、最先端の専門性を維持するにはマイナスとなる傾向がある。その他の開発組織は、上記二つの両極的な組織形態の組合せで、部門横断チームである。その代表的なチーム形態は、図4・11に示すような機能的組織形態(functional organization)をベースとして、部門横断プロジェクトへの専任度の低いものから高いものの順に、ライトウエイトチーム(light weight team)、ヘビーウエイトチーム(heavy weight team)、タイガーチーム(tiger team)である。部門横断チームの場合、通常の組織は機能的組織として存在するが、臨時的・短期的に部門横断的なチームが組織化され、開発プロジェクト全体に責任をもつチームリーダが、開発リソースの配置や他の組織との調整や折衝を遂行する。チームリーダは、必ずしも組織的な地位が高い人であるとは限らないため、後方から支援する地位の高い経営者による調整が必要なこともある。タイガーチームは、機能的組織から一時的に完全に隔離されて開発作業を遂行するチーム形態で(トラの縦じまが背中部分で分断されていることに由来したネーミング)、最もチームメンバの専任度が高く、時として人事考課さえもチームリーダが担当するチーム形態である。逆に、最も専任度が低いのがライトウエイトチームで、これは大半の開発作業を機能的組織で行い、必要とされるつどチームでの作業を行うチーム形態である。ヘビーウエイトチームは、これら二つの部門横断チームの中間に位置するもので、かなり重度の専任度をもつチーム形態である。b. DSM:部門横断チーム組織化の方法論開発作業間の相互関係に基づいて部門横断チームを組織化する方法論としては、DSM(design structure matrix)がある。これは、双方向の情報交換の必要性が高い場合、情報交換が必要とされる部門や個人を同じチームに配属させることによって、非公式な情報を含めた情報共有化の促進を実現する方法論である。小規模な部門横断チーム(5~10人程度)の場合には、密着して仕事を行うことができるため、さまざまな問題をチームメンバの相互理解と協調によって解決することができる。しかし、大規模なチームの場合には、メンバ全員の相互理解と協調を実現するのは困難であり、開発作業全体を小規模なサブチームが遂行できる単位に分解する必要がある。これにより、小規模チームのもつメンバ間の密接な関係というメリットを最大限生かすことが可能となる。しかし、開発作業は相互に密接な関連をもっているため、特定の設計案を採用した場合、設計パラメータに多大な影響を与えてしまう。したがって、独立な単位に開発作業とサブチームを分割するには、複雑な方法論が必要となる。DSM は、大規模チームにおいて小規模チームで得られるような密接な情報共有化環境を実現するために、相互関連性の高い開発作業を一つのサブチームに集約するようにグループ分けする方法論である。DSM による部門横断チーム組織化の手順は、開発作業のフローを定義することを出発点とする。開発作業のフローを定義するためには、技術や情報に関する作業間の依存関係を明示する必要がある。二つの開発作業間の関係には、三つのタイプがある。まず、作業1が終了してから初めて作業2が開始できるような従属関係にある典型的なシーケンシャル開発である。次に、作業1と2との間で開発者間の相互交流が不要な完全独立関係にあり、同時並行的に作業を行うことができるタイプである。そして三つめは、開発の進行過程で作業1は作業2の情報を必要とし、同時に作業2も作業1の情報を必要とする相互依存関係をもつものである。DSM は、図4・13に示すような開発部門やアクティビティ(開発作業と呼ぶ)間の相互関係の強さを表すマトリックスを用意することから始まる。ここで図4・13に示すDSM の概念図をもとに、部門横断チームの組織化の手順を示してみたい。図4・13の行と列のA からG
