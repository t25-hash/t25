# 4・2・3 デザインレビュー(design review)

> **章**: 第4章 設計の管理
> **節**: 4・2 設計開発プロジェクト計画と管理

デザインレビュー(以下、DR とする)は、もともとアメリカの軍事産業を中心とした複雑なシステム製品の信頼性問題を設計段階でチェックするために生まれた。そこでのDR は、文字どおりの設計審査であり、軍等の発注者がメインコントラクタの基本設計を審査し、さらにメインコントラクタは部品設計を分担するサブコントラクタの設計を審査するというように、いわゆる垂直的統合としてのDR であった。これがわが国に導入されると、源流管理の品質保証という考え方を実現するための道具として部門横断的な活動において活用されるようになった。すなわち、製品開発の節目で、設計の漏れやマイナス要因を前出しするための関連部門を交えた提案、着眼点の出し合いや、部門間の調整を行う水平的統合を目的としたDR である。表4・5に産業機械メーカでのDR の実施例を示す。企画、設計、試作、設備設計、生産準備、生産、そして製品出荷後まで、次のステップに移行する前にその可否を含めて、DR 1からDR 9までの9回のDR が開催されている。各DR では、主催する議長のもとに関連部署の参加を得て進められる。例えば表4・5の例では、DR 2およびDR 4を除いて事業部長が議長を務めている。構成メンバは、DR 1では設計技術、販売、購買、品質保証等の当該製品の担当者、そして担当者以外の設計技術部、販売部のメンバも参画し、市場調査に基づく企画目標の適正性と、明示的に示されていないさまざまな企画仕様上の問題点や漏れについて、将来の使用者のさまざまな条件や環境を想定したうえで審査や助言がなされる。各DR では、参加者のもつ固有技術やこれまでの経験を踏まえた知識から、特にあってはならない設計上の問題の摘出が行われる。加えてそのための議論を円滑にするために、QFD(品質機能展開:quality function deployment)や信頼性解析に関連したドキュメントを用意しておく必要がある。特に信頼性の問題では、信頼性解析の内容やその結果が用いられる。製品の信頼性をチェック、確保するために、試作段階で信頼性試験が行われる。しかしながら、信頼性試験だけでは実用状態のあらゆる故障モードを摘出するのは困難であり、かつ問題点の前出しという意味では遅すぎるともいえる。設計時点で信頼性を紙上やコンピュータ上でバーチャル(仮想的)に予測、評価する方法が不可欠であり、そのための方法論が信頼性解析である。信頼性解析には、拡張性解析や接近性解析のような暗示型の解析と、FMEA(fail-ure mode and effect analysis)やFTA(fault tree analysis)のような論理に基づく直接摘出型の解析がある。拡張性解析は、これまでの設計では経験やノウハウのない拡張部図4・13開発部門・アクティビティの相互関係に基づくグループ化名称審議事項移行可否を決める次工程1―160設計工学分を明確に顕在化させて潜在的問題点を予知し、対応を検討するものである。用途、材料、部品、設計法、加工法等の新規の拡張部分について、生じた理由、検討方法、内容について整理し、一覧表としてメンバに提出される。また接近性解析は、限られた空間に部品が組込まれるために生じる干渉を予知するための解析である。特にダイナミックな使用状況で起こる接近や接触と、それによる故障や不具合が推論される。ある「使用条件」で、「対象部品」に、「接近部品」が接近し、「介在要因(介在物、ストレス等の物理的条件)」によって、ある「状態」になり、「故障モード」が発生するというように、「」内のキーワードを適用することによって対象部品の受ける影響を推定し、事前に対策をとろうというものである。このような開発プロセスの節目、節目で、部門横断的メンバによるDR を効果的にマネジメントすることが、開発プロジェクトにおいて重要なことである。それによって潜在的な問題点をなるべく源流で摘出できれば、後工程に流出したことに伴う莫大な修正コストを削減でき、開発コストの増加を防ぎ、また設計変更等の手戻りに伴う開発の遅れを未然に防ぐことによって開発リードタイム短縮にもつながる。
