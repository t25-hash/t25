クリティカルパスによる日程管理プロジェクト全体の先行関係をアローダイアグラムで記述できたら、次にプロジェクト全体、あるいはプロジェクトの個々のステップ(アクティビティ、工程)の日程を決定していく。この日程に基づいて、プロジェクトの日程管理・進捗管理をしていくが、ここでは特に管理の重点としてクリティカルパス(critical path)という概念が重要である。クリティカルパスとは、そのプロジェクト全体の完了日程を決定するアクティビティのパスであり、その上にあるアクティビティが遅れるとプロジェクト全体の日程に影響を与えるものである。逆に、クリティカルパス上に載っていない工程は、アクティビティの完了時間に対して余裕をもっており、完了が遅れてもこの余裕時間の範囲内であれば、プロジェクト全体に対する完了時間を遅らすことはない。このように、クリティカルパスはプロジェクトの日程管理上非常に重要であり、これは通常、以下に述べるような、プロジェクトを構成する各工程に対する最早開始時刻と最遅完了時刻の計算によって簡単に計算することができる。このようなクリティカルパスを求めるための一連の計算手続きを、狭い意味でPERT 計算と呼ぶことがある。(1) 最早開始時刻(earliest starting time)の算出PERT ではネットワーク上でのそれぞれのアクティビティの状態はノード(結合点)で表現されているので、時刻(日程)0にプロジェクトを開始したときに、それぞれの工程のアクティビティに着手可能となる時刻である最早開始時刻は、最早結合点時刻と呼ばれることもある。すなわち、それぞれのノードを、最も早ければいつリリース(出発。そのノードに入ってくるアクティビティすべてが終了していなければリリースできない)できるかを意味する時刻である。図4・15のそれぞれのノードに付随している四角い欄の上側に示した数字が、最早結合点時刻である。これは以下のようにして簡単に求めることができる。まず、プロジェクト開始のノード(図4・15の例ではノード1)から始め(ここを時刻0にリリースする)、次のノードに到達する時刻を順次前方向に求めていく。すなわち、いま対象となっているノードがリリース可能となったらただちにそれに続くアクティビティを行い、それに連なるノードに到達する時刻を求める。この例では、時刻0にノード1から最初のアクティビティA がリリースされ、終了までに3週間かかるから、ノード2の最早開始時刻は3(週間)となる。次に、時刻3でノード2がリリース可能となるから、ノード3の最早開始時刻は、このノードに入ってくるアクティビティB の所要時間4を足して7になる。ノード2に連なる他のノード5についても、同様にアクティビティEの所要時間6を足して、最早開始時刻は9になる。しかし、ノード4に関しては、ノード2に直接つながっているアクティビティD のほかに、ノード3を経由してアクティビティCもこのノードにつながっている。ここで、アクティビティD はノード2の最早開始時刻3に所要時間6を足して、時刻9には完了する。一方、アクティビティCはノード3の最早開始時刻が7であるので、これに所要時間3を足して、時刻10まで完了しない。ノード4はこれら二つのアクティビティが両方とも完了して初めてリリースが可能となるので、このノードの最早開始時刻は10となる。このように、複数のアクティビティが入ってくる(合流している)ノードに対しては、それぞれのルートでの作業完了時刻を求め、そのうちの最も遅いものがそのノードの最早開始時刻になる。このように、最早開始時刻はアローダイアグラムにおけるアクティビティの先行関係を見ながら、プロジェクト開始からそれぞれのノードをリリースできる最も早い時刻を順次求めていくことであ1―161る。このような最早開始時刻の求め方を前進計算ということもある。(2) 最遅完了時刻(latest finishing time)の算出最遅完了時刻とは、プロジェクトの最後のノード(すなわち、プロジェクト全体が完成した状態)に、上述の前進計算で求められたプロジェクト全体の総所要時間で到達するために、途中の各ノードには遅くてもこの時刻までには到達していなければならない、つまりそのノードの直前のすべてのアクティビティがそれまでに完了していなければならない、という時刻である。この時刻は最遅結合点時刻と呼ばれることもある。図4・15のように、最遅結合点時刻は通常ノードに付随している四角い欄の下側に示してある。この計算は最早開始時刻とは逆方向に、プロジェクトの終了のノード(図4・15の例ではノード12)を起点に先行関係をさかのぼって始点まで順次計算していくので、後退計算と呼ばれている。図12に到達するためには、その前のノード9はアクティビティLの所要時間の2だけ前の時刻、つまり時刻19には遅くてもリリース可能となっていなくてはならない。この時刻がこのノードの最遅完了時刻である。このように、直接つながっている後続のノードの最遅完了時刻から対象作業の所要時間を引いたものが、そのノードの最遅完了時刻である。図4・15の例では次に、ノード9の前の一つであるノード11はダミー作業であるので、このノードの最遅完了時刻はノード9と同じ値の19となる。このノードの前のノード10も同様の計算により、16となる。ノード9のもう一つの直前のノード7は、ここから二つの矢印が伸びている。このように、複数のアクティビティが出ている(分岐している)ノードは、最早開始時刻の場合に複数のアクティビティが入っているノードと同様に、注意が必要である。すなわち、複数のアクティビティが出ているノードに対しては、それぞれのルートでの作業開始時刻を「前のノードの最遅完了時刻―アクティビティの所要時間」により求め、そのうちの最も早いものがそのノードの最遅完了時刻となる。ノード7の場合には、一つはノード10にダミー作業でつながっているものであり、これはノード10の最遅完了時刻と同じ16になる。もう一つのノード9に続くルートに関しては、ノード9の最遅完了時刻の19からアクティビティJ クリティカルパスの発見と余裕時間図4・15のように求まった各ノードの最早開始時刻と最遅完了時刻の差が、そのノードのリリースに対する時間的な余裕である。この値がゼロのノード、つまり最早開始時刻と最遅完了時刻が等しいノードを結んでいったアクティビティのパス上では、アクティビティの着手に全く余裕がない。このパスが前に述べたクリティカルパスである。図4・15では矢印を太くした、アクティビティA→B→C→G→J→Lがこのプロジェクトのクリティカルパスを形成している。前にも述べたように、クリティカルパス上にないアクティビティについては、ある程度の遅延はプロジェクト全体の総所要時間に影響を与えない。これは例えば図4・15のアクティビティD やEのように、その余裕時間の範囲内ではアクティビティが遅れても、他のアクティビティに影響を及ぼさないものがある。これを余裕時間の中でも特に、自由余裕時間ということがある。一方、アクティビティH などは、たとえ余裕時間の中でも遅延が生じれば、後続のアクティビティに影響を与える。例えば、後続作業の開始時刻が遅れたり、余裕時間が短くなったりするものがあり、これを干渉余裕時間という。1―162設計工学以上のように、クリティカルパス上にあるアクティビティかどうかによって、管理の重点をうまく調整することができる。また、アクティビティが余裕時間を超えて遅延を生じれば、クリティカルパスは変わってしまうことも、十分頭に入れておく必要がある。. CPM(critical path method):最小コストスケジューリング多くのアクティビティに要する時間とコストには関係がある。またコストには、アクティビティにかかる人件費等の直接費と、個々のアクティビティには直接関係しないプロジェクト全体に関わる間接費とがある。個々のアクティビティを通常見積もられる期間よりも短くしようと思えば、残業等の直接費の増加が見込まれる。一方、間接費のほうは一定か、プロジェクトの期間が長くなれば、その資源を拘束する期間が長くなることから、期間に対して増加関数となる。このような状況において、コストを最小にするようなスケジューリングがCPM-コスト(CPM-cost)である。図4・18に示すような四つのアクティビティからなる簡単な例を用いて、以下に説明する。この場合のプロジェクトの間接費は8日で一定8万円であり、1日延びるごとに5万円だけ増加するとする。またアクティビティの所要日数については、表4・7に通常の日数に加えて、最大限可能な特急日数(クラッシュ日数)、そしてそれぞれに対応するコストが示されている。ここでは、特急コスト(crash cost、クラッシュ図4・17 山崩しによる負荷の平準化図4・16山積みによるプロジェクト負荷の試算図4・18 CPM-コストの例題の初期クリティカルパス表4・7 CPM-コストの適用例題アクティビティ通常日数(日)通常コスト(万円)最大特急日数(日)通常日数A→B→Dコスト)と通常コストの差から、その中間の日数でのコストは、両者の直線補完で計算できるものとする。例えば、アクティビティBにおいて、(18-9)/(5-2)=3万円が1日短縮することによるコストの増額分であり、5日を3日にする場合は、9+(5-3)×3=15万円となる。このような前提で、コスト最小となるスケジュール案を求める手順を以下に示す。(1) 通常日数におけるクリティカルパスを求める。この場合、A→B→D であり、所要日数は10日である。そのときのトータルコストは、個々のアクティビティの通常日数のコストの合計26万円に間接費8+5×2=18万円を足した44万円である。(2) クリティカルパス上の最小コストとなるアクティビティに着目し、逐次日数を短縮していく。表4・7の右側にはコスト計算の補助表として、各アクティビティについて、1日当たり短縮コストの増額と、最大の短縮可能日数が示してある。表4・7から、A→B→D のうち1日短縮した場合にコスト最小となるのはD であることがわかる。(3) (2)におけるトータルコストと、そのときのクリティカルパスを求め直す。アクティビティB の2万円をプラスし、間接費5万円がマイナスされ、41万円と計算される。またこの場合のクリティカルパスはA→B→D で変わらない。以上のような(2)(3)の手順を繰返す。表4・8はその結果を示したものである。D を2日、B を1日短縮したところでCもクリティカルパスになり、最終的にD を2日短縮した8日がコスト最小スケジュールとなっていることがわかる。以上のような計算手順において、アクティビティの数やコストカーブが曲線で期間ごと補間が必要になる場合には、手計算が難しくなる。そのような場合は、この問題のモデルが線形計画法としてモデル化できることから、そのソフトを活用して解を求める必要がある。また現在では、PERT 計算を含めて多くのCPM のための市販ソフトが存在し、それらを活用することができる。
