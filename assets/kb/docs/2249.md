# 4・2・7 プロジェクトの管理・改革の方法論

> **章**: 第4章 設計の管理
> **節**: 4・21 にはPDPCの簡単な例を示している。これにはプロジェクト

開発プロジェクトの組織化、計画・立案から、実際にプロジェクトが進行していく中で、そこでの問題点や失敗を次の開発に盛り込み、プロセスや手順を改革・標準化することによって生かすことも、そのマネジメントの立場から重要な視点である。そのような立場からの方法論として以下のようなものがある。a. デザインラショナーレ多くの参加者を必要とする近年の設計開発業務において、度重なる条件変更等による繰返しプロセスを効率よく管理していく方法論として、デザインラショナーレ(design rationale)の重要性が注目され始めている。デザインラショナーレのデザインは設計であり、ラショナーレはその設計が生み1―165図4・22設計変更のパターン化を行ったデザインラショナーレフォーマット出された理由や原理を意味する。すなわちデザインラショナーレとは、設計の背後にある理由もしくは理由付けのプロセスの履歴に関する情報、と定義される。特許や研究論文は、成功体験の記録であり、特に次の開発に有効なノウハウの伝承や、避けるべき反省点等の試行錯誤のプロセスは見えてこない。デザインラショナーレとは、このような記録を組織の知識として標準化された形式で残そうというものである。設計の背景にある理由と合理性の記述に着眼したデザインラショナーレは、簡易的には、日報や議事録の形で文書化されている場合が多いと考えられる。しかしながら、(1)現在の設計開発状況はどうなっているか、前回は何を話し合ったのか、今回は何を話し合う1―166設計工学図4・23デザインラショナーレ図の例のか、未解決の問題は何か、現在それはどうなっているのか、(2)なぜそれが設計開発の問題となるのか、なぜそれが重要な問題として認識されているのか、(3)この設計問題に関連する問題は何か、前提が変化すると問題はどう変化するか、それについて考えなかったらどうなるのか、(4)代替案は何か、その是非はどうか、その理由は何か、(5)他ではこの問題はどのように扱われているか、それを過去の例から学ぶことはできないのか、というような問いに系統的に答えることができるためには、設計問題に適した文書化の様式と、それらの流れが全体として見えるようにするための方法が必要になってくる。一般に設計開発のプロセスは、(1)目的・要件に基づく代替案を創造する議論空間、(2)考え出されたアイデアである代替案とそれらの優劣を判断し議論を行う代替案空間、(3)実際の製品・仕様として目に見える形で残る部分である設計解空間、からなり、これら三つの空間を行き来することによって、最終的な設計解に到達するといわれる。したがって、設計会議ごとの内容を文書化する様式として、このような構造にのっとったものであることが望ましい。図4・22は、このような考え方に基づくフォーマットであり、またアメリカ国家規格(American National Standards Institute)で意思決定を文書化する様式にも対応している。加えてこの例では、議論の内容の部分の、機能定義、仕様、修正、確認、運用、メモ、テスト、スケジュール等のカテゴリー分けがあらかじめフォーマット化されている。さらに特にプロジェクトを遅らせる原因となる修正については、図4・22の右側に示すような四つのタイプのどれに相当するかをチェックすればよいようになっている。このようなカテゴリー化をしておくことで、後での整理や次期開発へ問題をフィードバックするための活用が容易となる。このような設計開発プロセスの断片を、その紆余曲折の全体を見通せるように、意味的に結びつけたものがデザインラショナーレ図(design rationale diagram)法である。図4・23は、実際の印刷設備の開発に適用した例であり、そのデザインラショナーレ図と呼ばれる。図4・23中の番号は個々の設計会議の文書に対応し、時間的に早いものから順番に番号がつけられている。時間的順序を表す番号からわかるように、本来3-13-14-15-12と進むべきところが、3から一度12に飛んで13に戻り、14,15と進み、再び12に達している様子がわかる。このような逆戻り的なプロセスは、設計・開発には往々にして見られることである。日常的に設計会議の内容をフォーマット化し、それに番号を与え、タイトルを記したメモカードを用意したうえで、事後的に図よって、このような問題点が摘出できる。デザインラショナーレ図法は、ある行為、方策に対する最終的な結果とその結果に至る過程を示したものであり、形式的には前述のPDPCに似ている。PDPCは基本的に計画図であるが、PDPCをデザインラショナーレ図のように、ノウハウ蓄積、紆余曲折の記録として用いることもできる。(a) IDEF 0の基本単位(A 0画面)さらにこれらの方法論を、開発プロセスの経過を追跡し、プロジェクトメンバ間でリアルタイムに共有化する図的表現の手段として用いる方法もある。それによって最終的な目指す方向に向けて、メンバ間の意思疎通を促進させることができる。さらに簡易的には、イントラネット上のホームページで開発過程や設計変更情報を共有化する取組みも多く見られる。b. IDEF従来、設計・開発プロセスは、つねに激しい生産性向上の環境下におかれてきた製造に比べ、標準化、効率化の対象にあまりされてこなかった。どちからといえば、創造的仕事の名のもとにそのような対象から聖域化され、業務の内容自体がブラックボックスとされてきた部分が多かった。ある程度繰返しがあるプロジェクトについては、設計・開発プロセスの業務の流れを標準化・体系化し、構成要素を明確化する必要がある。これがなければ設計・開発プロセスにおけるむり・むだ・むらが摘出できない。IDEF(ICAM definition、最近ではintegrated definitionの略とされる)は、もともとはアメリカ空軍のICAM(integrated com-puter aided manufacturing)プロジェクトで定義された情報技術を活用するための業務やプロセスの機能分析からシステム構築までの一連の方法の規定であった。その中でIDEF0は、業務システムのアクティビティとその関連をトップダウン的に体系化し、階層的に詳細化する図式表現である。最近では、業務やプロジェクトを行う場合に、何をすべきかという仕事の内容をツリー構造に体系化することを目的とするWBS(work breakdown structure)のためのツールとして、多くの市販ソフトが活用できる。IDEF0の図式表現の基本は、図4・24に示すようにボックスで表現される仕事を構成するアクティビティと、左からの矢印で示される入力(I)、右への矢印の出力(O)、上からの矢印で示される制約または制御(C)、そして下からの矢印で示される使用する資源またはメカニズム(M)によって構成される。アクティビティはビジネスプロセスを構成する機能要素であり、動詞の形で表現される。そしてその実行の要件と結果の産物がICOM で表現される。Iはアクティビティによって変化を受けるものや情報である。Oはその産物であり、他のアクティビティのIかCとなる。Cはアクティビティに影響を与える条件や制約・目標と定義されるが、明らかにIでないものはCと考えてよい。またM は、アクティビティを実行する場合の資源や手段である。1―167(b) A 0画面をブレークダウンしたA 1画面図4・24製品開発プロセスのIDEF 0による分析[I:入力、C:制約、M:メカニズム(資源)、O:出力]図4・24(a)に示すA 0は、トップのアクティビティであることを示し、これから逐次階層的にアクティビティは詳細化される。図4・24(b)はA 0の次の階層であり、A 1からA 5までのアクティビティに分解されている。さらに必要に応じて各アクティビティは、例えばA 1を詳細化したものであれば、A 11,A 12,A 13というような番号がつけられ、枝分かれ的にブレークダウンされる。IDEFによるモデル作成には、例えば1箇所を変更すると関連部分が連動して変更されるような修正が容易な市販ソフトの使用を前提としている。IDEF0を用いた業務分析では、As-Isモデル(As-Is model)、すなわち現状の業務プロセスを記述することから始められ、その分析を通してあるべき姿であるTo-Beモデル(To-Be model)の記述や構築がなされる。しかしながら多くの部門や異なる組織間にわたる業務プロセスでは、As-Isモデルを記述すること自体に意味がある。関係した部門や組織のメンバが集まり、IDEF画面をプロジェクタに投影し、意見交換をしながら記述することによって、それまで互いにブラックボックスとなっていた仕事の内容や情報の流れ、さらには問題点や冗長な単位業務を発見することができる。図4・24のIDEF図は、パーソナルコンピュータの開発プロセスの事例である。まずA 0は「製品を開発する」であり、それを1階層ブレークダウンすると、A 1「製品企画・条件を決定する」からA 5「工程設計・生産準備する」までの五つのアクティビティからなっていることがわかる。これらの各アクティビティは、さらに下位の階層に分解される。例えば、A 3「設計・試作をする」では、下位層画面ではA 31ゲートアレイ、A 32マザーボード、A 33本体ボデー、A 34電源ユニット設計等のアクティビティが並行的に描かれる。図4・24から、例えば次のような問題点が指摘される。工程設計・生産準備の後に設計変更要求として、設計・試作に情報がフィードバックされ、それが開発のリードタイムを遅らせ、余分な工数を強いている。同様にエンジニアリング評価にもフィードバックがあることから、工程設計・生産準備を設計・試作、評価のアクティビティと、よりオーバラップさせる必要がある。そのためには、製造容易性のデータベースの整備とそれをベースにしたDR の強化、そしてこれらのアクティビティ間を流れるデータ項目をブレークダウンして、逐次的にやるべきものと並行的にやるべきことを整理・1―168設計工学体系化することが前提となる。さらにこの例の場合、上流のアクティビティでは事業部、下流では生産子会社とそれぞれ主体が異なる組織が関わってくるために、それぞれの役割分担と連携の仕方も見直しの材料となる。IDEF0のソフトの多くは、アクティビティが明確にされた次のステップとしてのABCと連動しているものが多い。またIDEFのシリーズには、例えばIDEF3はプロセス記述モデル、IDEF6はデザインラショナーレというように目的に応じた図的表現がある。
