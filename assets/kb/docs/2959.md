# 4・2・7 プロジェクトの管理・改革の方法論

> **章**: 第4章 設計の管理
> **節**: 4・2 設計開発プロジェクト計画と管理

開発プロジェクトの組織化，計画・立案から，実際にプロジェク
トが進行していく中で，そこでの問題点や失敗を次の開発に盛り込
み，プロセスや手順を改革・標準化することによって生かすこと
も，そのマネジメントの立場から重要な視点である．そのような立
場からの方法論として以下のようなものがある．
a.デザインラショナーレ 多くの参加者を必要とする近年
の設計開発業務において，度重なる条件変更等による繰返しプロセ
スを効率よく管理していく方法論として，デザインラショナーレ
（design rationale ）の重要性が注目され始めている．デザインラシ
ョナーレのデザインは設計であり，ラショナーレはその設計が生み
出された理由や原理を意味する．すなわちデザインラショナーレと
は，設計の背後にある理由もしくは理由付けのプロセスの履歴に関
する情報，と定義される．特許や研究論文は，成功体験の記録であ
り，特に次の開発に有効なノウハウの伝承や，避けるべき反省点等
の試行錯誤のプロセスは見えてこない．デザインラショナーレと
は，このような記録を組織の知識として標準化された形式で残そう
というものである．
設計の背景にある理由と合理性の記述に着眼したデザインラショ
ナーレは，簡易的には，日報や議事録の形で文書化されている場合
が多いと考えられる．しかしながら，(１)現在の設計開発状況はど
うなっているか，前回は何を話し合ったのか，今回は何を話し合う
図4･ 23 デザインラショナーレ図の例
のか，未解決の問題は何か，現在それはどうなっているのか，(２)
なぜそれが設計開発の問題となるのか，なぜそれが重要な問題とし
て認識されているのか，(３)この設計問題に関連する問題は何か，
前提が変化すると問題はどう変化するか，それについて考えなかっ
たらどうなるのか，(４)代替案は何か，その是非はどうか，その理
由は何か，(５)他ではこの問題はどのように扱われているか，それ
を過去の例から学ぶことはできないのか，というような問いに系統
的に答えることができるためには，設計問題に適した文書化の様式
と，それらの流れが全体として見えるようにするための方法が必要
になってくる．
一般に設計開発のプロセスは，(１)目的・要件に基づく代替案を
創造する議論空間，(２)考え出されたアイデアである代替案とそれ
らの優劣を判断し議論を行う代替案空間，(３)実際の製品・仕様と
して目に見える形で残る部分である設計解空間，からなり，これら
三つの空間を行き来することによって，最終的な設計解に到達する
といわれる．したがって，設計会議ごとの内容を文書化する様式と
して，このような構造にのっとったものであることが望ましい．
図4･22 は，このような考え方に基づくフォーマットであり，
またアメリカ国家規格（American National Standards Institute ）
で意思決定を文書化する様式にも対応している．加えてこの例で
は，議論の内容の部分の，機能定義，仕様，修正，確認，運用，メ
モ，テスト，スケジュール等のカテゴリー分けがあらかじめフォー
マット化されている．さらに特にプロジェクトを遅らせる原因とな
る修正については，図4･22の右側に示すような四つのタイプのど
れに相当するかをチェックすればよいようになっている．このよう
なカテゴリー化をしておくことで，後での整理や次期開発へ問題を
フィードバックするための活用が容易となる．
このような設計開発プロセスの断片を，その紆余曲折の全体を見
通せるように，意味的に結びつけたものがデザインラショナーレ図
（design rationale diagram）法である．図4･23は，実際の印刷設
備の開発に適用した例であり，そのデザインラショナーレ図と呼ば
れる．図4･23中の番号は個々の設計会議の文書に対応し，時間的
に早いものから順番に番号がつけられている．時間的順序を表す番
号からわかるように，本来3-13-14-15-12と進むべきところが，3
から一度12に飛んで13に戻り，14，15と進み，再び12に達して
いる様子がわかる．このような逆戻り的なプロセスは，設計・開発
には往々にして見られることである．
日常的に設計会議の内容をフォーマット化し，それに番号を与
え，タイトルを記したメモカードを用意したうえで，事後的に図
4･23のようにそのカードを意味的に結びつけ，並べ換えることに
よって，このような問題点が摘出できる．
デザインラショナーレ図法は，ある行為，方策に対する最終的な
結果とその結果に至る過程を示したものであり，形式的には前述の
PDPCに似ている．PDPCは基本的に計画図であるが，PDPCを
デザインラショナーレ図のように，ノウハウ蓄積，紆余曲折の記録
として用いることもできる．

図4･24 製品開発プロセスのIDEF0による分析［I ：入力，C：制約，M：メカニズム（資源），O：出力］

さらにこれらの方法論を，開発プロセスの経過を追跡し，プロジ
ェクトメンバ間でリアルタイムに共有化する図的表現の手段として
用いる方法もある．それによって最終的な目指す方向に向けて，メ
ンバ間の意思疎通を促進させることができる．さらに簡易的には，
イントラネット上のホームページで開発過程や設計変更情報を共有
化する取組みも多く見られる．
b.IDEF 従来，設計・開発プロセスは，つねに激しい生
産性向上の環境下におかれてきた製造に比べ，標準化，効率化の対
象にあまりされてこなかった．どちからといえば，創造的仕事の名
のもとにそのような対象から聖域化され，業務の内容自体がブラッ
クボックスとされてきた部分が多かった．ある程度繰返しがあるプ
ロジェクトについては，設計・開発プロセスの業務の流れを標準
化・体系化し，構成要素を明確化する必要がある．これがなければ
設計・開発プロセスにおけるむり・むだ・むらが摘出できない．
IDEF（ICAM definition，最近ではintegrated definitionの略と
される）は，もともとはアメリカ空軍のICAM（integrated com-
puter aided manufacturing）プロジェクトで定義された情報技術
を活用するための業務やプロセスの機能分析からシステム構築まで
の一連の方法の規定であった．その中でIDEF0は，業務システム
のアクティビティとその関連をトップダウン的に体系化し，階層的
に詳細化する図式表現である．最近では，業務やプロジェクトを行
う場合に，何をすべきかという仕事の内容をツリー構造に体系化す
ることを目的とするWBS（work breakdown structure ）のため
のツールとして，多くの市販ソフトが活用できる．
IDEF0の図式表現の基本は，図4･24に示すようにボックスで表
現される仕事を構成するアクティビティと，左からの矢印で示され
る入力（I )，右への矢印の出力（O)，上からの矢印で示される制
約または制御（Ｃ)，そして下からの矢印で示される使用する資源
またはメカニズム（M）によって構成される．アクティビティはビ
ジネスプロセスを構成する機能要素であり，動詞の形で表現され
る．そしてその実行の要件と結果の産物がICOMで表現される．I
はアクティビティによって変化を受けるものや情報である．Oはそ
の産物であり，他のアクティビティのIかCとなる．Cはアクティ
ビティに影響を与える条件や制約・目標と定義されるが，明らかに
IでないものはCと考えてよい．またMは，アクティビティを実
行する場合の資源や手段である．
図4･24(ａ)に示すA0は，トップのアクティビティであること
を示し，これから逐次階層的にアクティビティは詳細化される．図
4･24(ｂ)はA0の次の階層であり，A1からA5までのアクティビ
ティに分解されている．さらに必要に応じて各アクティビティは，
例えばA1を詳細化したものであれば，A11，A12，A13という
ような番号がつけられ，枝分かれ的にブレークダウンされる．
IDEFによるモデル作成には，例えば1箇所を変更すると関連部分
が連動して変更されるような修正が容易な市販ソフトの使用を前提
としている．
IDEF0を用いた業務分析では，As -Isモデル（As - Is model ），
すなわち現状の業務プロセスを記述することから始められ，その分
析を通してあるべき姿であるTo- Beモデル（To- Be model ）の記
述や構築がなされる．しかしながら多くの部門や異なる組織間にわ
たる業務プロセスでは，As - Isモデルを記述すること自体に意味が
ある．関係した部門や組織のメンバが集まり，IDEF画面をプロジ
ェクタに投影し，意見交換をしながら記述することによって，それ
まで互いにブラックボックスとなっていた仕事の内容や情報の流
れ，さらには問題点や冗長な単位業務を発見することができる．
図4･24のIDEF図は，パーソナルコンピュータの開発プロセス
の事例である．まずA0は「製品を開発する」であり，それを1
階層ブレークダウンすると，A1「製品企画・条件を決定する」か
らA5「工程設計・生産準備する」までの五つのアクティビティか
らなっていることがわかる．これらの各アクティビティは，さらに
下位の階層に分解される．例えば，A3「設計・試作をする」で
は，下位層画面ではA31ゲートアレイ，A32マザーボード，A33
本体ボデー，A34電源ユニット設計等のアクティビティが並行的
に描かれる．
図4･24から，例えば次のような問題点が指摘される．工程設
計・生産準備の後に設計変更要求として，設計・試作に情報がフィ
ードバックされ，それが開発のリードタイムを遅らせ，余分な工数
を強いている．同様にエンジニアリング評価にもフィードバックが
あることから，工程設計・生産準備を設計・試作，評価のアクティ
ビティと，よりオーバラップさせる必要がある．そのためには，製
造容易性のデータベースの整備とそれをベースにしたDRの強化，
そしてこれらのアクティビティ間を流れるデータ項目をブレークダ
ウンして，逐次的にやるべきものと並行的にやるべきことを整理・
体系化することが前提となる．さらにこの例の場合，上流のアクテ
ィビティでは事業部，下流では生産子会社とそれぞれ主体が異なる
組織が関わってくるために，それぞれの役割分担と連携の仕方も見
直しの材料となる．
IDEF0のソフトの多くは，アクティビティが明確にされた次の
ステップとしてのABCと連動しているものが多い．またIDEFの
シリーズには，例えばIDEF3はプロセス記述モデル，IDEF6は
デザインラショナーレというように目的に応じた図的表現がある．
