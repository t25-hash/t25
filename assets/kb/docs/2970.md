# 4・4・5 製品系列の統合化設計のための数理的手法

> **章**: 第4章 設計の管理
> **節**: 4・4 製品系列の統合化と設計

a.設計プロセスから見た統合化の意味 製品系列の統合化に
はさまざまな内容が含まれることから，設計を進めるための手順す
なわち設計プロセス（design process ) を考えておくことも重要
である．問題解決過程としての設計プロセスを考えると，統合化設
計における処理内容は，単一の製品を対象としたシステム構造・形
態・属性のそれぞれに関する一連の処理内容を基底としつつ，図
4･49の製品表現における階層性を踏まえれば，以下のように定義
できる ．
(１) 設計要求の分析処理：設計目標を確立し，多様な設計要求と
それらの間の相関性を分析する．
(２) システム構造のシンセシス処理：システムの構造とその変形
を確立する．具体的には，基盤となるシステム構造を確立
し，適切なモジュールのサイズや相互関係についてのアーキ
テクチャ を選定して，製品群を展開するうえでの基本概
念を確立する．
(３) 形態のシンセシス処理：製品形態とその変形を定める．具体
的には，基盤となる形態を確立しつつ，そのもとでの幾何学
的な属性などの制御によって製品群を展開するための可能性
を模索する．
(４) 製品群の具体化処理：具体的な製品群を確立する．その処理
内容は，属性値を具体化することによって各モジュールの内
容を確定するための属性値の具体化と，可能なモジュールの
組合せの中から適切なものを定めることによって個々の製品
を具体化するためのモジュール組合せの選定とから構成され
ている．
これらのうち，第2段階のシステム構造と第3段階の形態に関する
内容は製品群の多様性を拡大するための可能性を探求するものとな
っている一方，第4段階の製品群の具体化に関わる内容は製品系列
を確定するために具体的な製品を確認して定めるものとなってい
る．
以上の分類は，そのまま実際の設計の進め方に対応するものでは
ないが，それにはどのような内容が含まれているかを考え方として
整理するうえでは有益である．また，何らかの支援手法やツールを
導入する際には前提とすべきものである．
b.顧客ニーズの把握 製品系列の統合化は顧客ニーズの多様
化や流動化とも連動している．このため，図4･51における関係性
をも含めつつ，顧客ニーズの幅や変化方向などを見定める必要があ
る．すなわち，与えられた顧客ニーズのもとで設計を行うのみでは
なく，設計が新しい顧客ニーズを呼び起こすという考え方にも立つ
ことが求められる．一般に，製品のもつ価値は使用（性）価値
（use value ）と感性価値（kansei value ）とに分類できるとされて
おり，製品が社会へと進展していく導入期・成長期・成熟期を通じ
て，顧客の関心事は当初は使用（性）価値におかれるが，次第に使
い勝手や意匠性などの感性価値にも注意が払われるようになって，
最終的にはそれらが差別化の手段となるとされている ．このよ
うな顧客ニーズの変化には何らかの様式が潜んでいたり ，また
要素技術の動向についても，別途，何らかの予測は可能であること
から，統合化においては，設計に先立つ企画においてその種の様式
を踏まえた戦略的な考察を行っておくことが，個別の設計を行う場
合にも増して求められることになる．
c.アーキテクチャの設計 アーキテクチャの設計はいわゆる
概念設計に関わる内容であり，機能構造そのものを処理することに
もなっている．このこともあって，それについての明確な方法を手
順として示すことは難しい．4･4･1･b項のスウォッチ社の事例は，
腕時計のアーキテクチャをインタフェースの意味で再発見すること
によって，ある機能構造に新しい意味をもたらしている場合であ
る．コンピュータ産業における状況は，業界標準としてモジュラア
ーキテクチャが定義されている典型例であるとともに，その技術革
新が図4･46に示すような階段状に進展している事例でもある．い
ずれにしても，アーキテクチャの設計には，ある設計要求を実現す
るうえでの機能的な構造が確定しているとしたときに，それを構成
する細かい部分をどのようなグループにまとめあげて個別モジュー
ルとするかを決定する課題が含まれていて ，それが重要な位置
を占めている．その際，望ましいアーキテクチャとはどのようなも
のであるかに関しては，コストや利益を具体的に論じるよりも，そ
れらの前提となるある種の構造に関してどのようなシナリオを描け
るか，その中でいずれの戦略を選択するかが重要となる ．
例えば，ライフサイクルエンジニアリング（life cycle engineer-
ing，3･6節参照）の視点を借りれば，製品が環境へ与える影響を
低減するに当たって描けるシナリオや選択できる戦略には多種多様
なものが存在している．具体的には，製品を製造した後に使用を進
めて故障したらすべてを単純に廃棄する場合，廃棄製品を一定のレ
ベルまで分解したうえでマテリアルリサイクルかサーマルリサイク
ルを行う場合，製品を保全（メンテナンス）しながらできるだけ長
く使用した後でリサイクルを行う場合，保全に加えてモジュールご
とに別製品での再利用（リユース）を繰返したうえでリサイクルを
行う場合などが考えられる．図4･53は，ある冷蔵庫に関して，一
定の想定条件のもとに三つのライフサイクルシナリオを想定したう

f .モジュール組合せによる設計 何らかのアーキテクチャと
形態を想定することができ，加えて各モジュールとして利用可能な
ものが事前に確定している場合，製品系列の統合化設計問題は製品
を構成するために実際に用いるモジュール群を候補群の中から製品
系列の競争力を最大化するように取捨選択するという問題になる．
個別の製品を機能面から見た場合，それぞれについての要求を充足
する最小機能を有するモジュールがそれぞれに存在しているはずで
ある．しかし，一連の製品を統合的に設計する場合，それらのすべ
てを導入したのでは必要なモジュールの種類が多すぎることになっ
ていて，個別的には過剰な機能をもつモジュールを用いても全体と
しては必要なモジュールの種類をあるところまで削減したほうがよ
い場合が想定できる．
このようなモジュール数の削減に向けては，モジュールの代替に
よる共通化がどこまで可能かどうかが鍵であり，それについての条
件は，図4･56にも示すように，以下の3種類の制約条件として整
理できる ．
(１) 代替可能性制約：モジュールを代替した場合に，その製品が
各要求機能を満足できるかどうかに関する制約．
(２) 代替同時性制約：機能面の連結関係により，あるモジュール
を代替した場合に，他の特定のモジュールも同時に代替しな
ければならないという制約．
(３) 容量制約：エネルギーなどについての需給関係が全モジュー
ル間で適切であるための制約．
