# 6・3・2 製品開発のための企業内統合情報システム

> **章**: 第6章 情報システム
> **節**: 6・3 企業内統合情報システム

a. 製品開発のための企業内統合システムの変遷製品開発のための企業内情報システムは、まず生産側で部品の所要量を展開し、生産工程の準備をするために必要な部品表のシステム化と図面をCAD で作成することから始まり、両者を統合化し、さらに関連する製品情報を統合化してきた。CAD は、最初は二次元CAD から始まり、次第に三次元CAD に移っていった。三次元CAD も、最初は単なる面の集合体として表現されていたが、ソリッドモデル技術が向上し応用範囲が拡大してきた。その後、CAD システムおよび部品表システムは統合化が進み、まず、CAD システムから部品構成を抽出して部品表を作るような連携が始まり、次に、部品表の構成から該当部品に関係するCAD 図面が検索できるようになって、図面管理機能として双方向の統合化が行われてきた。しかし、設計現場では、CAD 技術の発展に伴い、機構系および電気系の複数のCAD が導入され始め、かつ設計過程で発生する技術文書は複数の文書作成システムで個別に作成され始めた。その結果、設計データが多様なデータ形式でさまざまな場所に個別に保管されるようになってきた。さらに、市場主導の製品設計が求められてきたため、新製品の開発サイクルは短縮化され、かつコスト削減、機能強化等の設計変更の数も増えてきた。そのような中で製品開発情報を一元化して管理する必要性が高まり、PDM(product data man-agement)として製品データを統合管理するシステムに進化していった。当初、PDM は設計が完了し、量産出図直前から量産出図後の設計結果としての製品情報のマスタ管理機能であった。その後、設計段階の上流部分を含み、構想設計段階から支援できるような統合機能に発展していった。PDM の呼び名もproduct developmentmanagementといわれ始め、守備範囲を広げた。さらに、最近では、PLM(product lifecycle management)として、製品の構想段階から生産終了までをカバーしようという広範囲な統合情報システムが使用され始め、ERP(enterprise resource planning)とも連携されてきている。この一連の流れを図6・18に示す。b. 製品開発のための企業内統合システムの構成要素および全体図製品開発のための企業内統合システムは、製品情報の管理に加えて、市場情報、技術動向情報および収益管理と統合される。統合化された製品開発の流れを図6・19に示す。図の左側は、製品開発の最上流部分で、市場動向・技術動向等からどのような製品を開発するべきか模索している段階である。ここで製品がねらうべき市場、価格帯、製品コンセプトの概要が決まる。次に具体的な製品開発プロジェクトが発足し、意匠設計で製品の外観設計がなされ、それに基づいて大枠の外装および内装設計がなされ、設計する部位を定め、設計分担が決まる。次に詳細設計に移り、部品番号管理および設計変更管理が始まっていく。これらの製品開発の流れを統合的に支援する情報システムとしては、図6・20のようになり、市場・製品分析で多くの情報を集め、販売する地域、企業規模、顧客特性、製品特性等々を企業戦略に沿って多面的に分析し、製品開発方針を決める。それらの情報は、計画システムに出力されて需要予測等に活用され、さらに製品開発システムに出力されて製品開発の管理に活用されていく。c. material)と呼ばれ、業種によって独自の表現形式がある。部品表システムは、設計段階では、多段階の製品の構成を管理し、かつ設計変更を管理する必要がある。製造に渡された部品表は、設計部品表のどの部分を社内で製造し、どの部分を社外から調達するかなどを反映して製造部品表になる。製造部品表では、最下層の部品は調達部品として認識され、中間段階の部品は社内で製造する部品として識別される。例を図6・21に示す。部品表システムの重要な機能として設計変更管理機能がある。設計変更は、製品構成の変更履歴を管理すると同時に、設計側から製造側に変更を正確に伝達する手段である。これにより、同じ製品でも、どの設計変更レベルで製造されたかを管理でき、保守部品等の特定が容易にできる。設計変更の管理の区分はさまざまであるが、図6・22に示すように六段階で管理される場合がある。designは、設計者の設計段階において、開発側のみで参照および変更が可能で、生産側からは見えない。pre-releaseは、開発側から事前検討段階として生産側に出図される。これは、正式出図前に設計側が生産側からの検討内容(生産容易性検討、コストダウン要望反映等)を受取り、正式出図に反映するものである。releaseは正式出図で、これにより該当製品に関連する部品表、図面、技術文書が生産側に出図される。acceptは、出図された内容を生産側が検討完了し、生産準備に入ったことを示し、effectiveで生産段階に入ったことを示す。新たな設計変更が開発側から流れてきて、effectiveの段階になると、前の設計変更はclose段階になり、履歴として残される。e. 製品情報統合管理システム(PDM およびPLM)家電業界でも自動車業界でも組立産業では、開発期間短縮と低コスト化を図る手段の一つとして、調達・設計の分野において、製品に合わせて部品調達する従来のシステム(製品適合)から、部品に合わせて製品を設計する方向(部品適合)にシフトしつつある。部品は絶対原価を決めてグループ内企業にこだわらず安いところから調達し、各製品に共通した部品を使うことで量産効果を上げる努力をしている。生産部門ではラインのモジュール化、部品工場の分社化などが行われ、開発環境の分散や世界同時の生産立上げなど、グローバルな環境での作業が求められている。こうした変化の中で、それまで閉鎖的だった設計データ(CADデータ)管理に関する問題点が明確になってきた。例えば、担当者がいないと最新データの所在がわからない、他部署の人間が必要なデータにアクセスできないといった問題である。一部の担当者だけがすべてを把握し、ほかから見ればブラックボックスのような状態では、グローバルな環境での効率的な開発はできない。そこで、CAD データを管理するシステムを導入し、設計~生産部門間でデータの効率的な共有・再利用を図ろうという動きがあり、これはPDM(製品情報管理)と呼ばれる。加えて、PDM の管理下に複数のディジタル設計データ(三次元CAD)をおき、そこから立体モデル(DMU、ディジタルモックアップ)を起こすことが進められている。図面だけではイメージの違いによるトラブルなども起きたが、リアリティのあるDMU を使うことで、これを回避できるようになった。また、データの更新履歴や製品に関するデータがナレッジ(知識)として蓄えられるようになり、さらに進んだ技術を取り入れ、製造工程を三次元モデルでシミュレートすること(ディジタルマニュファクチャリング)もできるようになった。ディジタルマニュファクチャリングによって入手できるプロセスの時系列に沿ったデータは、設計者にとって大きな価値があり、十分なシミュレーションを行うことで確実性の高い設計ができ、試作段階での修正も減る。PDM に、このような時系列のデータ管理を含め、全体にインテグレーションしたものが、PLM である(図6・24)。PLMでは企画から設計、生産、そして営業、保守部門まで、製品のライフサイクルに携わるすべての部署でデータを共有し、単一化されたシステムで協力し合うことで生産の効率化を図ろうとしている。7―122生産システム工学
