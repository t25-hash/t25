# 6・4・2 企業間統合情報システムの基本要件

> **章**: 第6章 情報システム
> **節**: 6・4 企業間統合情報システム

企業間統合情報システムを実現させる場合、企業間統合情報システムの目的である企業間での情報共有と企業間の業務プロセスの連携と、企業間でのシステム構築の三つの基本要件について述べる。a. 情報共有企業間統合情報システムによる共有対象データは、統合データベース(integrated data base)へ格納される。統合データベースの構築は、運営主体(誰が)、データベース配置(どこに)、データの操作単位(何を)、企業内システムとの関係(どのように)、機能拡張性(なぜ)などの側面に着目して要件を明確にする必要がある。運営主体の検討では、技術情報の所有やその維持・管理を、誰が主体となって運営するかに着目する。データベース配置の検討では、企業間統合情報システムおよび統合データベースを、物理的にどこに設置して利用するかという、機器の設置やネットワークの構成に着目する。データの操作単位の検討では、企業間統合情報システムで提供される情報の操作可能なレベルに着目する。企業内システムとの関係の検討では、技術情報を提供する企業の企業内データベースと統合データベースとの分離・非分離に着目する。機能拡張性の検討では、企業間統合情報システムで実現する機能の将来的な機能拡張性に着目する。上記の各側面について、さまざまな要件を明確にすることにより、企業間統合情報システムによる情報共有の多くの形態が可能となる。. データベース(data base)配置要件データベース配置の形態としては、(1) 情報を利用する特定の企業の利便性を重視して、利用者側に配置する形態(2) 複数の利用者を対象として、情報を提供する企業側に配置して共有する形態(3) 複数の提供者からの情報を利用者側に配置して集約する形態(4) 情報の対象である製品の供給と調達という関係において、供給者と調達者がそれぞれ保有する情報を相互に提供するために、それぞれ企業間統合情報システムを構築する形態(双方向な形態)(5) アウトソーシングを考慮したVAN 業者等、第三者に配置する形態などを明確にする。. 運営主体要件運営の形態としては、(1) 技術情報の利用を重視して、利用者が運営する形態(2) 技術情報の提供とその維持・管理を重視して、提供者が運営する形態(3) 技術情報の対象である製品の供給と調達という関係で、供給者と調達者がそれぞれ企業間統合情報システムを構築し、それぞれ自分の担当分を運営する形態(4) 第三者に運営を委託する形態などを明確にする。以上のi,ii項より四つのパターン(個別対応、情報提供、情報集約、連携・統合化)を抽出できる。図6・35に要件抽出パターン図を示す。このパターン図では、横軸をデータベース配置の観点として、どこに企業間統合情報システムの設置を行うのかを示し、縦軸を運営主体の観点として、誰が企業間統合情報システムの運営を行うのかを示す。横軸のいちばん左には、特定の統合データベース利用者の利便性を重視した企業間統合情報システムを位置付け、提供側での利用にも留意した方向を経て、情報提供者を中心とした、複数の利用者への情報提供を実現する企業間統合情報システムをいちばん右に位置付ける。縦軸のいちばん下には、統合データベースの提供者が企業間統合情報システムの運営を行う形態を位置付け、利用者も運営に参加する方向を経て、双方の企業が企業間統合情報システムを構築し、協調して運営する形態をいちばん上に位置付ける。個別対応のパターンでは、特定の利用者が、特定の提供者から個別に技術情報の提供を受ける情報共有を想定している。特徴は、利用者側の利便性重視であり、船舶、航空機、プラント等大規模な製品・システムを調達する形態に向く。情報提供のパターンでは、一つの提供者が複数の利用者に情報を提供する情報共有を想定している。特徴は、部品を調達し、大規模な製品・システムを複数の企業に供給する形態に向く。情報集約のパターンでは、一つの利用者が複数の提供者から情報の提供を受け、集約して管理する情報共有を想定している。特徴は、大規模な製品・システムを、複数の企業からの部品調達により構築し保守・運用する形態に向く。連携・統合化のパターンでは、複数の企業が、それぞれ相互に情報を提供し、お互いに情報を利用し合う情報共有を想定している。特徴は、自社で企業間統合情報システムの構築・運営が難しい中小7―127企業等を対象にした第三者による構築・運営が主体となる。. 企業内システムとの関係要件技術情報を作成して提供する企業において、提供対象の文書・図面などを格納する企業内データベースと、統合データベースとの関係をどのように構成すべきかということは重要なテーマとなる。この形態は、(1) 企業内データベースと統合データベースを、完全に分離する形態(2) 文書・図面などのデータとそれを管理するデータを分けて、それぞれデータベースを作成することにより、提供する文書・図面などのデータのみ、または、文書・図面などを管理するデータのみの一方を公開する半分離の形態(3) 企業内データベースの一部を、統合データベースとして公開する非分離の形態などを明確にする。. 機能拡張性要件企業間統合情報システムの各機能については、その拡張性の方向によって、統合データベースの構築の方法や採用する技術が大きく異なる可能性がある。機能拡張性の方向としては、(1) 共有する文書・図面などを、ファイルそのままの単位で扱いながら、その履歴の管理やアクセス制御、コミュニケーション機能等を強化する方向(2) 構成情報等を付加することで、構成管理等を実現する方向(3) 文書・図面などのデータフォーマットとして、構造化されたデータモデルを採用することで、文書・図面を構成する要素単位での利用を実現する方向などを明確にする。. データ操作単位要件企業間統合情報システムにおいて共有するデータに、どのようなデータフォーマット・データモデルを採用するかは、その操作の可能性に大きく影響する。データ操作単位としては、(1) 文書・図面単位の電子ファイルを対象とする形態(2) 文書・図面単位での履歴・コメントを含めて対象とする形態(3) 複数文書・図面間の構成情報を合わせて対象とする形態(4) 文書・図面を構成する要素とその構造の情報を対象とする形態(5) 企業内の設備管理、文書管理システム等との統合を実現する形態などを明確にする。また、独自フォーマットと標準フォーマットといった問題もあるが、企業間統合情報システムにおいては基本的に標準フォーマットとすることが望ましい。以上のiii~v項より八つのパターン(第一段階、高度化、一元化、関連化、初動段階、要素管理、構成管理、統合化)を抽出できる。図6・36,6・37に二つの要件抽出パターン図を示す。図6・36のパターンでは、横軸をデータ操作単位の観点として、どのような単位でデータを利用、および維持・管理するかを示し、縦軸をデータベース構成の観点として、企業内データベースと統合データベースをどのように構成するかを示す。図6・35 業務プロセス(business process)企業間統合情報システムによる情報共有により企業間連携が実現できる。企業間連携では、単に業務間をつなげたり、適用する業務の範囲を広げるにとどまらず、相互に関連した形に業務プロセスを統合していく必要がある。したがって、企業間統合情報システムを利用した企業間連携の場合、どの業務プロセスを対象にするかという対象業務プロセスと、どの程度連携するかというプロセスの連携度合いを考えるべきである。これにより企業間統合情報システムによる業務プロセスの多くの形態が可能となる。. 対象業務プロセス企業間統合情報システムの対象とする業務の範囲は、製品のライフサイクル全体では、その企画から廃棄に至るまでに、おおむね以下のような業務がある。・企画・開発業務・生産・構築業務・運用・支援業務・撤去・廃棄業務企業間統合情報システムの対象範囲を、どの業務プロセスにするか、またはどのように特定の業務プロセスから徐々に適用範囲を拡大していくかにより、企業間統合情報システムの実現形態が異なる可能性がある。. プロセスの連携度合いプロセスの連携度合いは、おおむね、連係(つながりの意)から連携(ともに手を携えてものごとを行うことの意)への段階が考えられる。企業間統合情報システムとして見た場合、連係の段階では、企業間の接点を明確にし、そこでのオンラインによる技術情報の受渡しを効率良く行う実現形態が必要である。連携、統合と連携程度が深まるにつれて、企業間で協調して業務を遂行するために、業務プロセスの標準化やBPR(business process reengineering)が必要になる。企業間統合情報システムとしては、利用者が企業間統合情報システムと企業内システムを別システムとして意識せず、最終的には一体となったシステムとして実現する形態が必要になる。以上のi,ii項より四つのパターン(局所的連係、局所的連携、全体的連係、統合化)を抽出できる。図6・38に要件抽出パターン図を示す。このパターン図では、横軸をプロセス連携度合いの側面として企業内と企業間のプロセスがどの程度連携するかを示し、縦軸を対象業務プロセスの側面としてライフサイクル全体におけるさまざまな業務プロセスへの業務適用範囲を示す。いちばん左には、企業間のプロセスの接点を明確にしたうえで、企業間統合情報システムにより、そこでのオンラインによる技術情報の受渡しを効率良く行う連係の状態を位置付ける。いちばん右には、企業間統合情報システムにおけるステータスと企業内の業務プロセスが、境目なく自動的につながっている連携の状態を位置付ける。いちばん下には、特定の業務に限定して適用されている状態を位置付ける。いちばん上には、徐々に提供範囲を拡張していくことを経て、ライフサイクル全体にわたる全業務プロセスが対象となった状態を位置付ける。局所的連係のパターンは、特定の業務において企業間のプロセスの接点のみの連係を想定している。例えば、企業間統合情報システムにより提供される情報の参照は、関係者全員が行えるが、レビューの依頼や承認といったステータスを変更する操作については特定の人が行う。そのため、設計作業からレビュー依頼、レビュー中から承認といった流れの中に、企業内、企業間の壁が存在し、人を介してその壁を越える必要がある。局所的連携のパターンは、特定の業務のみを対象とするが、企業間と企業内のプロセスが境目なく自動的につながっている連携を実現することを想定している。例えば、利用者側、提供者側ともに、ワークフローシステムが企業間、企業内のそれぞれのプロセスの流れを中継するため、利用者、提供者には一連の流れに見せることができる。しかし、対象とする業務プロセスが限られているため、企業間と企業内のプロセスには、ギャップが存在する可能性があり、実際には個別に対応する必要がある。全体的連係のパターンは、ライフサイクル全体におけるさまざまな業務に、企業間のプロセスの接点のみの連係が拡大された状態を想定している。例えば、企業間統合情報システムを適用する業務の範囲が拡大され、さまざまな人が企業間プロセスに直接関わってくることが考えられる。そして、企業間統合情報システムで実現される企業間のプロセスの接点のみの連係を企業内において一連の流れにするため、企業内プロセスの見直しが必要とされる。統合化のパターンは、ライフサイクル全体におけるさまざまな業務に対して、企業間と企業内のプロセスが境目なく自動的につながっている連携が実現され、業務プロセスの統合が実現された状態を想定している。例えば、企業間統合情報システムにおける企業間プロセス自体が企業間の接点のみではなく、境目のない自動的な一連の流れとして実現され、企業間、企業内や部門間の壁がなくなって、ライフサイクル全体におけるさまざまな業務に対して業務プロセスが統合されていく。c. システム構築企業間統合情報システムの構築においては、複数の企業・業種との接続を考えるうえでの相互接続性の側面、システム構築の容易さ、アプリケーション依存性、機能レベル等に関わる情報機器の利用の要件を明確にしておく必要がある。. 相互接続性(interoperability)企業間統合情報システムの構築において、複数企業を前提に考えた場合、その相互接続性は重要な問題である。ここで考えるべき相互接続性は、ネットワークプロトコルレベルの接続にとどまらず、企業間統合情報システムが相互に接続し、その機能を実現できることが必要である。接続対象となる企業の増加に伴って、以下のような段階が考えられる。・専用線+独自インタフェース+独自フォーマット・インタネット+独自インタフェース+独自フォーマット+データ変換・インタネット+標準化された企業間統合情報システム間のインタ7―129フェース+標準フォーマット当然、これらの中間的な段階も存在する。. 情報機器の利用企業間統合情報システムの構築において、どの程度情報機器を利用するのか、どの程度情報機器の機能に依存するのかは、構築のための期間・コストや、各種の制約・要求、求められる機能のレベル等により異なる。高機能な情報機器を利用して、企業間統合情報システムを構築する場合は、利用する機能や使い方、標準化への対応動向等を十分考慮する必要がある。
