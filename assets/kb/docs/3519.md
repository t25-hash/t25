# 6・4・3 企業間統合情報システム例

> **章**: 第6章 情報システム
> **節**: 6・4 企業間統合情報システム

企業間における設計承認業務適用例を図6・39に示す.a. 設計承認業務(approval of design)の流れ図6・40に設計承認業務の具体的な流れを示す.(1) 提供者は,利用者からの指示により設計業務を行い,設計が完了した時点で,その設計図面の承認を受けるため,利用者に対してレビューの依頼を行う.依頼に際しては,レビュー時に必要となる関連情報を設計図面と合わせて添付する場合もある.(2) 利用者は,レビュー依頼の対象となっている設計図面(関連情報も含む)を参照してレビューを実施し,その結果(承認,もしくは再提出)を提供者に通知する.通知に際しては,設計図面に対するコメントや赤入れ内容等の情報を付加して通知する場合もある.(3) 提供者は,レビュー結果,および付加された情報(コメントや赤入れ内容)を参照し,再提出の場合はその内容を設計に反映し,再度レビューの依頼を行う.b. システム機能と効果設計承認業務を行っていく場合,以下のシステム機能が完備されている.(1) 企業間統合情報システムで設計承認業務を実施することを企業間で合意する場合,次の内容を含めた事前合意を行う.1システムの運用管理者(例えば,提供者がシステムを運用・管理するという合意)2システムを利用するための情報(例えば,利用時のユーザID,パスワード等の情報の合意)3レビュー対象となる技術情報に関連した管理情報(例えば,技術情報を識別したり,利用するための情報の名称,データ形式等)図6・39設計承認業務イメージ図6・40 設計承認業務の流れ7―130生産システム工学4技術情報の授受手順(例えば,図6・41に示すような,設計承認業務がどのような流れで実現されるかを合意)5手順に対応したステータスの内容,遷移,および各ステータスのシステムに対して可能な操作範囲(例えば表6・1によると,レビュー中の場合,提供者からはレビュー対象となっている技術情報への登録や変更ができないが,利用者からはレビュー時のコメントや赤入れの登録ができるなど)(2) 提供者は,レビュー依頼を行うために,設計が完了した技術情報をデータベースに登録する.このとき,この技術情報のステータスは設計中からレビュー依頼中に変わり,レビュー依頼の通知が利用者に送信される.その結果,提供者と利用者間で,企業間統合情報システムによる設計図面の情報共有がなされる.(3) 利用者は,レビュー依頼の通知を受信し,レビュー依頼の対象となる技術情報の内容を確認のうえ,レビュー依頼に対して応諾する.このとき,ステータスはレビュー依頼中からレビュー中に遷移する.(4) 利用者は,依頼された技術情報に対するレビューを行い,承認の場合は,提供者にその通知が送信され,ステータスはレビュー完了となる.また,再提出の場合も,提供者にその通知が送信される.この場合,レビュー時に付加されたコメントおよび赤入れは,この技術情報の添付情報としてデータベースに登録することができ,提供者との間で添付情報の共有が図られる.(5) 提供者は,レビュー結果の通知を受信し,再提出の場合は,利用者が登録した添付情報を基に再設計を行う.このとき,ステータスは再提出から設計中に遷移する.以上,企業間統合情報システムを利用することにより,従来の紙による設計承認業務の問題点が,以下のように解決される.(1) 共有されている技術情報の登録履歴やステータスの変更履歴などの履歴管理(誰が,いつ,何を行ったかという履歴の管理)機能により,技術情報の授受の履歴管理が実現される.(2) 電子的に情報の授受を行うことにより,リードタイムの短縮,ペーパレス化が図られる.(3) 提供者企業の設計ビジネスプロセスと利用者企業の承認ビジネスプロセスの連携が実現される.(4) 企業間の情報共有により,データの発生箇所でデータの一元管理をすることにより,唯一性が保証される.図6・41 設計承認業務の流れの例表6・1設計承認業務におけるステータスとその内容例ステータスステータス内容提供者可能操作利用者可能操作設計中提供者が設計を実施している状態レビュー依頼レビュー依頼中提供者が利用者に対してレビュー依頼した状態表データ/状況の取得・表示表データ/状況の取得・表示レビュー中利用者が提供者のレビュー依頼を応諾した状態表データ/状況の取得・表示レビューレビュー完了レビューにより利用者が承認した状態レビュー結果の取出しレビュー結果登録コメント登録再提出レビューにより利用者が再レビューを指示した状態(提供者が再提出を応諾すると設計中となる)レビュー結果の取出し表データ/状況の取得・表示
