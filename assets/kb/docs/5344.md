a.認証局（CA:certification authority) 7･2･2項で述べた
公開鍵暗号方式を使うと，相手認証，鍵交換，否認防止，電子署名
等ができることがわかった．ところが実際に公開鍵方式を使ってシ
ステムを構築する場合，公開鍵をどのように得るか，その保有者を
どのように信用するかという問題が発生する．例えば，ある人に成
り済まそうとする場合，自分の公開鍵を成り済ましたい人のものと
宣言して利用することができてしまうからである．そこで，公開鍵
を保証する機関として認証局がある．
認証局では，秘密鍵・公開鍵ペアの保有者から，認証情報，識別
子，公開鍵を受取り，保有者を認証した後，公開鍵に情報を付加し
た公開鍵証明書を生成し，認証局自身の秘密鍵を使って署名付けを
行う．公開鍵証明書は，一般には，認証局に取りに行くインタフェ
ースを設けるか，ディレクトリ等を用いて誰でも取得できるように
する．公開鍵利用者は，その公開鍵に対応する公開鍵証明書を取得
し，認証局の電子署名を検証することによりその公開鍵が正当な所
有者のものであるかどうかを認証する．このように，認証局によっ
て発行された公開鍵証明書を使うことにより，正しい公開鍵を利用
することができる．
認証局の秘密鍵は，証明書への署名付けを行うための重要なもの
であり，漏えい，破損がないように厳重に管理されることが要求さ
れる．そのため，専用ハードウェアで管理される場合もある．
b.認証局の構成 認証局を利用する際には，さらに認証局自
体をどうやって信用したらよいのかという問題が発生する．認証局

（自己署名）する．この認証局はルート認証局と呼ばれ，各社で運

い．例えばブラウザには前もっていくつかのルート認証局の証明書
が組込まれている．公開鍵証明書の利用者は，検証したい公開鍵証

c.公開鍵証明書 公開鍵証明書には，公開鍵保有者を確認す
るための情報，証明書の有効期限，証明書のポリシー，公開鍵に関
する情報，認証局に関する情報が含まれる．このフォーマットは，
ITU-T X. 509あるいはISO/ IEC9594-8：2001で規定されている．
証明書に有効期限を設けることにより，有効期限が切れた場合は再
度鍵保有者を認証し直す，あるいは公開鍵・秘密鍵ペアを作成し直
すといったことが可能になる．証明書の拡張領域（エクステンショ
ン）には，ポリシーに関する記述を設定できる．ポリシーエクステ

CPSの場合はURIも可）を指定する．ポリシーマッピングエクス
テンションは，CAの証明書において，自CAのあるポリシーに関
し，他のCAに関連するポリシーを同様と扱ってもよい場合にそれ
らのポリシーをOIDの組で指定するものである．それによって，
異なるCA間でも，自CAでのポリシーを利用した証明書の使用が
可能になる．また，ポリシー制約エクステンションを利用して，ポ
リシーマッピングを禁止したり，検証の経路上の証明書のポリシー
の整合性チェックを要求する指定も可能である．従来はCP，CPS
はテキストにて記述されているが，形式的言語で記述しようという
提案やポリシーの記述をXML（extensible markup language）で
行おうという提案もある．

authority）がある．そのほかシステム・製品によって，検証局
（VA:validation authority）がある．
EEは証明書を保有するものを指し，自分では証明書を発行しな
いものと定義される．システムを利用する人，機器，クライアント
ソフトウェア，サーバなどがある．RPは証明書を検証して利用す
るエンティティを指し，EEと同一になる場合もある．RAはEE

e .証明書の発行 EEは証明書保有者および鍵の保有者であ
るが，本項では証明書の発行を説明するため，以下証明書申請者と
呼ぶ．証明書を利用するには，まず証明書申請者が秘密鍵・公開鍵
ペアを生成する．公開鍵暗号方式は，暗号化に使われる場合と署名
に使われる場合があるが，署名で使われる場合はこの鍵ペアは証明
書申請者側のソフトウェアあるいはハードウェアで生成される．た
だし，証明書申請者側で生成する能力がない場合は外部で鍵生成が
行われる．一方，この鍵ペアを暗号化用に利用する場合は，暗号化
鍵（秘密鍵）を紛失した場合の対処として鍵バックアップ，鍵リカ
バリ機能が必要になるため，外部でこれらの機能を備えて鍵生成を
行い，生成した鍵ペアを証明書申請者側に送付する方式もある．秘
密鍵は証明書申請者で秘密に保持しなくてはならないものであるた
め，外部で鍵を生成する場合は特に鍵配送の方式を考慮しなくては
ならない．
証明書申請者は秘密鍵・公開鍵を生成すると次に証明書の発行申
請を行い，CAから証明書を受取る．
ここで証明書の発行依頼・受信プロトコルとしては，証明書申請
者，CA間で規定された二者間モデルと，証明書申請者，RA，CA
間で規定された三者間モデルがある．
二者間モデルは，CAが証明書申請者の公開鍵・秘密鍵ペアが正
しいことを確認した後に証明書等を発行するモデルである．証明書
申請者は，公開鍵を含んだ証明書申請要求に署名付けをしたものを
CAに送付し，CA側で署名検証後証明書を発行する．この場合，
CAは証明書申請者の正当性を確認することはできないため，この
モデルは証明書の再発行（更新）や失効情報リクエストに使われ
る．証明書の新規発行には，証明書申請者，RA，CAの三者間モ
デルが使われる．
三者間モデルのおもな方式としては，証明書申請者が自身の公開
鍵ペアを生成後，RAに対して証明書申請を出し，RAでは証明書
申請者の審査を行った後，署名付け申請リクエストをCAに送り
CAでは証明書生成後署名付けでデータをRAに返し，RAから証
明書申請者に証明書を返す方式がある．また，RAは，証明書申請
者を審査した後，一時的な暗号化鍵を生成して証明書申請者，CA
にそれぞれ渡し，その後，証明書申請者とCA間で暗号化鍵を用い
て通信を行う方式や，証明書申請者は公開鍵ペアを生成後，CAに
対して証明書申請を出し，CAではRAに対して証明書申請者の審
査を依頼し，結果を得てから証明書を発行する方式がある．
各登場者（証明書申請者，RA，CA）間で証明書の申請，受領
を行うためのデータフォーマットを規定したものとして，PKCS
（public key cryptography standards ）#10:Certification
Request Syntax Version 1.5（RFC 2314：1998），CRMF
（certificate request message format ，RFC2511：1999）がある．
また，任意のデータを暗号化，署名付けなどをするはん用的なフ
ォーマットとしてPKCS#7:Cryptographic Message Syntax
Version1.5（RFC2315：1998)，PKCS#7を包含した形のCMS
（cryptographic message syntax，RFC 2630：1999）がある．
PKCS#7は，CAから証明書を返す際のデータフォーマットとし
てよく利用される．
それらのデータを使ったプロトコルとしては，CRSS
（certification request syntax standard)，CMC（certificate man-
agement using CMS)，SCEP（simple certificate enrollment
protocol )，CMP（certificate management protocol ，RFC
2510：1999）がある．
証明書発行要求はCSR（certificate signing request ）と呼ばれ
ることもある．
f .失効情報の管理 いったん証明書を発行した後，何らかの
理由（例えば，秘密鍵の紛失や漏えい，公開鍵証明書の記載内容の
変更，鍵・証明書保有者が証明書を利用するサービスからの脱退，
認証局の鍵の危殆化等）によりその鍵ペア・公開鍵証明書の利用を
停止させたい場合がある．そのため，認証局では失効情報を管理す
る．失効情報の通知方法としては，認証局により失効した証明書の
情報を失効リストとして発行する方式と随時認証局にその証明書が
失効していないかを問合わせる方式がある．
失効リスト（CRL:certificate revocation list ）は，公開鍵証明
書を発行した認証局により署名が付けられ，定期的に発行される．
一般的には誰でも参照できるように公開される．失効リストには，
次回リスト生成時刻や，失効した証明書の識別番号，失効理由等の
エントリが含まれる．このリストは，証明書を発行した時点から有
効期限内の証明書のうち失効されたものをすべて保持しなくてはな
らないため莫大なサイズになりアクセスの際のオーバヘッドも大き
い．よって，CRLを分割し，証明書内のCRL distributed point
（配布点）にてCRLの配布場所を指定することにより分散配置す
る方法や，前回発行した失効リストからの差分を搭載するデルタ失
効リストもある．また，失効情報をツリー構造で管理するCRT
（certificate revocation tree）等の方式もある．さらに，階層が異
なる認証局で発行されたCRLを取扱えるようにするために複数の
CAが発行したCRLをまとめて1箇所に集めたIndirect （間接）
CRLがある．このCRLには証明書発行者とは異なる署名が付くこ
とになる．
一方，随時証明書の失効状態を問合わせる方式としては，OCSP
（online certificate status protocol )，SCVP（simple certificate
validation protocol )，DPV（delegated path validation)，DPD
（delegated path discovery）がある．OCSPは，OCSPレスポン
ダがCRLをもち，証明書利用者は証明書の状態をオンラインで問
合わせる方式であり，RFC2560：1999で定義されている．証明書
の識別子，発行者のDN（distinguished name）のハッシュ，発行
者の公開鍵のハッシュ，証明書のシリアル番号のリストで問合わせ
る．一般的には，CAとは明示的に分離したVAを設けてこの役割
を担う．この方式により，証明書検証者は自身でCRLを毎回取得
し検証する必要がなくなるが，証明書検証者の証明書の発行者と検
証対象の証明書の発行者が異なる場合，検証者は証明書の階層チェ
ックをするために異なるOCSPレスポンダに問合わせなくてはな
らないという問題がある．
CAの階層をたどり各OCSPレスポンダに問合せ結果を返すよう
なものとしてSCVPが提案された．SCVPの目的は，クライアン
トの証明書検証者であるクライアントの検証のための負荷を削減す
ることと，信頼，ポリシー管理を集中管理したいというところにあ
る．検証したい証明書，有効期限，ポリシーID，中間証明書や信
頼できるルートを渡し，問合せの状況や結果を返す．おもにhttp
プロトコル上に実装される．
その他の方式として，DPVやDPDがある．DPVは，証明書の
有効性（有効，失効，不完全）を確認するインタフェースである．
検証する証明書，検証ポリシー，パス参照の返却の必要性，パス値
の返却の必要性，タイムスタンプがなされる必要性などを受付け，
処理状況のステータスと検証結果を返す．DPDは，ローカルで証
明書の検証を行うために証明書のパスを得るためのものである．検
証する証明書の識別子，使われるルート証明書や検証ポリシーの情
報，失効情報の返却の必要性などを受付け，処理状況のステータス
と結果（パスの情報，失効の情報）を返す．これは，http，
snmp，ldapプロトコル上に実装される．

col ）を規定している．X. 500で規定されたDAPは非常に多くのリ

1997として標準化されており，現在多くのディレクトリ製品で使
