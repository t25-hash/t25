# 2・2・2 CPU

> **章**: 第2章 コンピュータ
> **節**: 2・2 コンピュータの基礎

a. データ形式CPU が処理対象とするデータは,‘0’または‘1’の値をもつビット(bit)の並びとして与えられる.標準的な長さ(ビット数)をもつデータはワード(word)と呼ばれ,データ間の演算は基本的にワード単位に行われる.また,主記憶装置上の位置を指定するアドレスにもワード長のビット列が用いられることが多い.一般にワード長が大きいコンピュータほど,より広い空間のデータにアクセスでき,より多くのデータを一度に処理できるので,能力が高い.コンピュータの発達とともにワード長は増加してきており,現在は32ビットないし64ビットが一般的である.コンピュータに意味のある処理をさせるためには,単なるビット列であるデータに,文字や数値,さらには画像データや音声データなど図2・1 コンピュータの基本構成図2・2 整数データの形式の意味付けを行う必要がある.ここでは,最も基本的な数値データについて簡単に紹介する.. 整数図2・2は32ビットで整数を表現する一般的な形式である.左端の1ビットは符号を意味し,‘0’は正数,‘1’は負数を表す.残りの31ビットを用いて数値を2進数表現する.ただし,負数の場合は2の補数表現を用いるのが一般的である.2の補数表現を得るには,2進表現の各けたを反転(0を1に,1を0に)させ,最下位けたに1を加算すればよい.この形式で,-2~2-1の整数値を表現することができる.. 浮動小数点数浮動小数点数の表現法はいくつか存在するが,ここでは最も標準的なIEEEの単精度(32ビット)と倍精度(64ビット)の形式を紹介する.図2・3に示すように,いずれの形式もデータを符号部,指数部,仮数部に分割して表現する.それぞれのビット数は,単精度が符号部1ビット,指数部8ビット,仮数部23ビット,倍精度が符号部1ビット,指数部11ビット,仮数部52ビットである.表現される数値は,単精度が-12욓1.,倍精度が-12욓1.である.b. 命令セットここでは,命令の形式や一般的な命令セットに含まれるおもな命令について説明する.個々の命令セットアーキテクチャによって細かい部分に違いはあるが,おおむねどれも同様の体系を有している.. 操作コードとオペランド命令も普通のデータと同様に,ある長さをもったビット列として表現される.どの命令も同じビット長をもつ固定長命令セットと,命令の種類によってビット長が異なる可変長命令セットがある.命令のビット列は,それぞれの命令の種類を表す操作コード(operation code)部と操作の対象となるデータ等を指定するオペランド(operand)部に大別される.命令によってはオペランドを複数もつものもある.図2・4に典型的な命令の形式を示す.命令の種類には,加算や乗算などの演算命令,主記憶にアクセスする命令,v項で示す分岐命令などがある.オペランドにはレジスタ番号を指定するレジスタオペランド,ビット列が表す値そのものを意味する即値オペランドなどがある.. レジスタ一般にCPU の動作速度に比べて主記憶アクセスは時間がかかるので,CPU が処理対象とするデータをそのたびに主記憶から読み書きするのは効率が悪い.そこで,より高速にアクセス可能な小容量の記憶領域をCPU の近くに備えるのが普通である.これをレジスタ(register)と呼ぶ.レジスタは整数デー図2・3 浮動小数点データの形式図2・4 命令形式の例タ用と浮動小数点数データ用をそれぞれ32個ないし64個程度もつことが多い.レジスタに格納されたデータをオペランドとして用いる場合は,命令中のオペランド部で(例えば32個なら0~31までの)番号で指定する.また,これらのはん用レジスタ以外に特殊な用途に用いる専用レジスタもある.. アドレシング主記憶アクセスを伴う命令はオペランド部によってアクセスするアドレスを表現する必要がある.これをアドレシング(addressing)と呼ぶ.アドレシングの方式として絶対アドレス方式と相対アドレス方式がある.前者はオペランド部によって計算される値がそのまま主記憶アドレスとして用いられる方式,後者はその値がある基準となるアドレスからの相対値(オフセット,変位などとも呼ぶ)として用いられる方式である.基準としては,特定のレジスタの値や命令自身の主記憶上のアドレス(プログラムカウンタ)などがある.. RISCとCISC主記憶アクセスを主記憶とレジスタ間のデータ転送に限定した命令セットと,演算のオペランドとして主記憶上のデータを指定できる命令セットがある.前者は命令セットをなるべく単純化することでCPU の動作周波数向上をねらったもので,RISC(reduced instruction set computer)と呼ばれ,後者はそれに対してCISC(complex instruction set computer)と呼ばれている.RISCは1980年代から爆発的な広がりを見せたワークステーション分野で広く採用され,代表的なものにSPARCアーキテクチャ,MIPS アーキテクチャなどがある.また,後者のおもなものにはIBM の370アーキテクチャ,インテルの8086アーキテクチャなどがある.. 命令実行順序と分岐命令命令自身が存在する主記憶上のアドレスをプログラムカウンタ(program counter)と呼ぶ.CPU が命令を実行するときは,主記憶上で隣り合った位置にある命令がプログラムカウンタの昇順に実行される.すなわち,プログラムカウンタがで長さが4バイトの命令の次に実行されるのは,プログラムカウンタが+4の命令である.この順序を変更する手段として,命令セット中に分岐命令が用意されている.分岐命令は主記憶アクセス命令と同様にオペランドとしてアドレスが計算され,そのアドレスをプログラムカウンタとする命令に制御が移行する.分岐命令は,無条件分岐命令と条件分岐命令に分類することができる.前者が実行されると必ず分岐が起こるのに対し,後者はオペランドとして指定された条件が満たされた場合のみ分岐が起こり,それ以外は通常通りプログラムカウンタ順の実行が継続される.条件としては,条件コードレジスタ(condition code register)と呼ぶ専用のレジスタの特定のビットの値(‘0’か‘1’か)が用いられることが多い.条件コードレジスタへの値の設定は演算命令の結果の正負などが反映される.これらの分岐命令を活用することで,複雑な制御構造をもつプログラムの実行が可能となる.c. パイプラインここではプロセッサの性能向上に必須の手法である,パイプライン(pipeline)処理について述べる.. パイプラインの概要CPU が1個の命令を実行するために行う仕事を考える.レジスタに格納されている2個の値を加算し,結果をレジスタに書き戻す命令を例にすると,次のようになる.(1) 主記憶装置から命令をCPU 内部に取込む(2) 命令の内容を解読する(3) 加算に必要なデータをレジスタから読み出す(4) 加算する(5) 加算の結果をレジスタに書き込むハードウェアでこれら5ステップの仕事を行う場合,それぞれ何らかの時間を要することになる.ここでは簡単のため,どのステップも一定時間だけかかるものとすれば,この加算命令実行に要する時間は5である.仮にどの命令も同様なステップを踏んで実行されるとすれば,個の命令実行に5×の時間がかかることに8―35なる.ここで,(1)~(5)の仕事の中身に注目すると,各ステップで行う仕事はそれぞれ独立していることがわかる.例えば(4)の仕事に用いるリソースは加算器のみであり,別の命令の(4)以外のステップが同時に行われても,それを妨げることはない.この性質を利用すれば,複数の命令の異なる実行ステップをオーバラップさせることが可能になる.これがパイプラインの基本的な考え方である.パイプライン計算機は,命令実行に関わる一連の処理をパイプラインステージ(pipeline stage)と呼ぶいくつかの独立部分に分割し,バケツリレーの要領で命令を次々にパイプラインに投入することで,パイプラインステージのオーバラップを利用して,単位時間当たりに処理される実効的な命令数を増やす方式である.パイプライン計算機および非パイプライン計算機の実行タイミングを図2・5に示す.なお,上記(1)を命令フェッチ(instruction fetch),(2)をデコード(decode),(3)をオペランドフェッチ(operandfetch),(4)を実行(execute),(5)を書戻し(writeback)と呼び,ここでは順にIF,D,OF,EX,WB と略記している.図2・5によれば,パイプライン計算機では,見掛け上は1個の命令が時間で実行されることになり,非パイプライン計算機に比べて大幅な性能向上が達成されることがわかる.現在のプロセッサはすべてこのようにパイプライン化されており,上記のに相当するのがサイクルタイム(動作周波数の逆数)になる.ただし,このようなパイプラインを実現するためには,どのステージにも一定時間を割当てる必要がある.逆にいえば,各ステージの所要時間にばらつきがあると,最も時間がかかるステージに全体を合わせざるを得ない.すなわち,なるべく各ステージの時間を均等に設計することがパイプライン計算機の設計において重要なポイントとなる.実際のCPU ではここで説明したよりもはるかに多段のパイプラインが実現されている例が多い.例えば,インテルのPentium4(ペンティアム4)ではその段数は20段以上にも達する.これは,制御方式が複雑なこと,細かく分割することで動作周波数の向上をねらっていることなどによる.. パイプラインストールと分岐予測パイプラインがスムーズに流れれば理想的であるが,実際にはさまざまな要因によってパイプラインに乱れが発生し,有効な処理ができない空きステージが生じてしまう.これをパイプラインストール(pipeline stall)と呼ぶ.ここでは大きく二つの要因について述べる.一つはOFステージでフェッチするオペランドが読込み可能な状態になっていないケースである.例えば,データをレジスタから読むのに比べ,メモリから読む場合は時間がかかるため,命令がOFステージに達したタイミングではメモリからのデータがCPU 内に到着していないケースがある.また,先行する演算命令の結果をオペランドとして読む場合,演算に時間がかかるためOFステージに到達した時点で演算が完了していないこともある.これらのケースでは,データが読み込める状態になるまで命令の実行がOFステージでストップする.もう一つの要因は,分岐命令に関連するケースである.分岐命令以外の命令ではプログラムカウンタは連続的に変化するので,後続図2・5 パイプライン計算機の実行タイミング8―36情報・メディア機器命令のプログラムカウンタは簡単に計算でき,IFステージが停滞することはない.しかし,分岐命令によってプログラムカウンタが不連続に変化する場合,分岐先のプログラムカウンタが求まるのは分岐命令がパイプラインをかなり進んだ時点(例えばEX ステージ)になるので,分岐先命令のIFステージはかなり遅れることになる.この様子を図2・6に示す.分岐命令のIFステージで何らかの方法で分岐先命令のプログラムカウンタを求めることができれば,後者のストールを回避することができる.しかし,特に条件分岐命令の場合,条件が成立すれば分岐するし,非成立なら分岐しないので,次にフェッチすべき命令のプログラムカウンタは動的に変化し一律ではない.この問題に対処する技術に分岐予測(branch prediction)がある.分岐予測は,個々の条件分岐命令の過去における分岐方向の履歴に基づき,次にその命令が実行されるときの分岐の方向を予測する技術である.分岐命令がフェッチされると速やかに予測を行ってその結果に基づいて後続命令をフェッチするので,予測が的中すればパイプラインストールを避けることができる.ただし,予測が外れた場合はパイプラインのキャンセルなど複雑な処理が必要になる.図2・7に分岐予測の概要を示す.分岐予測は,特にパイプラインステージ数の多いCPU にとって必須の技術であり,現在のプロセッサ製品には必ず実装されている.具体的な予測方法はさまざまな方式が提案されているが,いずれの方式でも,予測しやすいプログラムなら100%に近い確率で当てることができる.. スーパースカラとVLIW複数の命令を並列に動作させてパイプライン計算機の性能を向上させるというアイデアは自然である.例えば図2・8のように,2個の命令を並列に動作させることができれば,性能は単純に2倍になりそうに思える.しかし実際にはこのような並列実行がつねに可能ではなく,ii項のパイプラインストールが発生するケースはもちろん,それ以外にも次のような場合に問題が発生する.(1) 命令間に依存関係がある場合(2) 資源競合が起こる場合(1)はある命令A で計算した結果を別の命令B がオペランドとして参照する場合に生じる[これを真の依存関係(true depen-dency)と呼ぶ].このようなケースでは,命令A のEX ステージが終了しないと命令B のOFステージが開始できないため,これら二つの命令を並列にパイプラインに流すことはできない.(2)は例えば2個の加算命令を並列に実行しようとしても,加算器が1個しかないケースである.この場合,後の命令は加算器が空くまで実行を待たなければならない.このように,複数命令をパイプライン図2・6 分岐先命令実行の遅れ図2・7 分岐予測に投入する場合,これらの状況への対処が必須となる.これをすべてハードウェアで解決しようとするアプローチとしてスーパースカラ(super scalar)方式,ソフトウェアの力を借りて達成しようとするものとしてVLIW(very long instruction word)方式がある.これらの方式は1980年代から1990年代にかけて研究され,1990年代後半から実用化され始めている.スーパースカラプロセッサは,命令間の依存関係やハードウェア資源の使用状況などをハードウェアが自動的に認識し,命令実行を適切に行うように制御する.このため,制御論理は非常に複雑なものになるが,ソフトウェア(コンパイラ)への負担は特にない.これに対して,VLIW プロセッサはあらかじめ並列に実行する命令の組をソフトウェア(コンパイラ)が決定して,そのまま命令コード上に並べる方式をとる.つまり,VLIW プロセッサは複数の命令をパックしたものが実行単位となり,それをそのままパイプラインに流すことで命令が実行される.VLIW プロセッサはスーパースカラのような複雑な制御が不要なので,ハードウェアの作りはシンプルになるが,その分コンパイラの負担が増えるトレードオフがある.実製品では,1990年代中盤以降に出荷されたPentiumやSPARCは,ほとんどがスーパースカラプロセッサである.また,VLIW プロセッサとしてはインテルのIA64シリーズが代表的である.
