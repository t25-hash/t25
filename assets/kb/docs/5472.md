# 7・2・2 暗号技術

> **章**: 第7章 情報圧縮・セキュリティ技術
> **節**: 7・2 情報セキュリティ技術

a. 共通鍵暗号暗号技術は、大きく二つに分けられる。DES(data encryption standard)に代表される共通鍵暗号と、RSA 暗号[RSA:発明者であるリベスト(Rivest)、シャミア(Shamir)、エーデルマン(Adleman)の頭文字]やECC(ellip-tic curve cryptosystem)に代表される公開鍵暗号である。共通鍵暗号(common key cryptosystem,shared keycryptosystem)は、対称鍵暗号(symmetric key cryptosystem)や秘密鍵暗号とも呼ばれ、暗号化する際に使用する鍵と、復号化する際に使用する鍵が同一である特徴を有する。一般に処理が高速なため、データの暗号化や改ざん検証用の検証子(MAC: messageauthentication code)の生成に使用されることが多い。なお、共通鍵暗号で使用される鍵のことを共通鍵(common key,sharedkey)、対称鍵(symmetric key)と呼ぶ。8―128情報・メディア機器さらに、共通鍵暗号は暗号化する方式により大きく二つに分類される。ある程度まとまったブロック単位で暗号化を行うブロック暗号と、1ビットごとに乱数列等と排他的論理和を取って暗号化するストリーム暗号である。前者の代表例がDES で、1977年にアメリカ合衆国商務省標準局NBS(National Bureau of Standards)により標準暗号として制定され、1990年代におもに使用されたが、後に解読法(差分攻撃法、線形解読法)がいくつか提唱され安全性が危ぶまれた[1989年にビハム(Biham)とシャミア(Shamir)によって差分攻撃法が、1993年に松井によって線形解読法が考案される].そのため、NIST(National Institute of Standards andTechnology)によりDES の後継としてAES(advanced encryp-tion standard)が一般から募集され、最終的にラインデール(Rij-ndael)が採択された。なお、AES が広く使われるまでのつなぎとして、Triple-DES(DES を3段繰返し利用する方法)がANSIX9.52として標準化され、使われている。なお、ストリーム暗号に関しては割愛する。ここで、DES を用いたデータの暗号方式について簡単に説明する。まず、DES で暗号化されるブロック長は8バイトであるため、平文データ(D1からDが平文)を8バイト単位に分割する(D1~D-1)。最後のデータ(D)が8バイトに満たない場合は、適当なデータを埋めて8バイトに調整する。次に、IV(initialvector)と暗号化に使用する鍵を暗号者と復号者で決める。実際の暗号化は、最初にD1とIV の排他的論理和を計算し、その出力をDES 演算器に入力し、暗号者と復号者で取り決めた鍵で暗号化する。この出力が第一の暗号文E1である。次に、このE1と次の平文であるD2との排他的論理和を計算し、その出力を先ほどと同じ鍵でDES 暗号化する(出力E2)。以降これらを繰返し、最後の平文Dを暗号化するまで繰返す。この一連の処理で生成されたE1~Eが暗号文になる。この方法により、同じデータが連続して入力された場合でも出力データ(暗号文)が異なるため、入力データ(平文)の推測を難しくすることができる。このような暗号化方式をCBC(cipher block chaining)モードと呼ぶ(図7・8)。ほかにECB モード、CFB モード、OFB モードがある。復号文(平文)を得るには、次のように行う。まず、E1を先ほどと同一の鍵でDES 復号化し、その出力とIV との排他的論理和を取って平文D1を得る。次に、E2をDES 復号化し、その出力とE1との排他的論理和を計算して平文D2を得る。以降、これらを繰返し、平文Dまで得る。最後のDには、ブロック長を8バイトにするための調整データが含まれている場合があるので、そのデータを削除し、元の平文を完全に得る。次に、DES-CBCを用いたメッセージ検証子MACの計算方法および改ざん検出方法を説明する。はじめに、暗号化処理と同じようにメッセージを8バイトごとのブロックに分割する。最後のブロックが8バイト未満の場合は、取り決めに従ったデータを埋める。次に、メッセージ送信者と受信者で取り決めたIV と暗号鍵を用い、DES-CBCで暗号化処理を行う。そして、最後に出てきた値Eをメッセージ検証子MACとし、メッセージに添付して受信者に送付する。これを受信した受信者は、送信者と同様にDES-CBCで暗号化処理を行い、最後に出てきた値Eが受信したメッセージ検証子MACと同じかどうか比較する。同一であれば改ざんがなされていないと判断し、異なっていればどこかのブロックが改ざんされていると判断する。最後に、長いデータを暗号化する際の代表的な方式を説明する。図7・9に示すように、まずデータを適当なサイズに分割する。このサイズはデータの利用しやすさを基準に適宜決めてかまわない。次に、乱数によりブロック鍵を生成し、この鍵で1データブロックを暗号化する(暗号化方法は例えば上記DES-CBC等)。そして、ブロック鍵はデータ鍵で暗号化される(ブロック鍵の暗号化はTriple-DES 相当の強度が好ましい)。以降、第2ブロックから第ブロックまで暗号化し、必要に応じてメッセージ図7・8 DES-CBC モード図7・9長大データの暗号化検証子MACも付加する。これにより、処理速度をさほど落とさず、かつデータの暗号強度を強化することが可能となる。ブロック鍵はDES 相当の鍵強度であるため解読される危険性があるが、その影響範囲は限定的である。一方、データ鍵はTriple-DES 相当になっているため解読されにくい。また、データそのものはDESで暗号化され、ブロック鍵のみTriple-DES で暗号化されているため、処理速度はDES の暗号化速度とほぼ同じである。b. 公開鍵暗号公開鍵暗号(public key cryptosystem)は、非対称鍵暗号(asymmetric key cryptosystem)とも呼ばれ、暗号化する際に使用する鍵と、復号化する際に使用する鍵が異なる特徴を有し、一般に処理負荷が大きい。公開鍵暗号で使用される鍵のうち、広く公開する鍵を公開鍵(public key)、利用者が秘密裏に保持する鍵を秘密鍵(private key,secret key)と呼ぶ(共通鍵暗号における鍵を秘密鍵と呼ぶ場合もあるが、公開鍵暗号における秘密鍵と混同する場合があるため、前者は共通鍵や対称鍵と呼ぶほうがよい)。公開鍵暗号の代表的な暗号方式としては、RSA(Rivest ShamirAdleman)暗号とECC(elliptic curve cryptosystem)がある。RSA 暗号は1980年代から今日まで使われ続けてきたが、高速素因数分解法(おもに数体ふるい法)の進歩によりその安全性が危ぶまれ、徐々に鍵長が長くなってきている。実際、当初は512ビットの鍵長が使われていたが、2000年に入って1024ビット、2048ビットの鍵長が使われ始めている。鍵長の伸長に伴い、演算速度の低下やハードウェア化の困難さが指摘され、他の暗号方式への移行が模索されている。一方ECCは、歴史的には1985年にコブリッツ(Koblitz)とミラー(Miller)により独立して提案されていたものの、実用化され始めたのは1990年代ごろからであり、その意味で実績は少ない。しかしRSA 暗号の1024ビットに対し、ECCの160ビットが同程度の強度を有するといわれているように、ハードウェア化しやすく、高速に演算できることから、今後大いに使われていくものと考えられる。公開鍵暗号の特徴は、暗号鍵と復号鍵が異なることである。このことを利用し、暗号鍵として公開鍵を、復号鍵として秘密鍵を用い、メッセージを特定の人にだけ安全に送ることができる。例えば、A さんが公開鍵、秘密鍵のペアをもっていて、B さんがA さんにメッセージを秘密裏に送る場合、A さんの公開鍵でメッセージを暗号化してA さんに送信する。すると、このメッセージを復号化できるのはA さんだけなので、親展としてメッセージを送ることができたことになる(秘密鍵はA さんしか知り得ないため)。なお、公開鍵暗号を用いてメッセージ全体を暗号化するのは非効率であるため、共通鍵でメッセージを暗号化し、この共通鍵を公開鍵で暗号化して送信するほうがよい。ここで、簡単に暗号化方式を説明する。次に公開鍵暗号を用いた電子署名について説明する。メッセージ検証子MACにおいては、送信者と受信者が同一の鍵を所有しているため、送信者側が送信したデータを受信者側で改ざんすることができる。このため、送信者が後になって、データの送信を否認することも可能となる(データおよびMACを受信者が作ったと主張すること)。これに対し、公開鍵暗号を用いた電子署名は改ざんと否認を防止することができる。例えば、A さんが公開鍵および秘密鍵のペアをもっていると仮定する。そして、A さんからB さんにメッセージを送信するとき、まずA さんはメッセージのハッシュ値を計算する。ハッシュ値とは、SHA-1(secure hash algorithm)やMD 4(message digest algorithm)、MD 5などの一方向性アルゴリズムを用いて計算した出力値のことである。次に、このハッシュ値からA さんがもつ秘密鍵を用いて電子署名を計算する。最後に、A さんからB さんにメッセージと電子署名を送信する。一方、これを受信したB さんは、A さんの公開鍵を用いて電子署名の正当性を検証する。これにより、B さんはメッセージが改ざんされていないか、このメッセージをA さんが作ったかどうかが判定できる。もし、電子署名が正当であれば、電子署名を作れるのはA さんだけである(MACのときと異なり、受信者であるB さんは電子署名が作れない)ため、否認も防止できる。ここで、署名生成・検証方式を簡単に説明する。RSA 暗号においては、電子署名は、ハッシュ値を秘密鍵で暗号化することにより生成される。一方、電子署名を検証するには、この値を公開鍵で復号化し、元のハッシュ値と比較することにより行われる(同一であれば正当であると判定できる)。
