# 7・2・2 項で述べた

> **章**: 第7章 情報圧縮・セキュリティ技術
> **節**: 7・2 情報セキュリティ技術

公開鍵暗号方式を使うと、相手認証、鍵交換、否認防止、電子署名等ができることがわかった。ところが実際に公開鍵方式を使ってシステムを構築する場合、公開鍵をどのように得るか、その保有者をどのように信用するかという問題が発生する。例えば、ある人に成り済まそうとする場合、自分の公開鍵を成り済ましたい人のものと宣言して利用することができてしまうからである。そこで、公開鍵を保証する機関として認証局がある。認証局では、秘密鍵・公開鍵ペアの保有者から、認証情報、識別子、公開鍵を受取り、保有者を認証した後、公開鍵に情報を付加した公開鍵証明書を生成し、認証局自身の秘密鍵を使って署名付けを行う。公開鍵証明書は、一般には、認証局に取りに行くインタフェースを設けるか、ディレクトリ等を用いて誰でも取得できるようにする。公開鍵利用者は、その公開鍵に対応する公開鍵証明書を取得し、認証局の電子署名を検証することによりその公開鍵が正当な所有者のものであるかどうかを認証する。このように、認証局によって発行された公開鍵証明書を使うことにより、正しい公開鍵を利用することができる。認証局の秘密鍵は、証明書への署名付けを行うための重要なものであり、漏えい、破損がないように厳重に管理されることが要求される。そのため、専用ハードウェアで管理される場合もある。b. 認証局の構成認証局を利用する際には、さらに認証局自体をどうやって信用したらよいのかという問題が発生する。認証局も上位の認証局から公開鍵証明書(public key certificate)を発行してもらうことにより、自身の公開鍵を保証する。上位の認証局はさらに上位の認証局により公開鍵証明書を発行してもらうということになるが、いちばん上の認証局は自分自身で公開鍵証明書に署名(自己署名)する。この認証局はルート認証局と呼ばれ、各社で運営されている認証局で通常は唯一存在する。このルート認証局の証明書は、何らかの形で事前に証明書利用者に渡されるケースが多い。例えばブラウザには前もっていくつかのルート認証局の証明書が組込まれている。公開鍵証明書の利用者は、検証したい公開鍵証図7・10 Kerberos8―130情報・メディア機器図7・11 認証局概念図明書を発行した認証局のルート認証局の証明書にたどり着くまで上位の認証局をたどって検証する。ところが、実際には世の中にいくつものルート認証局があるため、証明書利用者がすべてのルート証明書を保持し、その関係を管理することは事実上不可能である。そこで、異なるルートをもつCA 間で発行された証明書もスムーズに検証できるような仕組みが必要になる。方式としては、複数のCA が相互に認証する方式、CA 間の認証を担うブリッジCA を用いる方式がある。相互認証方式では、CA 間で相互に公開鍵を交換して公開鍵証明書を発行する。一方ブリッジCA は、各CA 間が相互認証するのではなく、各CA がブリッジCA と認証を行うことによりCA 間の相互認証を可能とした方式である(図7・11)。認証局は第三者機関として公正性が要求され、また定期的に特定の監査を受けることによりその信頼性を確保することが要求される。認証局は、発行する証明書の発行方法(発行する際の認証方法など)や利用アプリケーション、利用方法などを証明書ポリシー(CP: certificate policy)として定める。また認証局の運用に関しては、運用規定(CPS : certification practice statement)を定める。証明書ポリシーおよび運用規程用のガイドラインとしては、RFC 2527:1999があり、多くの認証局のCP,CPS はこのRFCをベースに書かれている。CP,CPS は証明書の利用者が確認・特定できるように、オブジェクト識別子(OID)を取得し、公開される。c. 公開鍵証明書公開鍵証明書には、公開鍵保有者を確認するための情報、証明書の有効期限、証明書のポリシー、公開鍵に関する情報、認証局に関する情報が含まれる。このフォーマットは、ITU-T X.509あるいはISO/IEC 9594-8:2001で規定されている。証明書に有効期限を設けることにより、有効期限が切れた場合は再度鍵保有者を認証し直す、あるいは公開鍵・秘密鍵ペアを作成し直すといったことが可能になる。証明書の拡張領域(エクステンション)には、ポリシーに関する記述を設定できる。ポリシーエクステンションではb項で述べたCP,CPS(それぞれのOID またはCPS の場合はURIも可)を指定する。ポリシーマッピングエクステンションは、CA の証明書において、自CA のあるポリシーに関し、他のCA に関連するポリシーを同様と扱ってもよい場合にそれらのポリシーをOID の組で指定するものである。それによって、異なるCA 間でも、自CA でのポリシーを利用した証明書の使用が可能になる。また、ポリシー制約エクステンションを利用して、ポリシーマッピングを禁止したり、検証の経路上の証明書のポリシーの整合性チェックを要求する指定も可能である。従来はCP,CPSはテキストにて記述されているが、形式的言語で記述しようという提案やポリシーの記述をXML(extensible markup language)で行おうという提案もある。d. PKI のエンティティこのように、公開鍵方式をシステムに利用する際に必要となるインフラストラクチャ、つまり認証局をはじめとした基本構成要素をPKI(public key infrastructure)という。PKIを構成するエンティティとしては、エンドエンティティ(EE: end entity)、検証者(RP: relying party)、登録局(RA : registration authority)、認証局(CA : certificationauthority)がある。そのほかシステム・製品によって、検証局(VA : validation authority)がある。EEは証明書を保有するものを指し、自分では証明書を発行しないものと定義される。システムを利用する人、機器、クライアントソフトウェア、サーバなどがある。RPは証明書を検証して利用するエンティティを指し、EEと同一になる場合もある。RA はEEへ証明書を発行するに当たり、発行の受付け、審査、CA への証明書発行要求を行う。CA の主たる業務は証明書発行であるため、発行局(IA : issuing authority)として明示的に定義する場合もある。その場合は、IA とRA を合わせてCA と呼ぶ。VA は、証明書の有効性を確認するエンティティである。e. 証明書の発行EEは証明書保有者および鍵の保有者であるが、本項では証明書の発行を説明するため、以下証明書申請者と呼ぶ。証明書を利用するには、まず証明書申請者が秘密鍵・公開鍵ペアを生成する。公開鍵暗号方式は、暗号化に使われる場合と署名に使われる場合があるが、署名で使われる場合はこの鍵ペアは証明書申請者側のソフトウェアあるいはハードウェアで生成される。ただし、証明書申請者側で生成する能力がない場合は外部で鍵生成が行われる。一方、この鍵ペアを暗号化用に利用する場合は、暗号化鍵(秘密鍵)を紛失した場合の対処として鍵バックアップ、鍵リカバリ機能が必要になるため、外部でこれらの機能を備えて鍵生成を行い、生成した鍵ペアを証明書申請者側に送付する方式もある。秘密鍵は証明書申請者で秘密に保持しなくてはならないものであるため、外部で鍵を生成する場合は特に鍵配送の方式を考慮しなくてはならない。証明書申請者は秘密鍵・公開鍵を生成すると次に証明書の発行申請を行い、CA から証明書を受取る。ここで証明書の発行依頼・受信プロトコルとしては、証明書申請者、CA 間で規定された二者間モデルと、証明書申請者、RA,CA間で規定された三者間モデルがある。二者間モデルは、CA が証明書申請者の公開鍵・秘密鍵ペアが正しいことを確認した後に証明書等を発行するモデルである。証明書申請者は、公開鍵を含んだ証明書申請要求に署名付けをしたものをCA に送付し、CA 側で署名検証後証明書を発行する。この場合、CA は証明書申請者の正当性を確認することはできないため、このモデルは証明書の再発行(更新)や失効情報リクエストに使われる。証明書の新規発行には、証明書申請者、RA,CA の三者間モデルが使われる。三者間モデルのおもな方式としては、証明書申請者が自身の公開鍵ペアを生成後、RA に対して証明書申請を出し、RA では証明書申請者の審査を行った後、署名付け申請リクエストをCA に送りCA では証明書生成後署名付けでデータをRA に返し、RA から証明書申請者に証明書を返す方式がある。また、RA は、証明書申請者を審査した後、一時的な暗号化鍵を生成して証明書申請者、CAにそれぞれ渡し、その後、証明書申請者とCA 間で暗号化鍵を用いて通信を行う方式や、証明書申請者は公開鍵ペアを生成後、CA に対して証明書申請を出し、CA ではRA に対して証明書申請者の審査を依頼し、結果を得てから証明書を発行する方式がある。各登場者(証明書申請者、RA,CA)間で証明書の申請、受領を行うためのデータフォーマットを規定したものとして、PKCS(public key cryptography standards)#10: CertificationRequest Syntax Version1.5(RFC2314:1998)、CRMF(certificate request message format,RFC 2511:1999)がある。また、任意のデータを暗号化、署名付けなどをするはん用的なフォーマットとしてPKCS #7: Cryptographic Message SyntaxVersion 1.5(RFC 2315:1998)、PKCS #7を包含した形のCMS(cryptographic message syntax,RFC2630:1999)がある。PKCS #7は、CA から証明書を返す際のデータフォーマットとしてよく利用される。それらのデータを使ったプロトコルとしては、CRSS(certification request syntax standard)、CMC(certificate man-agement using CMS)、SCEP(simple certificate enrollmentprotocol)、CMP(certificate management protocol,RFC2510:1999)がある。証明書発行要求はCSR(certificate signing request)と呼ばれることもある。f. 失効情報の管理いったん証明書を発行した後、何らかの理由(例えば、秘密鍵の紛失や漏えい、公開鍵証明書の記載内容の変更、鍵・証明書保有者が証明書を利用するサービスからの脱退、認証局の鍵の危殆化等)によりその鍵ペア・公開鍵証明書の利用を停止させたい場合がある。そのため、認証局では失効情報を管理する。失効情報の通知方法としては、認証局により失効した証明書の情報を失効リストとして発行する方式と随時認証局にその証明書が失効していないかを問合わせる方式がある。失効リスト(CRL: certificate revocation list)は、公開鍵証明書を発行した認証局により署名が付けられ、定期的に発行される。一般的には誰でも参照できるように公開される。失効リストには、次回リスト生成時刻や、失効した証明書の識別番号、失効理由等のエントリが含まれる。このリストは、証明書を発行した時点から有効期限内の証明書のうち失効されたものをすべて保持しなくてはならないため莫大なサイズになりアクセスの際のオーバヘッドも大きい。よって、CRLを分割し、証明書内のCRL distributed point(配布点)にてCRLの配布場所を指定することにより分散配置する方法や、前回発行した失効リストからの差分を搭載するデルタ失効リストもある。また、失効情報をツリー構造で管理するCRT(certificate revocation tree)等の方式もある。さらに、階層が異なる認証局で発行されたCRLを取扱えるようにするために複数のCA が発行したCRLをまとめて1箇所に集めたIndirect(間接)CRLがある。このCRLには証明書発行者とは異なる署名が付くことになる。一方、随時証明書の失効状態を問合わせる方式としては、OCSP(online certificate status protocol)、SCVP(simple certificatevalidation protocol)、DPV(delegated path validation)、DPD(delegated path discovery)がある。OCSPは、OCSPレスポンダがCRLをもち、証明書利用者は証明書の状態をオンラインで問合わせる方式であり、RFC2560:1999で定義されている。証明書の識別子、発行者のDN(distinguished name)のハッシュ、発行者の公開鍵のハッシュ、証明書のシリアル番号のリストで問合わせる。一般的には、CA とは明示的に分離したVA を設けてこの役割を担う。この方式により、証明書検証者は自身でCRLを毎回取得し検証する必要がなくなるが、証明書検証者の証明書の発行者と検証対象の証明書の発行者が異なる場合、検証者は証明書の階層チェックをするために異なるOCSPレスポンダに問合わせなくてはならないという問題がある。CA の階層をたどり各OCSPレスポンダに問合せ結果を返すようなものとしてSCVPが提案された。SCVPの目的は、クライアントの証明書検証者であるクライアントの検証のための負荷を削減することと、信頼、ポリシー管理を集中管理したいというところにある。検証したい証明書、有効期限、ポリシーID、中間証明書や信頼できるルートを渡し、問合せの状況や結果を返す。おもにhttpプロトコル上に実装される。その他の方式として、DPV やDPD がある。DPV は、証明書の有効性(有効、失効、不完全)を確認するインタフェースである。検証する証明書、検証ポリシー、パス参照の返却の必要性、パス値の返却の必要性、タイムスタンプがなされる必要性などを受付け、処理状況のステータスと検証結果を返す。DPD は、ローカルで証明書の検証を行うために証明書のパスを得るためのものである。検8―131証する証明書の識別子、使われるルート証明書や検証ポリシーの情報、失効情報の返却の必要性などを受付け、処理状況のステータスと結果(パスの情報、失効の情報)を返す。これは、http,snmp,ldapプロトコル上に実装される。g. 証明書の利用証明書の利用者は、証明書の署名検証、証明書の有効期限をチェック、証明書保有者名(subject name)を確認、証明書発行者名(issuer name)を確認、失効されていないかどうかの確認、拡張領域にポリシーに関する記述がある場合はポリシーのチェック、その他の拡張領域に設定されている情報に応じてパス制約のチェックなどを行う。b項で述べたように、検証したい証明書を発行した認証局の証明書をルート認証局の証明書にたどり着くまで検証する必要がある。h. PKI の関連技術証明書や失効リストのリポジトリ(格納庫)としてディレクトリが一般的に使われる。ディレクトリはITU-T X.500にてディレクトリでのデータ格納方式と、格納されているデータへのアクセス方式DAP(directory access proto-col)を規定している。X.500で規定されたDAPは非常に多くのリソースを要するため、軽量版としてLDAP(lightweight direc-tory access protocol)が開発された。LDAP v3は、RFC 2251:1997として標準化されており、現在多くのディレクトリ製品で使われている。PKIを構成する他の機能としては、秘密鍵のアーカイブやタイムスタンプがある。公開鍵ペアを暗号化目的で利用する場合、復号に使う秘密鍵を紛失してしまうとデータを復号できなくなるという問題が発生する。そこで、秘密鍵を暗号化して保存し、必要に応じて復旧するというサービスを提供する場合がある。通常は、認証局の機能とは別に定義される。タイムスタンプは、PKIシステムにおいても重要な機能要素となる。なぜなら電子署名された時刻が意味をなすからである。RFC3161:2001で、TSA(time stamp authority)の要件やTSA へのタイムスタンプの発行要求・受取りインタフェースが規定されている。関連した情報として、電子署名された文書を長期保存する際の技術として、長期間正当性を維持できる仕組みを規定したIETF Internet Draft(IETFインタネット草案、electronic sig-nature formats for long term electronic signatures)や信頼された第三者機関となるサーバである時点の公開鍵証明書の署名を検証した後にデータ検証証明書DVC(data validation certificate)を発行する仕組み(RFC 3029:2001)も提案されている。i. PKI の利用例SSL(secure socket layer)、TLS(trans-port layer security)はクライアント・サーバ間でセキュア通信を行うための仕組みであり、サーバ認証、クライアント認証、セキュアセッション構築があり、web環境では多く利用されている。関連商品も多くある。S/MIME(secure/multipurpose internet mail extensions)は、電子メールへの署名付けやメール文書の暗号化を行うために開発された仕組みであり、現在多くのメーラーが対応している。現在の電子商取引で多く使われているXMLにおいても、送受信者を認証するために署名付けを行うためのXML署名のスキームや署名付けのためのデータの正規化方法、公開鍵またはその公開鍵証明書を受渡す方式がW3C(world wide web consortium)にて規定されている。公開鍵証明書は、システムを利用する利用者だけではなく、機器が正当な機器なのか(例えば、正当な機器メーカが製造した機器なのか)を認証する場合にも使われる。WAP forumにて規定されたWAP(wireless application protocol)、CableLabsによって規定されたDOCSIS(data over cable service interface specifi-cations)がある。WAPは、ワイヤレス通信ネットワーク上のアプリケーションに対して利用されるプロトコル仕様を規定したものである。その中8―132情報・メディア機器で、クライアント・サーバ間のPKI認証の仕組みが定義されている。ここで使われる公開鍵証明書には、X.509で規定されたもの、X.9.68で規定されたもの、およびワイヤレス通信機器用にメモリサイズ、計算能力、クライアント・サーバ間の通信幅の制約を考慮してWAPが定めたもの(WTLS 形式)がある。DOCSIS はケーブルモデムとサービス提供者間で、サービス提供者がケーブルモデムを認証する仕組み、暗号化鍵を共有してセキュア通信を行う仕組み、ケーブルモデムがソフトウェアダウンロードをセキュアに行う仕組みをPKIベースに規定したものである。このように、PKIは現在広い分野で使われている。
