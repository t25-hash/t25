# 7・2・4 項では公開鍵方式を用いて通信相手の特

> **章**: 第7章 情報圧縮・セキュリティ技術
> **節**: 7・2 情報セキュリティ技術

定や認証,取引データの保証等を行う際に公開鍵保有者の公開鍵を図7・12 属性証明書概念図保証する公開鍵証明書について述べた.つまり,公開鍵証明書は,公開鍵保有者を証明するものである.しかし,実際のサービスでの利用においては,相手を特定・認証するだけではなく,ある特定のサービスやコンテンツを認証した相手に提供してよいかどうか,あるいはある役割を与えてよいかどうかといったアクセス制御も必要になる.そうした背景から権限管理:PMIの概念が出てきた.PMIを実現する方法としては,7・2・4項で述べた公開鍵証明書の拡張領域の一部を利用して行う場合と新たな証明書を利用する場合がある.この新たな証明書は,公開鍵証明書に対して属性証明書(AC: attribute certificate)と呼ばれる.属性証明書は属性認証局(AA : attribute authority)と呼ばれるエンティティから発行することになっており,このAA は認証局(CA)とは明示的に分離される.認証局が第三者機関として公的に存在するのに対し,AA はローカルに存在することもありうるからである.b. 属性証明書(図7・12)属性証明書には,その証明書の発行者の情報,保有者の情報,保有者の公開鍵証明書への参照ポインタ,証明書の有効期限等が含まれる.属性証明書は,属性(特権)を保証するものである.属性証明書保有者には,公開鍵証明書を保持していることが前提となっており,公開鍵証明書が発行されているエンティティに対しておのおのの属性証明書が発行されるという形態になる.したがって,属性証明書には公開鍵証明書を参照できるためのリンク情報(公開鍵証明書のシリアル番号など)が含まれる.また,属性証明書の有効期限は一般的には公開鍵証明書に比べて短いため,失効リストは発行しないモデルも定義されている.公開鍵証明書と,発行者,証明書の有効期限,配布方法,利用範囲(グローバル・ローカル),失効方法が同一の場合,公開鍵証明書の拡張領域を利用することも可能である.属性証明書のフォーマットは,IETF Internet Draft(IETFインタネット草案)およびITU-T X.509,ISO/IEC 9594-8:2001で定義されている.AA 自体は,CA から証明書を発行してもらうことによりその身分が証明できる.AA がさらに他のエンティティにAA として権限を与える属性証明書を発行することもできる.一番上位のAAつまりCA から証明書を発行してもらうことによりその身分が証明されたAA をSOA(source of authority)と呼ぶ.単純なモデルではAA は一つで,AA とSOA は同一である.属性証明書利用者(検証者)は,以下の確認を行う.属性証明書保有者の公開鍵証明書を検証,属性証明書の署名を確認,属性証明書発行者の公開鍵証明書を確認し発行者として正しいことを確認,そして,属性証明書の有効期限,属性を確認する.アプリケーションに属性証明書の内容を問合わせるインタフェースをもたせる場合もある.c. PMI 利用例本項で述べたPMIの技術は,Webサイトのアクセスの制御,ネットワーク内のファイルその他のリソースへのアクセス制御,音楽・映画・ゲームといったコンテンツ配信の際の利用チェックなどに利用される.8―133
